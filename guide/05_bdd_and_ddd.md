# 05. BDD・DDDの活用

このドキュメントは、BDD（振る舞い駆動開発）とDDD（ドメイン駆動設計）をAIとの協調開発にどう活用するかを説明する。
「なぜ振る舞い定義を中心に置くのか」「DDDとどう組み合わせるのか」を理解するために読んでほしい。

---

## 振る舞い定義（BDD）を中心に置く理由

開発の起点として、Gherkin形式のシナリオ（Given/When/Then）を中心に置く。

理由は4つある。

1. **ユーザーも開発者も技術知識なしに参加できる**
   「こういうときこうなってほしい」は誰でも言える。

2. **ヒアリング内容を直接マッピングできる**
   Given/When/Then はヒアリングの言葉とほぼ1対1で対応する。

3. **バックエンド・フロントエンド共通で使える**
   同じGherkin形式で書いたシナリオを、バックエンド・フロントエンドそれぞれのテストツールで実行できる。

4. **そのままテストの受け入れ条件になる**
   「このシナリオを満たす実装をしてください」がAIへの最も明確な指示になる。

---

## Gherkinシナリオの書き方

```gherkin
Feature: 機能の名前

  Background:
    Given 前提条件（全シナリオ共通）

  Scenario: シナリオ名
    Given 前提状態
    When  ユーザーのアクション
    Then  期待される結果
    And   追加の期待される結果
```

AIへの渡し方として、これが最もブレが少ない指示になる。

```
このシナリオを満たす実装をしてください
```

> **参照:** Gherkin形式の詳しい書き方・アンチパターンは `appendix/A_bdd.md` を参照。

## DDDとの接続

振る舞い定義（BDD）とDDDは組み合わせることで精度が上がる。

振る舞い定義を起点に、イベントストーミングで「シナリオの裏で何が起きているか」を掘り下げる。

### コマンド・イベントとGherkinの対応

```
イベントストーミング    →  Gherkinシナリオ

コマンド              →  When  ○○を実行する
ドメインイベント        →  Then  ○○が生成される
ポリシー              →  Given ○○の場合 / Then ○○する
```

コマンドとイベントが整理されると、GherkinシナリオがほぼMECE（漏れなくダブりなく）に書ける。

### イベントストーミングで作られる成果物

```
documents/requirements/domain/
├── ubiquitous_language.md  # ユビキタス言語（用語の定義）
├── commands_and_events.md  # コマンド・イベント・ポリシー一覧
└── domain_model.md         # エンティティ・状態遷移・集約境界
```

> **参照:** イベントストーミングの進め方は `appendix/C_event_storming.md`、
> ドメインモデル・ユビキタス言語の作り方は `appendix/B_ddd.md` を参照。

## 実例マッピングとイベントストーミングの関係

### 2つの発見手法

このガイドでは、2つの発見手法を使い分ける。

| 観点 | 実例マッピング | イベントストーミング |
|------|-------------|-------------------|
| **出典** | Matt Wynne (2015) / BDD | Alberto Brandolini (2013) / DDD |
| **問い** | 「このストーリーは具体的にどう振る舞うか？」 | 「このシステムで何が起きるか？」 |
| **視点** | ユーザーの外側から見た振る舞い | システムの内側で起きていること |
| **粒度** | 1ストーリー単位（細かい） | システム全体 or 機能群（広い） |
| **成果物** | ルール・例・疑問 → Gherkinシナリオ | イベント・コマンド・ポリシー → DDD成果物 |
| **所要時間** | 25分程度/ストーリー | 数時間〜半日/セッション |
| **参加者** | 2〜4人（ストーリーの関係者） | 5〜15人（ドメインエキスパート含む） |

**共通点:**
- どちらも「具体例」を起点にする（抽象的な議論ではなく「このとき何が起きるか」で進める）
- どちらも発見の手法であり、成果物を作ることが目的ではない
- どちらも「疑問・未解決の問題」を明示的に記録する
- どちらもAIを壁打ち相手にして1人でも実施できる

**相違点の本質:**
実例マッピングは「ユーザーから見た振る舞い」を掘り、イベントストーミングは「裏側の構造」を掘る。
両者は同じドメインを**異なる角度から照らす**関係にある。

### 相互に及ぼす影響

2つの手法は独立ではなく、互いの精度を高め合う。

```
実例マッピング → イベントストーミングへの影響
─────────────────────────────────────────
  実例マッピングで発見したルール・例が、
  イベントストーミングの「入力」になる。

  例: 「隣接カラムにのみ移動できる」というルール
    → イベントストーミングで CardMoved と MoveRejected の
      2つのイベントが必要だと気づく
    → MoveRejected に対応するポリシーの議論が始まる

イベントストーミング → 実例マッピングへの影響
─────────────────────────────────────────
  イベントストーミングで発見したポリシー・集約が、
  実例マッピングの「抜け漏れチェック」になる。

  例: 「注文確定時にキッチンへ自動通知」というポリシーを発見
    → 「通知が失敗した場合はどうなるか？」という例が
      実例マッピングに追加される
    → そのルールに対するシナリオが書かれる
```

### 実施順序とサイクル

どちらを先にやるかは、プロジェクトの状況による。

**パターンA: 実例マッピング → イベントストーミング（推奨）**

```
ストーリーがある程度見えている場合:

  実例マッピング（ストーリー単位で具体例を発見）
    ↓
  Gherkinシナリオを書く（正常系）
    ↓
  イベントストーミング（シナリオの裏側を掘る）
    ↓
  ポリシー・集約からシナリオの抜け漏れに気づく
    ↓
  実例マッピングに戻る（異常系・エッジケースを追加）
```

ストーリーが具体化されている方がイベントストーミングの精度が上がるため、
多くの場合こちらが自然な流れになる。

**パターンB: イベントストーミング → 実例マッピング**

```
ドメインが複雑で全体像が見えない場合:

  イベントストーミング（全体のイベントフローを発見）
    ↓
  集約・コンテキストの境界を決める
    ↓
  各集約に対して実例マッピング（具体例を発見）
    ↓
  Gherkinシナリオを書く
```

レストランの注文システムや予約システムのように、複数のアクターとポリシーが絡み合う場合は、
先に全体の流れを見てから各機能を掘り下げる方が効率的なことがある。

### 規模別の推奨度合い

| 規模 | 実例マッピング | イベントストーミング | サイクル回数 |
|------|-------------|-------------------|------------|
| 1人・単機能 | ◎ 必須（軽く） | △ 省略可 | 1〜2周で十分 |
| 小チーム・複数機能 | ◎ 必須 | ○ 複雑な箇所のみ | 2〜3周 |
| 中規模・複数チーム | ◎ 必須 | ◎ 必須 | 3周以上 |
| 大規模・複雑ドメイン | ◎ 必須 | ◎ 必須（複数回） | 継続的に回す |

**1人・単機能の場合:**
実例マッピングだけで十分なことが多い。ルールと例を洗い出してシナリオを書けば、
イベントストーミングなしでも実装に必要な情報は揃う。

**中規模以上の場合:**
両方をサイクルで回すことが強く推奨される。片方だけでは、振る舞いの定義（外側）と
ドメインの構造（内側）のどちらかに偏り、抜け漏れが生じやすい。

> **参照:** 実例マッピングの詳細は `appendix/A_bdd.md`、イベントストーミングの詳細は `appendix/C_event_storming.md`。

## サイクルを回す

振る舞い定義とDDDは一方通行ではなく、サイクルを回しながら精度を上げる。
**1周目から完璧にしようとしない**のが最大のコツ。粗いシナリオは「次の問いを生む素材」として機能する。

```
1周目: 実例マッピングで正常系のルール・例を発見 → Gherkinシナリオを書く
  ↓
2周目: イベントストーミングで構造を掘る → ポリシーから異常系シナリオを追加
  ↓
3周目: 実例マッピングでエッジケースを発見 → コマンド・集約を整理
  ↓
さらなる抜け漏れに気づく → 実例マッピング or イベントストーミングに戻る
  ↓
収束したら → フェーズ2（詳細設計）へ
```

## 規模とDDDの使い分け

規模が小さいうちは軽量なアプローチで十分。規模に応じて段階的に深めていく。

| 規模 | 推奨アプローチ |
|------|-------------|
| 1人・単機能 | コマンド抽出だけ軽く行う |
| 小チーム・複数機能 | イベントストーミングを複雑な箇所のみ実施 |
| 中規模・複数チーム | イベントストーミング + 集約の定義 |
| 大規模・複雑ドメイン | 境界付けられたコンテキスト + コンテキストマップ |

---

## コーディングAI観点でのポイント

規模が上がるほど、**コーディングAIに渡す文脈の「切り方」**が重要になる。

大きな仕様書を丸ごと渡すより、**境界付けられたコンテキスト単位で渡す**方が品質が安定する。

```
コンテキストAの実装依頼
  → コンテキストAに関係するdocuments・specificationだけ渡す

コンテキストBの実装依頼
  → コンテキストBに関係するものだけ渡す
```

DDDのコンテキスト分割は、そのままコーディングAIへの委譲単位になり得る。
これが大規模開発でDDDが効く本質的な理由のひとつである。

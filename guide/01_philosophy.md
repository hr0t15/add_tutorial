# 01. 基本思想・人間とAIの役割

このドキュメントは、AIを活用した開発における基本的な考え方を記述する。
「なぜこのプロセスで開発するのか」を理解するための出発点として読んでほしい。

---

## コーディングAIとは何か

コーディングAIは「分かっている人間がより速く動ける道具」である。

指示に忠実に動く性質上、指示が的外れだったり曖昧だったりすると、忠実に**間違った方向へ**走る。

```
分かっている人間 × コーディングAI = 成果が加速する
分かっていない人間 × コーディングAI = 間違いが加速する
```

これはコーディングAIの欠点ではなく、**道具としての本質**である。
使う側がどれだけ「分かっているか」が、そのまま成果の質に反映される。

---

## 人間に求められること

AIを使いこなすために必要なのは、技術的な知識よりも以下の3つである。

- **何を作りたいかが言語化できる**
- **何が良くて何が悪いか判断できる**
- **どこで止まって確認すべきか分かる**

AIの機能そのものは「聞けば教えてもらえる」。
一方でドメイン知識・業務の形式知は自分の中にしかなく、AIが代わりに持つことはできない。
陳腐化しないのも、技術知識ではなくドメイン知識の方である。

---

## 役割の分担

人間とAIの役割を明確に分けることが、協調の品質を決める。

```
人間が単独で担うべきもの
  └─ ビジョン・目的（なぜ作るか）
  └─ 価値判断の基準（何が良くて何が悪いか）
  └─ ドメインの暗黙知
  └─ 最終的な承認・責任

協調して作るべきもの
  └─ 要求・要件定義
  └─ 仕様書・設計ドキュメント
  └─ アーキテクチャの選択肢とトレードオフ整理
  └─ ユビキタス言語・用語集

AIに委譲できるもの
  └─ 実装・テスト
  └─ ドキュメントの構造化・清書
  └─ 選択肢の列挙・比較表
  └─ 既存コードからの仕様逆引き
```

---

## 「委譲するものを協調しながら作る」という価値

AIが得意な領域が広がるほど、**「何を・どう委譲するか」を決める上流の判断こそが人間固有の価値**になる。

上流で人間とAIが協調して「何を作るか」を固めるほど、下流の委譲品質が上がる。

> **ドメイン知識 × 言語化力 × AIとの壁打ち**

この3つが揃うと、開発の質とスピードが大きく上がる。

---

## 言語化の2段階

AIとの協調において「言語化」は最も重要なスキルである。言語化には2つの段階がある。

```
第1段階: 頭の中にあるものを言葉にする
  → 壁打ちが有効なのはここ
  → 問い返されることで「自分がそう思っていたこと」に気づける

第2段階: 言葉にしたものを相手に伝わる形に整える
  → AIが整理・補完を担える
```

壁打ちの価値は、第1段階を一人でやるより速く・深くできる点にある。
最初のシナリオや仕様が粗くても構わない。往復するうちに言語化が深まり、仕様が育っていく。

---

## コンテキストを設計するという考え方

コーディングAIには会話のコンテキストウィンドウという制約がある。
一度に渡せる情報量に上限があるため、**何を渡すかの設計**が品質に直結する。

```
悪い例: 全部まとめて渡す
  設計・仕様・実装の情報をすべて一度に渡す
  → コンテキストが肥大化し、重要な情報が埋もれる
  → 指示の精度が下がる

良い例: フェーズごとに分けて渡す
  フェーズ2: 設計ドキュメントだけ渡す → 詳細設計を生成させる
  フェーズ3: 詳細設計だけ渡す → 実装させる
  → 各フェーズで必要な情報だけに絞られる
  → AIが「今何をすべきか」を誤解しにくくなる
```

これは「コンテキストの節約」ではなく**「コンテキストの設計」**である。
渡す情報を意図的に絞ることで、ノイズが減り指示の精度が上がる。

このプロセスで採用している「要求定義 → 詳細設計 → 実装」の3フェーズは、この原則を構造として体現したものである。

> 各用語とディレクトリの対応は `02_development_process.md` の「用語とディレクトリの対応」を参照。
> コンテキスト設計の実践的なパターン（ポインタ渡し・スコープ限定・設定ファイル活用等）は `appendix/F_context_engineering.md` を参照。

## このガイド自身の限界と課題

このガイドは「構造化された知識がAIの効果を高める」という前提に基づいているが、
ガイド自身にも未解決の課題がある。正直に記しておく。

### 成果物セットが重い可能性がある

このガイドはフェーズ1で最大5種類の成果物（要件概要・振る舞い定義・ドメイン定義・判断記録・UI設計）を定義している。
`02_development_process.md` に規模別の省略目安はあるが、**「この規模ならこの組み合わせで十分」という最小有効セットは確立されていない。**

```
未検証の問い:
  - 1人開発で overview.md + features/ だけで回した場合、品質はどう変わるか？
  - domain/ を省略して features/ だけで渡した場合、AIの実装精度はどこまで落ちるか？
  - ADRなしで進めた場合、2ヶ月後に「なぜこうなっているか」問題はどの頻度で起きるか？
```

「全部やれば良い」とは限らない。成果物を作るコスト自体が開発を遅くするなら本末転倒である。
どの組み合わせが費用対効果のスイートスポットなのかは、実践を通じて検証・更新していく必要がある。

### 発見プロセスの有効性が十分に検証されていない

このガイドは実例マッピングとイベントストーミングを発見プロセスの柱としているが、
これらを「AIとの協調開発」の文脈で体系的に検証したエビデンスは限られている。

- 実例マッピングがAIへの仕様伝達の精度を実際にどの程度向上させるかは未測定
- イベントストーミングを1人+AIで行う場合と、チームで行う場合の品質差は未知
- BDD・DDDの知識がない開発者にとっての学習コストと効果のバランスは不明確

### 足りない要素がある可能性

現在のガイドがカバーしていない、あるいは薄い領域がある。

| 領域 | 現状 | 課題 |
|------|------|------|
| 非機能要件 | overview.md での観点リストとADRでの判断記録を定義（`02_development_process.md`, `appendix/D_adr.md`） | 発見・記録の手法は整備したが、非機能要件の自動検証（パフォーマンステスト・セキュリティスキャン等）の体系化は未対応 |
| 運用・監視 | カバーしていない | 実装後の運用設計をどう仕様に組み込むかの指針がない |
| チーム間の合意形成 | 協調マトリックスで役割は定義 | 複数チーム間での仕様の共有・承認プロセスが弱い |
| 仕様の腐敗防止 | 改修プロセスで同期確認の手法を定義（`07_modification_process.md`） | 乖離の検出は手動（AIへの依頼）のまま。自動検出の仕組みは未対応 |
| レガシーとの統合 | カバーしていない | 既存コードベースにこのプロセスを導入する移行パスがない |

### ハーネスエンジニアリングとの関係

`appendix/I_harness_engineering.md` で述べている通り、ハーネスエンジニアリングは
コーディングAIの実行環境を設計する技術であり、2026年時点で急速に重要性が高まっている。
本ガイドの成果物のうち、Gherkinシナリオ・ADRはハーネスとの相性が良い一方で、
overview.md・domain_model.md等の記述的ドキュメントには「腐敗」のリスクがある。
この緊張関係への対処は `appendix/I_harness_engineering.md` の「このガイドとの関係」で詳述している。

### スペック駆動開発との関係における不確実性

`appendix/H_spec_driven_development.md` で述べている通り、スペック駆動開発はまだ確立途上であるが、
このガイド自身がスペック駆動開発に対してどう位置づけられるかも不確実である。
`appendix/H_spec_driven_development.md` の「このガイド自身の弱み」セクションに、より詳細な分析がある。

- スペック駆動のツールチェーンが成熟した場合、このガイドの成果物体系が**冗長**になる可能性
- 逆に、発見プロセスの重要性が再認識され、このガイドの価値が**増す**可能性
- いずれにせよ、AIの進化速度に対してガイド自体の更新が追いつかないリスクがある

> このガイドは「完成品」ではなく「現時点のベストプラクティス仮説」として扱ってほしい。
> 実践の中で「これは不要だった」「これが足りなかった」が見えたら、ガイド自体を更新すべきである。

## まとめ

AIを使いこなすとは、**上流の判断力を磨き、適切な単位で委譲する設計をすること**である。

```
ドメイン知識を持つ
  └─ 言語化して壁打ちする
       └─ 設計ドキュメントに落とす
            └─ 適切な粒度でAIに委譲する
                 └─ レビューして次の判断をする
```

このサイクルを回し続けることが、AIとの協調開発の本質である。

> **参照:** このガイドのアプローチが有効である研究的裏付けは `appendix/G_ai_adoption_insights.md` を参照。

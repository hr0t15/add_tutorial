# 07. 改修プロセス

このドキュメントは、本ガイドのプロセスで構築したシステムを**更新・改修する**ときの進め方を説明する。
`02_development_process.md` が「ゼロから作る」流れであるのに対し、
このドキュメントは「既にあるものを変える」流れを扱う。

> **注意:** 改修プロセスはまだ実践知が十分に蓄積されていない領域である。
> このドキュメントは「現時点の推奨方針」として読んでほしい。
> 実践を通じて「これは過剰だった」「これが足りなかった」が見えたら更新すべきである。

---

## 新規構築と改修の違い

```
新規構築（02_development_process.md）:
  何もないところから成果物を育てる
  → 発見 → 言語化 → 構造化 → 委譲

改修:
  既にある成果物群とコードの整合性を保ちながら変更する
  → 影響分析 → 変更範囲の特定 → 部分更新 → 同期確認
```

改修には新規構築にはない固有の判断が3つある。

| 判断 | 問い |
|------|------|
| **どこまで遡るか** | 実装だけ直す？ 設計まで？ 要求定義まで？ |
| **何を同期させるか** | 変更の波及で更新すべき成果物はどれか？ |
| **壊していないことをどう確認するか** | 既存機能への回帰をどう検証するか？ |

---

## 変更規模の分類と対応フロー

変更の規模によって、遡るべき深さが変わる。

### 軽微な変更

**定義:** 既存の仕様・設計に影響しない変更。

```
例:
  - バグ修正（仕様通りに動いていなかった）
  - 文言・スタイルの調整
  - リファクタリング（振る舞いを変えない構造改善）
```

```
対応フロー:
  実装を修正する
    ↓
  テストが通ることを確認する
    ↓
  （成果物の更新は不要）
```

**コーディングAIへの指示例:**

```
以下のバグを修正してください。

バグ: [具体的な症状]
期待する動作: [正しい動作]
該当する仕様: documents/requirements/features/xxx.feature のシナリオ「○○」

既存のテストがすべて通ることを確認してください。
```

> 仕様通りに動いていなかった場合は実装のバグである。
> 仕様自体が間違っていた場合は「中規模の変更」に該当する。

---

### 中規模の変更

**定義:** 既存の仕様や設計の一部を変更する必要がある変更。

```
例:
  - 機能追加（既存ドメインに新しい振る舞いを追加）
  - 仕様変更（既存のルールを変える）
  - 非機能要件の変更（パフォーマンス目標の引き上げ等）
```

```
対応フロー:
  ① 影響分析: 何が変わるかを特定する
    ↓
  ② 要求定義の部分更新: 該当するfeatureファイル・overview.mdを修正
    ↓
  ③ 詳細設計の再生成: 変更部分の設計を更新
    ↓
  ④ 実装: 変更部分を実装
    ↓
  ⑤ 回帰確認: 既存テストがすべて通ることを確認
```

#### ① 影響分析

変更の起点がどこかを特定し、波及範囲を見極める。

**AIへの影響分析依頼:**

```
以下の変更を行いたいです。
影響範囲を分析してください。

変更内容: [何をどう変えたいか]

以下のファイルを読んで、影響を受ける成果物を特定してください:
- documents/requirements/overview.md
- documents/requirements/features/ 配下のfeatureファイル
- documents/requirements/domain/ 配下のドメイン定義
- documents/requirements/adr/ 配下のADR

影響分析の観点:
- 変更が必要なfeatureファイルはどれか
- ドメインモデル（用語・集約・イベント）に影響はあるか
- 既存のADRの判断と矛盾しないか
- 非機能要件に影響はあるか
```

#### ② 要求定義の部分更新

影響分析の結果に基づき、該当する成果物だけを更新する。

**featureファイルの更新依頼:**

```
以下の仕様変更を反映してください。

変更内容: [何をどう変えるか]
対象ファイル: documents/requirements/features/xxx.feature

既存のシナリオを確認し:
- 変更が必要なシナリオを修正してください
- 新しいシナリオが必要なら追加してください
- 不要になったシナリオがあれば削除してください
- 変更していないシナリオには手を加えないでください
```

**overview.mdの更新依頼:**

```
以下の変更を overview.md に反映してください。

変更内容: [何をどう変えるか]

変更に関係する箇所のみ更新してください。
変更していない機能要件・非機能要件・スコープ外はそのまま残してください。
```

#### ③④ 詳細設計の再生成と実装

更新した要求定義を渡して、設計と実装を進める。
この部分は `02_development_process.md` のフェーズ2・3と同じ流れだが、**スコープを変更部分に限定する**。

```
以下のファイルを読んで、変更部分の詳細設計を更新してください。

変更した要求定義:
- documents/requirements/features/xxx.feature（変更箇所）
- documents/requirements/domain/domain_model.md（変更があれば）

既存の詳細設計:
- documents/specification/（現在の設計）

変更が必要な設計箇所のみ更新してください。
変更していない設計はそのまま残してください。
```

#### ⑤ 回帰確認

変更後に、既存のテストがすべて通ることを確認する。

```
実装が完了しました。以下を確認してください:
1. 変更したシナリオに対応するテストが通ること
2. 変更していないシナリオのテストも通ること（回帰確認）
3. 変更内容がADRの判断と矛盾していないこと
```

---

### 大規模な変更

**定義:** ドメインモデルの構造・コンテキスト境界・アーキテクチャに影響する変更。

```
例:
  - ドメインモデルの根本的な見直し（集約の分割・統合）
  - 境界付けられたコンテキストの追加・分割
  - アーキテクチャの変更（モノリスからマイクロサービスへ等）
  - 大幅な機能削除
```

```
対応フロー:
  ① 変更の動機と影響範囲をADRに記録する
    ↓
  ② 要求定義を広範に見直す（overview.md, features/, domain/）
    ↓
  ③ 詳細設計を再生成する
    ↓
  ④ 実装する（段階的に移行）
    ↓
  ⑤ 全テストの回帰確認
```

大規模な変更は、事実上 `02_development_process.md` のフェーズ1から再走することに近い。
ただし「ゼロから作り直す」のではなく、**既存の成果物を起点にして差分を育てる**。

**変更起点のADR作成依頼:**

```
以下の変更を検討しています。ADRを作成してください。

変更の動機: [なぜ変更が必要か]
現在の状態: [今どうなっているか]
目指す状態: [どうしたいか]

以下を読んで、影響範囲を分析した上でADRに含めてください:
- documents/requirements/overview.md
- documents/requirements/domain/
- documents/requirements/adr/（既存のADRとの矛盾がないか確認）

特に、既存のADRの判断を覆す場合は、
覆されるADRの番号と理由を明示してください。
```

## 変更規模の判断基準

どの規模に該当するかの判断は、以下のフローで行う。

```
変更は既存のfeatureファイルのシナリオ内で表現できるか？
  │
  ├─ Yes → featureファイル自体の修正は不要か？
  │          │
  │          ├─ Yes → 軽微な変更（実装のみ）
  │          └─ No  → 中規模の変更
  │
  └─ No → ドメインモデル（用語・集約・イベント）の変更が必要か？
            │
            ├─ No  → 中規模の変更（featureの追加・修正で済む）
            └─ Yes → 大規模の変更
```

> **迷ったら中規模として始める。** 影響分析の結果、想定より波及が大きいと分かったら大規模に切り替える。
> 最初から大規模として全面見直しするのは過剰になりやすい。

## 成果物の同期

改修で最も注意すべきは、**変更した箇所と関連する成果物の同期**である。

### 変更起点ごとの同期マップ

| 変更の起点 | 更新すべき可能性がある成果物 |
|-----------|------------------------|
| **要求（overview.md）** | features/ → domain/ → specification/ → implement/ |
| **振る舞い（features/）** | domain/（用語・イベント追加の可能性） → specification/ → implement/ |
| **ドメイン（domain/）** | features/（シナリオの表現が変わる可能性） → specification/ → implement/ |
| **設計（specification/）** | implement/（設計変更の反映） |
| **実装（implement/）** | なし（バグ修正の場合）。仕様自体が間違いなら features/ まで遡る |

### AIに同期確認を依頼する

```
以下の変更を行いました。成果物の同期状態を確認してください。

変更したファイル:
- [変更したファイルのリスト]

以下のファイルを読んで、変更内容と矛盾していないか確認してください:
- documents/requirements/overview.md
- documents/requirements/features/ 配下のfeatureファイル
- documents/requirements/domain/ 配下のドメイン定義
- documents/specification/ 配下の設計書

矛盾や不整合があれば指摘してください。
更新が必要なファイルがあれば、具体的な修正内容を提案してください。
```

## 仕様の腐敗を防ぐ

改修を重ねると、成果物とコードの間に乖離（腐敗）が生じやすくなる。

### 腐敗が起きやすい成果物と起きにくい成果物

| 成果物 | 腐敗リスク | 理由 |
|--------|----------|------|
| **Gherkinシナリオ** | 低い | テストとして実行されるため、乖離があればテストが落ちる |
| **ADR** | 低い | 不可変の記録であり、内容が古くなっても「当時の判断」として価値がある |
| **overview.md** | 中程度 | 機能追加時に更新を忘れやすい。ただし影響は限定的 |
| **domain_model.md** | 高い | コードの構造変更が反映されにくい。乖離するとAIへの指示精度が下がる |
| **specification/** | 高い | 実装の変更が反映されにくい。再生成で対処可能 |

### 腐敗防止の実践

```
実行可能な成果物（シナリオ・テスト）:
  → テストが通る限り腐敗しない。最も信頼できる

記述的な成果物（overview.md, domain_model.md）:
  → 改修のたびに同期確認を行う（上記の同期確認プロンプトを使う）
  → 大規模な変更の後は、AIに「現在の実装から成果物を再生成」させて差分を確認する

詳細設計（specification/）:
  → 改修のたびに再生成するのが最も安全
  → 前回の設計との差分を確認し、意図しない変更がないかレビューする
```

**実装からの成果物再生成で乖離を検出する:**

```
implement/ の現在のコードを読んで、
 以下の成果物と矛盾している箇所がないか確認してください。

- documents/requirements/overview.md
- documents/requirements/features/ 配下のfeatureファイル
- documents/requirements/domain/ 配下のドメイン定義

矛盾がある場合:
- コードと成果物のどちらが正しいか判断できる場合は、修正案を提示してください
- 判断できない場合は、矛盾の内容だけ報告してください
```

> この検出は手動であり、自動化されていない。
> `01_philosophy.md` の「仕様の腐敗防止」で認めている通り、自動検出の仕組みは今後の課題である。
> `appendix/I_harness_engineering.md` のHooksやCI統合を活用することで、部分的な自動化は可能である。

## 改修時の協調パターン

`04_collaboration_matrix.md` の協調パターンは改修でも基本的に同じだが、
改修固有のパターンが1つある。

### 差分レビュー型

改修では「AIが変更案を出し、人間が既存との整合性を確認する」パターンが頻出する。

```
差分レビュー型:
  AI: 変更前と変更後の差分を提示する
  人間: 意図しない変更がないか確認する
  人間: 既存機能への影響を判断する
```

新規構築のレビュー型との違いは、**「何が変わったか」だけでなく「何が変わっていないか」も確認する**点にある。

## このプロセスの限界

### 検証が不十分な領域

```
未検証の問い:
  - 成果物の同期確認をAIに任せて、どの程度の精度で乖離を検出できるか？
  - 大規模な変更で「要求定義から再走」した場合、既存成果物がどこまで再利用できるか？
  - 改修を重ねた成果物は、ゼロから作った成果物と比べて品質がどう変化するか？
  - 軽微・中規模・大規模の判断を誤った場合のリカバリーコストはどの程度か？
```

### 手動に頼っている部分

- 成果物の同期確認は人間またはAIへの依頼による手動プロセス
- 変更規模の判断は人間の経験に依存する
- 仕様とコードの乖離検出は自動化されていない

> 改修プロセスは新規構築以上に実践知の蓄積が必要な領域である。
> 「これは過剰だった」「これが足りなかった」が見えたら、このドキュメント自体を更新してほしい。

---

## まとめ

```
軽微な変更:
  実装を直す → テストで確認 → 成果物更新不要

中規模の変更:
  影響分析 → 該当する成果物を部分更新 → 設計再生成 → 実装 → 回帰確認

大規模の変更:
  ADRで動機を記録 → 要求定義から再走 → 段階的に移行

共通原則:
  - 迷ったら中規模から始める
  - Gherkinシナリオが最も信頼できる同期の基準点
  - 改修のたびに成果物の同期を確認する
  - 仕様が間違っていたら実装ではなく仕様を直す
```

> **参照:** 成果物の構成は `03_directory_structure.md`、
> 仕様の腐敗防止の課題は `01_philosophy.md` の「足りない要素がある可能性」、
> ハーネスエンジニアリングによる部分的な自動化は `appendix/I_harness_engineering.md` を参照。

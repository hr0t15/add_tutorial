# 06. UI設計とテスト戦略

このドキュメントは、ユーザーインターフェース（GUI / CLI）の設計成果物をどう作るか、
テスト戦略をどう決めるかの **共通原則** を説明する。

GUI固有の詳細は `06_01_gui.md`、CLI固有の詳細は `06_02_cli.md` を参照。

---

## バックエンドとの対称性

UIはGUI・CLIを問わず、バックエンドと対称的な構造で設計できる。
特に重要なのは、**Gherkin形式がバックエンド・UI共通で使える**という点である。

| バックエンド | GUI | CLI |
|------------|-----|-----|
| ドメインモデル | 画面・コンポーネント定義 | コマンド・サブコマンド定義 |
| コマンド・イベント | ユーザーアクション・画面状態遷移 | コマンド実行・出力・終了コード |
| Gherkinシナリオ | Gherkinシナリオ（同じ形式） | Gherkinシナリオ（同じ形式） |
| ユニットテスト | コンポーネントテスト | コマンドユニットテスト |
| APIインターフェース定義 | APIモック定義 | 外部サービスモック定義 |

バックエンドと同じ思想・同じ形式で書けるため、documents/ の構造も対称的になる。

---

## プロセス上の位置づけ

UI設計は `02_development_process.md` のフェーズ1・ステップ⑤に対応する。
バックエンドの振る舞い定義（ステップ②）やドメイン構造化（ステップ③）と並行して進めてよい。

```
① 要件の言語化   → overview.md が書かれた後に着手できる
② 振る舞い定義   ← ⑤と並行して進めると、バックエンド・UI両面からシナリオの抜け漏れに気づける
③ ドメインの構造化
④ 判断の記録
⑤ UI設計 ← ここ（GUI or CLI）
```

## 共通原則

### 1. ユーザーとの接触面をドメインロジックから分離する

UIの種類（GUI / CLI）が変わっても、ドメインロジックは変わらない。

```
GUI:  ボタンクリック → ドメインロジック → 画面に結果表示
CLI:  コマンド実行   → ドメインロジック → stdoutに結果出力
```

この分離があることで、UIの変更がドメインに波及しない。

### 2. Gherkinシナリオが受け入れ条件になる

GUI・CLIどちらでも、featureファイルのシナリオがそのままテストケースになる。

```gherkin
# GUI の場合
Scenario: 注文を確定する
  Given ユーザーが注文画面にいる
  When  「注文確定」ボタンをクリックする
  Then  注文完了画面に遷移する

# CLI の場合
Scenario: 注文を確定する
  Given 注文ID "ORD-001" が未確定状態である
  When  `order confirm ORD-001` を実行する
  Then  "注文 ORD-001 を確定しました" と出力される
  And   終了コード 0 で終了する
```

ビジネスルール（「注文を確定できる」）は同じ。表現する面が異なるだけである。

### 3. テスト戦略は規模で決める

GUI・CLIどちらでも、テストの粒度は規模に応じて段階的に追加する。
小さいうちはシンプルなE2E（統合）テストだけで十分。

| 規模 | 方針 |
|------|------|
| 1人・単機能 | 正常系のE2E/統合テストだけで十分 |
| 小チーム・複数機能 | 異常系を追加。テストヘルパーで保守性を上げる |
| 中規模・複数チーム | モックを導入。チーム間の境界をテストでカバー |
| 大規模 | 全粒度を揃える |

具体的なテスト粒度は GUI / CLI で異なるため、それぞれの詳細ドキュメントを参照。

### 4. 設計成果物がテストの根拠になる

UI設計の成果物は「何を作るか」の定義であると同時に「何をテストするか」の根拠でもある。

```
GUI: 画面遷移図 → E2Eテストのフロー設計
CLI: コマンド体系 → 統合テストのシナリオ設計
```

この対応関係を意識して成果物を作ると、テスト設計が自然に決まる。

## GUI / CLI の選び方

チュートリアルではすべてGUIを題材にしているが、実務ではCLIが適切な場合もある。

| 観点 | GUI が向いている | CLI が向いている |
|------|----------------|----------------|
| ユーザー層 | 非技術者・一般ユーザー | 開発者・運用担当者 |
| 操作の性質 | 視覚的・探索的 | 反復的・自動化可能 |
| データ量 | 1画面に収まる程度 | 大量・パイプで連携 |
| 環境 | ブラウザ・モバイル | ターミナル・CI/CD |

> 両方を持つシステム（Web UI + CLI）も多い。
> その場合はドメインロジックを共有し、UI層だけをGUI/CLIで分けるのが理想的な設計である。

# Appendix F: コンテキストエンジニアリング実践ガイド

> **コンテキストエンジニアリング**とは、AIに渡す情報（コンテキスト）を
> システムとして設計する考え方である。
> 単発のプロンプトを工夫する「プロンプトエンジニアリング」から一歩進み、
> **どの情報を・いつ・どの形式で・どの範囲だけ渡すか**を設計対象とする。
>
> この概念はDORA 2025で体系的に言及され、
> `appendix/G_ai_adoption_insights.md` で述べた「AIは組織の鏡」という知見と直結する。
> 構造化された知識がなければ、AIに渡すコンテキストも構造化できない。
>
> コンテキストエンジニアリングは「ハーネスエンジニアリング」の一部でもある。
> AIに渡す情報の設計（本Appendix）と、AIが動作する実行環境の設計（`appendix/I_harness_engineering.md`）は
> 相互補完の関係にある。

---

## コンテキストウィンドウを「作業机」として考える

コーディングAIのコンテキストウィンドウは、物理的な作業机に似ている。

```
作業机（コンテキストウィンドウ）:
  ┌─────────────────────────────────────┐
  │  今の作業に必要な資料だけ置く        │
  │                                     │
  │  資料が多すぎる → 探すのに時間がかかる │
  │  資料が少なすぎる → 判断材料が足りない │
  │  無関係な資料がある → 混乱する        │
  └─────────────────────────────────────┘
```

### 原則: 必要十分な情報を、適切なタイミングで渡す

| やること | やらないこと |
|---------|------------|
| 今のタスクに関連するファイルだけ渡す | リポジトリ全体を一度に読ませる |
| 判断に必要な背景（ADR等）を添える | 「よしなにやって」と丸投げする |
| 成果物を書き出してコンテキストを解放する | すべてを会話の中だけで完結させる |

## ファイルに書き出すことでコンテキストを解放する

このガイドが成果物をファイルに書き出すことを重視する理由の1つは、
**コンテキストウィンドウの有効活用**にある。

```
壁打ちの結果 → overview.md に書き出す
  → 次の会話では overview.md を読ませるだけでよい
  → 壁打ちの過程（試行錯誤・脱線）はコンテキストに残らない
  → 「結論だけ」を渡せるので、AIの精度が上がる
```

### ファイルに書き出す判断基準

| 状況 | 書き出す | コンテキストに残す |
|------|---------|-----------------|
| 判断が確定した | ✅ ファイルに記録 | — |
| まだ議論中 | — | ✅ 会話で継続 |
| 後の工程で参照する | ✅ ファイルに記録 | — |
| 今の会話でしか使わない | — | ✅ 会話に残す |

### フェーズ間のコンテキスト設計

このガイドの3フェーズ開発は、コンテキストエンジニアリングの観点でも合理的である。

```
フェーズ1（要求定義）:
  入力: ドメイン知識・壁打ちメモ
  出力: overview.md, features/, domain/
  → ここでコンテキストを「清書」する

フェーズ2（詳細設計）:
  入力: フェーズ1の成果物だけ（壁打ちメモは不要）
  出力: detailed_design.md
  → フェーズ1の試行錯誤はコンテキストに入れない

フェーズ3（実装）:
  入力: detailed_design.md + ADR
  出力: コード
  → overview.md すら不要な場合がある（詳細設計に集約済み）
```

> **ポイント:** フェーズが進むごとに、前フェーズの「結論だけ」を渡す。
> これがこのガイドの「フェーズを分けて情報を渡す」原則の本質である。

## コーディングAIの設定ファイルを活用する

多くのコーディングAIは、プロジェクトルートに置く設定ファイルで
AIの振る舞い（前提知識・コーディング規約・禁止事項等）を制御できる。

これは**会話のたびに繰り返す指示を、永続的なコンテキストとして定義する**仕組みであり、
コンテキストエンジニアリングの基本ツールである。

### ツール別の設定ファイル

| ツール | 設定ファイル | 配置場所 | 備考 |
|--------|------------|---------|------|
| Claude Code | `CLAUDE.md` | プロジェクトルート、サブディレクトリ、`~/.claude/` | 階層的に配置可能 |
| Cursor | `.cursorrules` | プロジェクトルート | プロジェクト単位で適用 |
| GitHub Copilot | `copilot-instructions.md` | `.github/` 配下 | リポジトリ単位で適用 |
| Windsurf | `.windsurfrules` | プロジェクトルート | プロジェクト単位で適用 |
| Gemini Code Assist | `.gemini/settings.json` 等 | プロジェクトルート | Google Cloud統合が強み。設定仕様は発展中 |
| Codex (OpenAI CLI) | `AGENTS.md`, `codex.md` | プロジェクトルート | サンドボックス実行が特徴 |

> ツールの仕様は頻繁に更新される。最新の情報は各ツールの公式ドキュメントを参照すること。

### 設定ファイルに書くべきこと

どのツールでも、設定ファイルに書く内容の考え方は共通する。

```
1. プロジェクトの前提:
   - 技術スタック（言語・フレームワーク・バージョン）
   - ディレクトリ構成の概要
   - 主要な成果物へのポインタ

2. コーディング規約:
   - 命名規則
   - ユビキタス言語（domain/ubiquitous_language.md へのポインタ）
   - 禁止パターン（例: any を使わない、直接 DOM 操作しない）

3. 開発プロセスの指示:
   - テストの実行方法
   - コミットメッセージの形式
   - レビュー時の注意点

4. このガイド固有の指示（推奨）:
   - 「実装前に detailed_design.md を確認すること」
   - 「ユビキタス言語に定義されていない用語を使わないこと」
   - 「ADR に記録された判断に反する実装をしないこと」
```

### 構成例（Claude Codeの場合）

以下は Claude Code の `CLAUDE.md` の構成例である。
他ツールでも内容の考え方は同じで、ファイル名と配置場所を読み替えればよい。

```markdown
# CLAUDE.md

## プロジェクト概要
美容院予約システム。予約・スタッフ・顧客の3コンテキストで構成。

## 技術スタック
- TypeScript / React / Vite
- テスト: Vitest + Playwright

## ディレクトリ構成
spec/ 配下に要求定義がある。実装前に必ず参照すること。
- spec/overview.md — 要件概要
- spec/features/ — Gherkinシナリオ
- spec/domain/ — ドメインモデル・ユビキタス言語
- spec/adr/ — 判断記録

## コーディング規約
- spec/domain/ubiquitous_language.md の用語を使うこと
- any 禁止。型を明示する
- コンポーネントは関数コンポーネントのみ

## テスト
- `npm test` でユニットテスト実行
- `npm run test:e2e` でE2Eテスト実行

## 注意事項
- spec/adr/ の判断に反する実装をしないこと
- 新しい技術的判断をする場合は、先に ADR を書くこと
```

### 階層的な設定（Claude Codeの場合）

Claude Code の `CLAUDE.md` はサブディレクトリにも配置でき、階層的に適用される。
コンテキスト分割されたプロジェクトでは、この仕組みが有効である。

```
project/
├── CLAUDE.md                    ← 全体共通の規約
├── src/
│   ├── reservation/
│   │   └── CLAUDE.md            ← 予約コンテキスト固有の指示
│   ├── staff/
│   │   └── CLAUDE.md            ← スタッフコンテキスト固有の指示
│   └── customer/
│       └── CLAUDE.md            ← 顧客コンテキスト固有の指示
```

> この階層構造は Claude Code 固有の機能である。
> 他ツールでは、プロジェクトルートの単一ファイルに集約するか、
> コメント内のセクション分けで代替する。

## コンテキスト設計のパターン

### パターン1: ポインタ渡し

ファイルの内容を直接貼るのではなく、パスだけを渡す。

```
❌ 「以下がドメインモデルです: [100行のドメインモデルを貼る]」
✅ 「ドメインモデルは spec/domain/domain_model.md を参照してください」
```

コーディングAIはファイルを自分で読めるため、パスを渡すだけで十分である。
これにより、指示のコンテキストを小さく保てる。

### パターン2: スコープ限定

作業範囲を明示的に限定する。

```
❌ 「予約システムを実装してください」
✅ 「予約コンテキストの CreateReservation コマンドを実装してください。
    仕様は spec/features/reservation.feature の最初のシナリオです。
    他のコンテキスト（スタッフ・顧客）は触らないでください」
```

このガイドの「コンテキスト単位での委譲」（チュートリアル3）は、
このスコープ限定パターンの実践である。

### パターン3: ブリッジコンテキスト

新しい会話を始めるとき、前の会話の結論だけを渡す。

```
前の会話: 壁打ち → 実例マッピング → overview.md を生成
  ↓
新しい会話: 「spec/overview.md を読んで、詳細設計を生成してください」
  ↓
さらに新しい会話: 「spec/detailed_design.md を読んで、実装してください」
```

各会話は独立しているが、ファイルを介してつながっている。
これがフェーズ間のコンテキスト設計の具体的な実践である。

### パターン4: コンテキストのリフレッシュ

長い会話でコンテキストが汚れてきたら、新しい会話を始める。

```
兆候:
  - AIが同じミスを繰り返す
  - 初期の指示を忘れたような振る舞いをする
  - 試行錯誤の過程が多く蓄積されている

対処:
  - 現時点の成果をファイルに書き出す
  - 新しい会話を開始する
  - ファイルだけを渡して続きを指示する
```

## アンチパターン

### 1. コンテキストの過積載

```
❌ 「overview.md と features/ と domain_model.md と ADR 4つと
    detailed_design.md を全部読んで、実装してください」

✅ 「detailed_design.md の Section 3 を実装してください。
    ユビキタス言語は spec/domain/ubiquitous_language.md を参照」
```

必要な情報だけ渡す。詳細設計に情報が集約されていれば、上流の成果物まで遡る必要はない。

### 2. コンテキストなしの丸投げ

```
❌ 「予約機能を作って」

✅ 「spec/features/reservation.feature のシナリオを満たす実装をしてください。
    技術スタックは spec/adr/001_tech_stack.md を参照」
```

コンテキストが少なすぎると、AIは推測で補完する。推測が正しいとは限らない。

### 3. 会話内での情報の揮発

```
❌ 壁打ちで良い結論が出た → ファイルに書き出さずに次の作業へ
   → 会話が長くなり、AIが壁打ちの結論を忘れる

✅ 壁打ちで良い結論が出た → overview.md に書き出す
   → 以降は overview.md を参照するだけでよい
```

**結論はファイルに書き出す。** これがコンテキストエンジニアリングの最も基本的な実践である。

## このガイドの成果物とコンテキストエンジニアリングの対応

このガイドが定義する成果物は、コンテキストエンジニアリングの観点で以下の役割を持つ。

| 成果物 | コンテキストとしての役割 | 渡すタイミング |
|--------|---------------------|-------------|
| `overview.md` | プロジェクト全体の要約。新しい会話の起点 | 詳細設計の生成時 |
| `features/` | 振る舞いの仕様。受け入れ条件として機能 | 実装時・テスト生成時 |
| `domain/` | ドメインの構造。用語と境界の定義 | 実装時（特にドメイン層） |
| `adr/` | 判断の記録。「なぜ」の文脈を提供 | 設計判断が必要な場面 |
| `detailed_design.md` | 実装仕様の集約。最も直接的な実装指示 | 実装時（主たるコンテキスト） |
| 設定ファイル | 永続的な前提・規約。毎回自動で読まれる | 常時（自動） |

> **ポイント:** すべてを一度に渡す必要はない。
> フェーズと作業内容に応じて、必要な成果物だけを渡す。

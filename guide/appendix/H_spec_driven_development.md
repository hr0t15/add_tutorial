# Appendix H: スペック駆動開発（Spec-Driven Development）

> **注意:** スペック駆動開発は2025年頃から注目を集めている新しいアプローチであり、
> 定義・手法・ツールチェーンのいずれもまだ確立途上にある。
> このAppendixでは、現時点で見えている方向性と、このガイドとの関係を整理する。
> 今後のエコシステムの成熟に伴い、内容が大きく変わる可能性がある。

---

## スペック駆動開発とは

スペック駆動開発（Spec-Driven Development, SDD）は、**自然言語やMarkdownで書かれた仕様書を開発の唯一の信頼できる情報源（Single Source of Truth）として扱い、そこからコード・テスト・ドキュメントを生成する**アプローチである。

従来の開発では、仕様書とコードは独立した成果物であり、時間の経過とともに乖離する（仕様書の腐敗）。
スペック駆動開発は、この問題を「仕様から自動的にコードを導出する」ことで解消しようとする。

```
従来:
  仕様書 → 人間が読んで実装 → コード
  → 仕様変更時にコードを手動で修正（乖離が発生しやすい）

スペック駆動:
  仕様書（Spec）→ AI/ツールがコード・テストを生成 → コード
  → 仕様変更時にSpecを更新 → コード・テストを再生成
```

---

## なぜ今注目されているのか

スペック駆動開発が注目される背景には、以下の技術的・文化的な変化がある。

### 1. LLMの「仕様からコードを生成する」能力の向上

2024〜2025年にかけて、LLM（大規模言語モデル）のコード生成能力が大きく向上した。
特に、構造化された仕様書からの生成精度が高まり、
「仕様を正確に書ければ、実装の大部分をAIに委譲できる」状況が現実になりつつある。

### 2. 「コードを書く」から「仕様を書く」へのシフト

コーディングAI（Claude Code, Cursor等）の普及により、
開発者の主な活動が「コードを書くこと」から「AIに何を作らせるかを定義すること」に移行しつつある。
この「何を作らせるか」の定義が、まさにスペック（仕様）である。

### 3. コンテキストエンジニアリングの成熟

`appendix/G_ai_adoption_insights.md` で述べたDORA 2025の「コンテキストエンジニアリング」の概念は、
スペック駆動開発の基盤となる考え方と共通する。
AIに渡す文脈を構造的に設計することが、生成品質を左右する。

## スペック駆動開発の基本的なフロー

現時点で見えているスペック駆動開発の典型的なフローを整理する。
ただし、これは一つの型であり、プロジェクトやツールチェーンによって変わる。

```
① 仕様の記述
     └─ 自然言語・Markdown・構造化フォーマットで「何を作るか」を書く
          ↓
② 仕様のレビュー・検証
     └─ 人間が仕様をレビューする
     └─ AIに仕様の矛盾・抜け漏れを指摘させる
          ↓
③ コード・テストの生成
     └─ 仕様からコード・テストを自動生成する
     └─ AIが仕様を読んで実装する
          ↓
④ 検証
     └─ 生成されたコードが仕様を満たしているかをテストで確認
     └─ 仕様 ↔ 実装の一致を検証する
          ↓
⑤ 仕様の更新（ループ）
     └─ 仕様を変更 → ③に戻る
```

## このガイドとの関係

このガイドが採用している「要求定義 → 詳細設計 → 実装」のフローは、
スペック駆動開発の考え方と多くの共通点を持つ。

### 共通する思想

| 観点 | このガイド | スペック駆動開発 |
|------|----------|---------------|
| 仕様が先 | 要求定義（フェーズ1）を先に固める | Specを先に書く |
| AIへの入力 | 構造化された設計書をAIに渡す | Specからコードを生成する |
| 人間の役割 | 上流の判断は人間が担う | Specの記述・レビューは人間が担う |
| 変更管理 | 仕様変更 → 詳細設計再生成 → 再実装 | Spec変更 → コード再生成 |

### 異なる点

| 観点 | このガイド | スペック駆動開発 |
|------|----------|---------------|
| 仕様のフォーマット | BDD（Gherkin）・DDD成果物等、複数の形式 | 単一のSpec形式を志向する傾向 |
| 生成の範囲 | AIに「依頼」して生成する（人間がトリガー） | 仕様変更に連動して自動生成を志向 |
| 発見プロセス | 実例マッピング・イベントストーミング等の発見手法を重視 | 発見プロセスの体系化はまだ弱い |
| 成熟度 | BDD・DDDは20年以上の実践知がある | 手法・ツールチェーンとも確立途上 |

### このガイドの位置づけ

このガイドは、スペック駆動開発の「仕様を正確に書く」部分に
BDD・DDDという成熟した手法を提供する立場にある。

```
スペック駆動開発が問う: 「仕様を書けば、コードは生成できる」
このガイドが答える: 「では、良い仕様はどう書くのか？」

  実例マッピング → Gherkinシナリオ（良い振る舞い仕様）
  イベントストーミング → DDD成果物（良いドメイン仕様）
  ADR → 判断記録（良い設計仕様）
```

つまり、スペック駆動開発が成熟していく中でも、
このガイドの発見プロセス（実例マッピング、イベントストーミング等）と
成果物フォーマット（Gherkin、DDD成果物等）は引き続き有効であると考える。

### このガイド自身の弱み

スペック駆動開発との比較において、このガイドには以下の弱みがある。
公平のために明示しておく（`01_philosophy.md` の「このガイド自身の限界と課題」も参照）。

**1. 成果物セットの重さが検証されていない**

このガイドは最大5種類の成果物（overview.md, features/, domain/, adr/, frontend/）を定義している。
しかし、**「この規模ならこの組み合わせで十分」という最小有効セットが確立されていない。**

スペック駆動開発が「単一のSpec」で完結しようとするのに対し、
このガイドは複数の成果物を組み合わせるため、導入コストが高い可能性がある。

```
問い: 本当に全部必要か？
  - overview.md + features/ だけで8割の効果が出るなら、残りは過剰投資かもしれない
  - domain/ の作成コストに見合うだけAIの実装精度が上がるのかは未測定
  - 1人開発でADRを書くコストは、後から「なぜ」を思い出すコストより本当に小さいか？
```

**2. カバーしていない領域がある**

| 足りない要素 | 説明 |
|------------|------|
| 非機能要件の仕様化 | 発見（overview.md の観点リスト）と判断記録（ADR）は整備したが、自動検証の体系化は未対応 |
| 仕様の腐敗防止 | 仕様とコードの乖離を検出する仕組みが手動のまま。スペック駆動が目指す自動同期に対して弱い |
| レガシーへの導入パス | 既存コードベースにこのプロセスを途中から適用する移行方法が未定義 |
| 運用設計 | 実装後の運用・監視をどう仕様に組み込むかの指針がない |

**3. 発見プロセスの効果が未検証**

実例マッピングやイベントストーミングをAIとの協調開発の文脈で用いたとき、
仕様の精度がどの程度向上するかの定量的エビデンスがない。
「BDD・DDDは20年の実践知がある」と言っても、それはAI以前の文脈であり、
AI時代にもそのまま通用する保証はない。

**4. AIの進化によって陳腐化するリスク**

AIが仕様の曖昧さを自ら補完できるようになった場合、
「構造化された仕様を人間が書く」というこのガイドの前提自体が過剰になる可能性がある。
その場合、スペック駆動開発の「自然言語で十分」というアプローチの方が正しくなる。

> このガイドは「現時点のベストプラクティス仮説」であり、完成品ではない。
> 弱みを認識した上で使い、実践を通じて更新していくことを推奨する。

## 現時点での課題と不確定要素

スペック駆動開発はまだ確立途上であり、以下の課題がある。

### 仕様のフォーマットが定まっていない

「何をSpecとして書くか」の標準がない。
自然言語のMarkdown、Gherkin、OpenAPI、独自DSLなど、
プロジェクトやツールによって異なる。

```
候補:
  - 自然言語Markdown（柔軟だが曖昧になりやすい）
  - Gherkin（振る舞いに特化、構造的）
  - OpenAPI / GraphQL Schema（API定義に特化）
  - 独自DSL（ツール依存）
  - 上記の組み合わせ
```

このガイドは「複数の成果物を組み合わせる」アプローチを採っており、
単一フォーマットに収束するかどうかは今後の動向次第である。

### 生成と検証のギャップ

「仕様からコードを生成する」ことと「生成されたコードが仕様を満たしているか検証する」ことの間には
まだギャップがある。特に以下が課題。

- **仕様の曖昧さ:** 自然言語の仕様は解釈の幅があり、生成されたコードが「仕様通り」かの判断が難しい
- **非機能要件:** パフォーマンス・セキュリティ等の非機能要件は仕様化・自動検証が困難
- **暗黙の前提:** 仕様に書かれていない暗黙の前提をAIが正しく補完できるとは限らない

### ツールチェーンの未成熟

2025年時点では、スペック駆動開発を端から端まで支援するツールチェーンは限られる。
多くの場合、以下を手動で組み合わせている。

```
仕様の記述:     エディタ + Markdown
仕様のレビュー:  PR レビュー
コード生成:     コーディングAI（Claude Code, Cursor等）
テスト:         既存のテストフレームワーク
仕様↔実装の同期: 手動
```

「仕様を変更したら自動的にコードが更新される」というビジョンは、
まだ部分的にしか実現されていない。

### 発見プロセスの体系化が弱い

スペック駆動開発は「仕様を書いてから」の話が中心で、
「良い仕様をどうやって発見するか」の体系化が弱い。

BDDの実例マッピングやDDDのイベントストーミングのような
**発見プロセスとの統合**が、スペック駆動開発の次の進化に必要だと考えられる。

## 実務での取り入れ方

現時点でスペック駆動開発の考え方を実務に取り入れるなら、以下を推奨する。

### 今すぐできること

1. **仕様を先に書く習慣を持つ** — コードを書く前に、何を作るかをテキストで定義する。
   このガイドのフェーズ1がまさにこれである

2. **仕様とコードの対応を意識する** — featureファイルのシナリオがテストに対応し、
   domain_model.md がコードの構造に対応する関係を維持する

3. **仕様変更時に影響範囲を意識する** — `03_directory_structure.md` の「変更の影響範囲」の考え方を実践する

### 今はまだ早いこと

1. **完全な自動生成パイプラインの構築** — ツールチェーンが未成熟なため、投資対効果が不明確
2. **独自のSpec DSLの策定** — 標準が定まっていない中での独自規格は将来の負債になるリスクがある
3. **仕様の自動検証** — 仕様 ↔ 実装の自動的な整合性チェックは研究段階

---

## 今後の注目ポイント

スペック駆動開発の動向を追う上で、以下のポイントに注目するとよい。

- **Specフォーマットの標準化の動き:** 業界全体で合意されるフォーマットが出てくるか
- **仕様↔コードの双方向同期:** コードの変更が仕様に反映される（逆方向の同期）が実現するか
- **AIによる仕様の検証:** 仕様の矛盾・抜け漏れをAIが自動検出する精度の向上
- **発見プロセスとの統合:** 実例マッピングやイベントストーミングとスペック駆動のワークフローの融合
- **DORA等の大規模調査による実証:** スペック駆動開発が実際にデリバリー指標を改善するかのエビデンス

### 直接的な関連

- [swyx, "Why Spec-Driven Development Will Define the AI Era"](https://www.latent.space/p/spec-driven-development) — Latent Space。スペック駆動開発の概念を広めた記事。「仕様が新しいコードである」という主張

### 背景となる思想

- [DORA State of AI-assisted Software Development 2025](https://dora.dev/research/2025/dora-report/) — AIが組織の既存の強さを増幅する（Mirror Effect）。構造化された知識がAIの効果を左右する
- [Martin Fowler, "Specification By Example"](https://martinfowler.com/bliki/SpecificationByExample.html) — 実例による仕様記述。BDDの文脈で「仕様 = テスト = ドキュメント」という考え方
- [Gojko Adzic, "Specification by Example" (2011)](https://gojko.net/books/specification-by-example/) — 実例による仕様の体系的な解説。スペック駆動開発の思想的な先駆

### AIとコード生成

- [Anthropic, "Claude Code"](https://docs.anthropic.com/en/docs/claude-code) — 仕様を渡してコードを生成するワークフローの実践ツール

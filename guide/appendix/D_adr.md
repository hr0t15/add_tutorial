# Appendix D: ADR（アーキテクチャ決定記録）詳細ガイド

## ADRとは

ADR（Architecture Decision Record）は、技術的・設計的な判断を「なぜそう決めたか」とセットで記録するドキュメントである。

「何を決めたか」だけでなく「なぜ決めたか」「何を選ばなかったか」を残すことが最大の価値。
時間が経つと「なぜこうなっているのか」が分からなくなるのを防ぐ。

AIとの協調開発では、**ADRがあるとAIへの指示精度が上がる**。
判断の背景ごと渡せるため、意図に反した実装が減る。

---

## ADRのテンプレート

```markdown
# ADR-XXX: タイトル（何を決めたかを一言で）

## ステータス
採用済み / 提案中 / 廃止 / 置き換え済み（ADR-YYYに置き換え）

## 文脈
どういう状況・制約の中でこの判断が必要になったか。
背景・前提・制約を書く。

## 決定
何を選んだか。1〜2文で明確に書く。

## 理由
なぜそれを選んだか。
選ばなかった選択肢とその理由も必ず書く。

## 影響
この決定によって生じる制約・影響・トレードオフ。
良い影響も悪い影響も書く。

## 参照
関連するADR・仕様書・外部リソース
```

---

## 各項目の書き方詳細

### ステータス

| ステータス | 意味 |
|-----------|------|
| **提案中** | まだ決定していない。レビュー中 |
| **採用済み** | 決定して実装済み、または実装方針確定 |
| **廃止** | 採用したが取り消した |
| **置き換え済み** | 別のADRに置き換えられた |

### 文脈（最重要）

「なぜこの判断が必要になったか」を書く。制約・前提・トレードオフが何であったかを明示する。

```markdown
# 良い例
## 文脈
予約システムで同一時間帯に同じスタッフへの二重予約を防ぐ必要がある。
複数ユーザーが同時に予約操作を行う可能性があり、競合制御が必要。
予約の整合性を担保しつつ、ユーザー体験を損なわないことが望ましい。

# 悪い例
## 文脈
二重予約を防ぎたい。
```

### 理由（最重要）

**選ばなかった選択肢とその理由を必ず書く。** これがないとADRの価値が半減する。

```markdown
# 良い例
## 理由
- 楽観的ロックはDBレベルで実装でき、追加インフラが不要
- 予約の競合頻度は低いため、楽観的ロックで十分な衝突検出が可能
- 悲観的ロックはDBコネクションを長時間占有し、スケーラビリティに難があるため不採用
- 分散ロック（Redis等）は追加インフラが必要で、現時点の規模には過剰なため不採用

# 悪い例
## 理由
楽観的ロックが良さそうだから。
```

## ADRを書くタイミング

| タイミング | 説明 |
|-----------|------|
| **決定した直後** | 最優先。時間が経つと理由を忘れる |
| **複数の選択肢で迷ったとき** | 迷った記録自体が価値になる |
| **後から「なぜこうなっているのか」と疑問を持ったとき** | 遡って書く |
| **既存の決定を覆すとき** | 古いADRを廃止にして新しいADRを書く |

### 非機能要件の判断もADRに記録する

非機能要件にはトレードオフを伴う判断が多い。これらもADRの対象である。
overview.md で「何が必要か」を発見し、ADRで「なぜその水準にしたか」を記録する。

> 非機能要件の洗い出し方は `02_development_process.md` の「非機能要件の洗い出し」を参照。

```markdown
# ADR-005: API応答時間をP95で200ms以内とする

## ステータス
採用済み

## 文脈
ユーザー調査で3秒以上の待機は離脱率が上がると分かった。
一方で、応答時間の目標を厳しくするほどインフラコストが増大する。
コストとUXのバランスを取る判断が必要。

## 決定
P95で200ms以内を目標とする。

## 理由
- 50ms以内は現行インフラでは過剰投資（CDN・エッジコンピューティングが必要）
- 500msではUX調査の結果から離脱リスクがある
- 200msは既存のDB構成＋キャッシュで達成可能な範囲

## 影響
- キャッシュ戦略が必須になる
- バッチ処理とリアルタイム処理の分離が必要
- 今後ユーザー数が10倍になった場合は再検討が必要（→ その際は新しいADRを書く）
```

```markdown
# ADR-006: 個人情報をカラムレベルでAES-256暗号化する

## ステータス
採用済み

## 文脈
個人情報保護法への対応として、個人を特定できる情報（氏名・電話番号・メールアドレス）を
保存時に暗号化する必要がある。暗号化の粒度と方式を決める必要がある。

## 決定
DBのカラムレベル暗号化（AES-256）を採用する。

## 理由
- ディスク暗号化（TDE）のみではDBアクセス権があれば平文で読めてしまう
- アプリケーションレベル暗号化は検索・ソートが困難でパフォーマンス影響が大きい
- カラムレベル暗号化はDB機能で完結し、アプリケーション変更が最小限

## 影響
- 暗号化カラムへの部分一致検索ができなくなる（完全一致のみ）
- 暗号化/復号化のオーバーヘッド（P95で+10ms程度の見込み、ADR-005の目標内）
- 鍵管理の運用が必要になる

## 参照
- ADR-005: API応答時間をP95で200ms以内とする（パフォーマンス影響の制約）
```

### 書かなくていいもの

- 自明な決定（「Goで書く」と決まっているのに「なぜGoを使うか」は不要）
- 一時的な決定（後で必ず変わると分かっているもの）
- 実装の詳細（ADRはアーキテクチャレベルの判断を記録する）

## ファイルの命名規則と管理

```
documents/adr/
├── ADR-001_（決定内容の概要）.md
├── ADR-002_（決定内容の概要）.md
└── ADR-003_（決定内容の概要）.md
```

- 番号は連番にする
- ファイル名に決定内容の概要を入れる（一覧を見ただけで内容が分かるように）
- ADR間の参照を `## 参照` セクションに記載する

### ADRを廃止・置き換えるとき

古いADRは削除しない。ステータスを変更して残す。

```markdown
# ADR-003: 楽観的ロックで二重予約を防止する

## ステータス
置き換え済み（ADR-007: 分散ロックで二重予約を防止する に置き換え）

## 文脈
...（元の内容はそのまま残す）
```

## 雑多なメモからのADR生成

設計議論や壁打ちで出た判断メモから、ADRのフォーマットに整理できる。

```
以下は設計判断についての議論メモです。
 これをADR形式に整理してください。

 フォーマット:
 - タイトル: 何を決めたかを一言で
 - ステータス: 採用済み or 提案中
 - 文脈: なぜこの判断が必要になったか（制約・前提・背景）
 - 決定: 何を選んだか
 - 理由: なぜそれを選んだか。選ばなかった選択肢とその理由も必ず含める
 - 影響: この決定によるトレードオフ
 - 参照: 関連するADR・仕様

 メモの中で理由が明示されていない判断があれば、
 『理由が不明確です。以下のどれかでしょうか？』と候補を提示してください。

 ---
 [議論メモ・壁打ちログを貼る]
```

複数の判断が混在するメモの場合:

```
以下のメモには複数の設計判断が含まれています。
 判断ごとに分割して、それぞれ別のADRとして整理してください。
 ADR間に依存関係がある場合は、参照セクションで相互にリンクしてください。

 ---
 [議論メモを貼る]
```

> **ポイント:** 議論の直後にADRを書くのが理想だが、現実には後回しになりがちである。
> メモだけでも残しておけば、後からAIに整理を依頼できる。
> 「理由が不明確」と指摘されたら、それは記憶が薄れる前に埋めるべきサインである。

---

## コーディングAIへの渡し方

ADRをAIに渡すことで、過去の判断を文脈として理解させられる。

```
ADR-002に従い、二重予約防止には楽観的ロックを使ってください。
 予約作成時にバージョン番号をチェックする実装にしてください。
```

また、新しい実装を依頼するときにADR一覧を渡すと、過去の判断と矛盾しない実装が生まれやすい。

```
documents/adr/ のすべてのADRを読んだ上で、
 以下の機能を実装してください: ...
```

## 参考文献

### ADRの原典

- [Michael Nygard, "Documenting Architecture Decisions" (2011)](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions) — ADRの提唱者による原典記事。ADRの形式と「なぜ記録するか」の理由を解説
- [Michael Nygard, "Architecture Decision Records"](https://adr.github.io/) — ADRの公式リファレンスサイト。テンプレート・FAQ・採用事例

### テンプレート・ツール

- [Joel Parker Henderson, "Architecture Decision Record Templates"](https://github.com/joelparkerhenderson/architecture-decision-record) — 多様なADRテンプレート集。Nygard形式・MADR形式・Y-Statements形式等を収録
- [Nat Pryce, "adr-tools"](https://github.com/npryce/adr-tools) — ADRの作成・管理を支援するCLIツール。連番管理・ステータス変更・リンク管理を自動化
- [MADR (Markdown Any Decision Records)](https://adr.github.io/madr/) — MarkdownベースのADRテンプレート。より構造化された形式を提供

### 背景・採用事例

- [ThoughtWorks Technology Radar](https://www.thoughtworks.com/radar/techniques/lightweight-architecture-decision-records) — ThoughtWorksがLightweight ADRを「Adopt（採用推奨）」に分類。業界での評価
- [Spotify Engineering, "When Should I Write an Architecture Decision Record"](https://engineering.atspotify.com/2020/04/when-should-i-write-an-architecture-decision-record/) — Spotifyでの実践事例。書くタイミング・粒度の判断基準

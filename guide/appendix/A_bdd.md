# Appendix A: BDD（振る舞い駆動開発）

## BDDとは

BDD（Behavior-Driven Development）は、「システムがどう振る舞うか」を中心に開発を進める手法である。
ユーザー・開発者・AIの全員が共通の言語でシステムを語れるようにすることが目的。

テストを書くための手法ではなく、**コミュニケーションの手法**として理解するのが正確である。
結果としてテストが生まれる。

BDDには3つのプラクティスがある。

```
Discovery（発見）  → 何を作るべきかを具体例で発見する
Formulation（定式化）→ 発見した例をGherkin形式に定式化する
Automation（自動化） → 定式化したシナリオを自動テストにする
```

**最も重要なのはDiscovery（発見）であり、Gherkinを書くこと自体ではない。**
Gherkinはあくまで発見の成果物を記録する形式である。

## BDDの発見プロセス: 実例マッピング（Example Mapping）

### 実例マッピングとは

実例マッピング（Example Mapping）は、Matt Wynneが2015年に提唱した
**ユーザーストーリーを具体例で掘り下げるワークショップ手法**である。

実例マッピングの入力は**ユーザーストーリー**である。
`02_development_process.md` の「機能要件をユーザーストーリー形式で書く」で作成したストーリー
（`As a [役割], I want [目的], so that [理由]`）を1つ取り上げ、
25分程度の短いセッションで以下の4つの要素を使って分解する。

| 要素 | 色（カード） | 説明 |
|------|------------|------|
| **ストーリー** | 黄色 | 検討対象のユーザーストーリーや機能 |
| **ルール** | 青色 | ストーリーの中にある業務ルール・制約 |
| **例（Example）** | 緑色 | ルールを具体化する実例。Given/When/Then で表現できる |
| **疑問** | 赤色 | セッション中に解決できなかった未決事項 |

```
ストーリー: 「カードを隣接カラムに移動できる」

  ルール1: 隣接カラムにのみ移動できる
    例: TodoからIn Progressに移動できる ✅
    例: TodoからDoneには直接移動できない ❌

  ルール2: 逆方向にも移動できる
    例: DoneからIn Progressに戻せる ✅

  疑問: 「In ProgressからTodoに戻すことはできるか？」
```

### なぜ実例マッピングが先か

Gherkinシナリオをいきなり書こうとすると、以下の問題が起きやすい。

- ルールが暗黙のまま、例だけが列挙される
- 抜け漏れに気づかないまま正常系だけ書いてしまう
- シナリオの粒度がバラバラになる

実例マッピングを先にやることで、**ルールが明示され、例が体系的に整理され、疑問が可視化される**。

### テキストベースでの実例マッピング

物理カードが使えない場合、Markdownで実施できる。

```markdown
# 実例マッピング: カード管理

## ストーリー
カードを作成・移動・削除してタスクを管理できる

## ルールと例

### ルール: カードは隣接カラムにのみ移動できる
- 例: TodoのカードをIn Progressに移動できる → 成功
- 例: TodoのカードをDoneに直接移動しようとする → 失敗
- 例: DoneのカードをIn Progressに戻せる → 成功

### ルール: カードにはタイトルが必須
- 例: タイトル「買い物リスト」でカード作成 → 成功
- 例: タイトル空でカード作成 → 失敗、エラー表示

### ルール: カードは作成時にTodoに配置される
- 例: カードを作成する → Todoカラムに表示される

## 疑問
- [ ] In ProgressからTodoに戻せるか？
- [ ] カードのタイトルに文字数上限はあるか？
```

このテキストが、Gherkinシナリオの**設計図**になる。

### AIとの実例マッピング

AIを壁打ち相手として実例マッピングを進められる。

```
以下のストーリーについて実例マッピングを一緒にやりたいです。

ストーリー: テーブルの注文を入力できる

ルール・例・疑問を洗い出してください。
特に『このルールが破られたら何が起きるか？』の例を重視してください
```

AIはルールの候補を列挙し、各ルールに対する例（正常系・異常系）を提案する。
**人間はルールの妥当性と例の正しさを判断する。**

### 雑多なメモからの実例マッピング生成

壁打ちで出た雑多なメモや議論ログから、実例マッピングのフォーマットに整理できる。

```
以下は機能についての壁打ちメモ・議論ログです。
 これを実例マッピングの形式に整理してください。

 フォーマット:
 - ストーリー: 1文で
 - ルールと例: ルールごとに、成功する例・失敗する例を列挙
 - 疑問: メモの中で結論が出ていないもの、暗黙の前提になっているもの

 メモに書かれていないが当然ありそうなルール（暗黙のルール）も候補として提示してください。

 ---
 [壁打ちメモ・議論ログを貼る]
```

実例マッピングが整理されたら、続けてGherkinへの変換を依頼できる。

```
上の実例マッピングの内容を、そのままGherkin形式の.featureファイルに変換してください。
 ルールごとにコメントで区切ってください。
 疑問の項目は # TODO コメントとして残してください。
```

> **ポイント:** 2段階に分けることで、「メモ → 実例マッピング」の段階で人間がルールの妥当性を確認でき、
> いきなりGherkinが出てくるよりもレビューしやすくなる。

## 実例マッピングからGherkinへ

### 変換の流れ

実例マッピングの成果物は、ほぼ機械的にGherkinに変換できる。

```
実例マッピング           Gherkin
─────────────          ──────────
ストーリー          →   Feature
ルール              →   コメント or Rule キーワード
例（成功）          →   Scenario（正常系）
例（失敗）          →   Scenario（異常系）
疑問（解決済み）     →   追加のScenario
疑問（未解決）       →   TODO コメント
```

### 変換の具体例

実例マッピング:
```
ルール: カードは隣接カラムにのみ移動できる
  例: TodoからIn Progressに移動できる → 成功
  例: TodoからDoneに直接移動しようとする → 失敗
```

↓ Gherkin:

```gherkin
# ルール: カードは隣接カラムにのみ移動できる

Scenario: カードをTodoからIn Progressに移動する
  Given Todoカラムに「タスクA」がある
  When 「タスクA」をIn Progressに移動する
  Then In Progressカラムに「タスクA」が表示される

Scenario: TodoからDoneに直接移動できない
  Given Todoカラムに「タスクA」がある
  When 「タスクA」をDoneに移動しようとする
  Then 移動は行われない
  And Todoカラムに「タスクA」が残っている
```

### AIによるGherkin生成支援

実例マッピングのテキストを渡せば、AIがGherkinに変換できる。

```
以下の実例マッピングの成果を、正式なGherkin形式のfeatureファイルに変換してください。
 ルールごとにコメントで区切ってください。
 1シナリオ = 1つの例 にしてください。

[実例マッピングのテキストを貼る]
```

> **ポイント:** 実例マッピングなしにいきなり「Gherkinシナリオを書いてください」と
> AIに頼むと、ルールが暗黙のままシナリオが生成される。
> 実例マッピングを先にやることで、**ルールが明示的になり、シナリオの網羅性が上がる**。

## Gherkin形式の基本

BDDのシナリオはGherkin形式で記述する。

```gherkin
Feature: 機能の名前
  この機能が何のためにあるかを1〜2行で書く

  Background:
    Given 全シナリオに共通する前提条件

  Scenario: シナリオの名前（具体的に書く）
    Given 前提となる状態
    When  ユーザーが行うアクション
    Then  期待される結果
    And   追加の期待される結果
    But   期待されない結果（否定）

  Scenario Outline: パラメータ化されたシナリオ
    Given <入力> を渡した
    When  処理を実行する
    Then  <期待値> が返る

    Examples:
      | 入力     | 期待値   |
      | "hello"  | "HELLO"  |
      | "world"  | "WORLD"  |
```

### キーワードの使い分け

| キーワード | 役割 | 使うタイミング |
|-----------|------|-------------|
| `Feature` | 機能の名前と説明 | ファイルの先頭に1つ |
| `Background` | 全シナリオ共通の前提 | 繰り返しを避けたいとき |
| `Scenario` | 1つの振る舞いのシナリオ | 具体的な1ケースごとに |
| `Given` | 前提状態 | 「〜の状態で」 |
| `When` | アクション | 「〜したとき」 |
| `Then` | 期待される結果 | 「〜になる」 |
| `And` | 前の行と同じ種類の続き | Given/When/Thenの継続 |
| `But` | 否定の結果 | 「〜にはならない」 |
| `Scenario Outline` | パラメータ化されたシナリオ | 同じ構造で複数ケースを書くとき |

## featureファイルの作り方

### ファイル構成

```
documents/features/
├── 01_（主要機能）.feature
├── 02_（前処理系）.feature
├── 03_（出力系）.feature
└── 04_（エラー系）.feature
```

番号をつけて処理の流れ順に並べると読みやすい。

### 1ファイルの構成ルール

- 1ファイル = 1機能（Feature）
- 1シナリオ = 1つの振る舞い（シナリオを詰め込みすぎない）
- 正常系・異常系を同じファイルにまとめる

### 良いシナリオの書き方

**具体的に書く**
```gherkin
# 悪い例（抽象的すぎる）
When 処理を実行する
Then 結果が表示される

# 良い例（具体的）
When 「タスクA」をIn Progressに移動する
Then In Progressカラムに「タスクA」が表示される
```

**UIの操作詳細を書きすぎない**
```gherkin
# 悪い例（実装詳細が入りすぎ）
When id="submit-button" のボタンをクリックする

# 良い例（振る舞いに集中）
When 「カード作成」ボタンをクリックする
```

**1シナリオに複数のWhenを入れない**
```gherkin
# 悪い例（アクションが複数）
When カードを作成する
And In Progressに移動する
And Doneに移動する

# 良い例（1つのシナリオに1つの振る舞い）
# → それぞれ別のシナリオに分ける
```

## シナリオの種類と網羅の考え方

### 正常系
最も基本的な「うまくいくケース」。最初に書く。

```gherkin
Scenario: カードをTodoからIn Progressに移動する
  Given Todoカラムに「タスクA」がある
  When  「タスクA」をIn Progressに移動する
  Then  In Progressカラムに「タスクA」が表示される
```

### 異常系
エラーが発生するケース。2周目以降に追加する。

```gherkin
Scenario: タイトル空でカードを作成しようとする
  Given カード作成画面を開いている
  When  タイトルを空のまま作成ボタンを押す
  Then  「タイトルは必須です」エラーが表示される
  And   カードは作成されない
```

### 境界値・エッジケース
イベントストーミングや実例マッピングで気づくことが多い。

```gherkin
Scenario: TodoからDoneに直接移動しようとする
  Given Todoカラムに「タスクA」がある
  When  「タスクA」をDoneに移動しようとする
  Then  移動は行われない
  And   Todoカラムに「タスクA」が残っている
```

## コーディングAIへの渡し方

featureファイルは、そのままAIへの実装指示として使える。

```
documents/features/01_card_management.feature のすべてのシナリオを満たす実装をしてください。
 documents/specification/architecture.md の構成に従ってください。
```

シナリオが具体的であるほど、AIの実装精度が上がる。

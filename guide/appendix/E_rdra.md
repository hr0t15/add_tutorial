# Appendix E: RDRA（リレーションシップ駆動要件分析）

## RDRAとは

RDRA（Relationship Driven Requirement Analysis）は、神崎善司氏（バリューソース社）が提唱した日本発の要件定義フレームワークである。
要件間の**「関係性」**を軸に、4つのレイヤーで要件を構造化する。

最大の特徴は**トレーサビリティ** — すべての機能が「なぜ必要か」を要求まで遡れる構造になっていること。

---

## 4つのレイヤー

```
① システム価値         — 誰が・なぜ使うか（アクター、要求）
② システム外部環境      — どんな業務・利用シーンがあるか（業務モデル、概念モデル）
③ システム境界         — システムとの接点は何か（ユースケース、画面、イベント）
④ システム             — 何を管理するか（機能、データモデル）
```

下位レイヤーから上位レイヤーに「なぜ？」と辿れる依存関係がある。
これにより「この機能はなぜ存在するか」を構造的に説明できる。

---

## このガイドとの対応関係

このガイドの成果物は、RDRAの各レイヤーと以下のように対応する。

| RDRAレイヤー | 扱う問い | このガイドの対応する成果物 |
|------------|---------|----------------------|
| ① システム価値 | 誰が・なぜ使うか | 要件概要（目的・スコープ外） |
| ② システム外部環境 | どんな業務・利用シーンがあるか | 要件概要（機能要件）+ 振る舞い定義 |
| ③ システム境界 | システムとの接点は何か | UI設計 + 振る舞い定義 |
| ④ システム | 何を管理するか | ドメイン定義 |

個々の成果物は似た領域をカバーしているが、**決定的に違う点が1つある**。

---

## 最大の違い: トレーサビリティ

RDRAでは、どの画面がどの利用シーンに対応し、どの利用シーンがどの要求に対応するかが**構造的に追跡可能**になっている。

このガイドでは：
- 要件概要に「なぜ」がある
- 振る舞い定義に「ユーザーから見て何が起きるか」がある
- ドメイン定義に「裏で何が動くか」がある

しかし、**それらの間の対応関係は暗黙的**になりやすい。
例えば「このGherkinシナリオはoverview.mdのどの要求に対応するか」が明示されていない場合がある。

---

## トレーサビリティの導入方法

RDRAの本質的な価値を、このガイドの軽量さを壊さずに取り込むには、**要件概要に要求トレーサビリティ表を追加する**のが最も効果的である。

```markdown
## 要求トレーサビリティ

| 要求ID | 要求 | 対応するシナリオ | 対応するドメイン概念 |
|-------|------|----------------|-------------------|
| R1 | 注文を入力できる | 01_order.feature | Order |
| R2 | 調理状況を管理できる | 02_kitchen.feature | Dish |
| R3 | 会計処理ができる | 03_payment.feature | Payment |
```

この表があることで：
- **上から下:** 要求 → どのシナリオでテストされるか → どのドメイン概念が関与するか
- **下から上:** このドメイン概念は → どのシナリオに登場し → どの要求を満たすためにあるか

が構造的に辿れるようになる。

### 規模による導入判断

| 規模 | 推奨 |
|------|------|
| 1人・単機能 | なくても可。要件概要の中で自然に追跡できる規模 |
| 小チーム・複数機能 | 推奨。要求とシナリオの対応が見えなくなり始める規模 |
| 中規模以上 | 必須。要求の抜け漏れ検出に不可欠 |

---

## RDRAのテキストベース化とAI親和性

RDRAは元々UML拡張の図が中心だったが、近年テキストベースへの展開が進んでいる。

- **RDRA ZeroOne** — Web UIからRDRA形式のモデルデータを自動生成
- **RDRA Agent** — Claude Code上で動く要件定義エージェント
- **Cursor Agent版** — 有志による Cursor 対応版

テキスト表現が可能になったことで、**AIがアクセスしやすい形での要件構造化**が実現しつつある。
これは `appendix/G_ai_adoption_insights.md` で述べている「AI-accessible internal data」の具体的な実現手段と言える。

このガイドの成果物（要件概要、振る舞い定義、ドメイン定義、判断記録）もすべてテキストベースであり、RDRAと同じ方向性にある。

---

## このガイドに取り入れるべき要素・取り入れなくてよい要素

### 取り入れる価値が高い要素

| 要素 | 理由 |
|------|------|
| **トレーサビリティ表** | 要求 → シナリオ → ドメイン概念の対応を明示できる。抜け漏れの検出に直結する |
| **システム価値レイヤーの発想** | 要件概要の「アクター」「要求」の構造化を強化できる |
| **利用シーンの概念** | Gherkinシナリオより上位の粒度で、シナリオを書く前の整理に有効 |

### 取り入れなくてよい要素

| 要素 | 理由 |
|------|------|
| **RDRA固有のモデル記法** | このガイドはツール非依存を志向しているため |
| **4レイヤーの厳密な分離** | 小規模では過剰。このガイドの段階的導入方針と合わない |

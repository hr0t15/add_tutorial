# 06_01. GUI設計とテスト戦略

`06_ui_design.md` の共通原則を踏まえた、GUI（Web / モバイル）固有の設計成果物とテスト戦略。

---

## GUI の設計成果物

GUIの設計では、以下の成果物を作る。
これらはバックエンドの requirements/ と同様に「何を作るか」を定義するものである。

```
documents/requirements/frontend/
├── screens.md          # 画面一覧・URL定義
├── screen_flow.md      # 画面遷移図
├── components.md       # UIコンポーネント定義・状態定義
├── error_states.md     # エラー・空状態の定義
└── features/           # Gherkinシナリオ（フロントエンド）
    ├── 01_xxx.feature
    └── 02_xxx.feature
```

### 各成果物の役割

| 成果物 | 設計上の役割 | テストへの活用 |
|-------|------------|-------------|
| 画面一覧・URL定義 | 画面の全体像を把握する。URLの設計根拠になる | テスト対象の画面・フローが確定する |
| 画面遷移図 | 画面間の関係と遷移条件を明確にする | E2Eテストのフロー設計の根拠になる |
| UIコンポーネント定義 | 共通部品の責務・状態を定義する | コンポーネントテストの設計根拠になる |
| エラー・空状態の定義 | 正常系以外の画面状態を網羅する | 異常系シナリオに直結する |
| Gherkinシナリオ | ユーザー操作の期待される振る舞いを定義する | そのままE2Eテストケースになる |

### なぜこの4つか

バックエンドでは「ドメインモデル」が設計の核になるが、
GUIでは**「画面とその状態」**が設計の核になる。

```
バックエンド: エンティティの状態遷移が正しいか？
GUI:         画面の状態遷移がユーザーの期待通りか？
```

この4つの成果物は、GUIにおける「状態」を異なる粒度で定義している。

```
screens.md       → アプリ全体の画面状態
screen_flow.md   → 画面間の遷移（マクロな状態遷移）
components.md    → 個別コンポーネントの内部状態（ミクロな状態遷移）
error_states.md  → 例外的な状態（エラー・空・ローディング等）
```

## GUI 向け Gherkin シナリオ

バックエンドと同じGherkin形式で書く（書き方の詳細は `appendix/A_bdd.md` を参照）。

```gherkin
Scenario: テーブルの注文を作成する
  Given ユーザーが注文画面にいる
  When  テーブルT1を選択する
  And   「カレー」を注文に追加する
  And   「注文確定」ボタンをクリックする
  Then  注文完了メッセージが表示される
  And   T1のステータスが「注文済み」になる
```

### バックエンドとの違い

GUIのシナリオは「画面上で何が起きるか」を記述する。

```
バックエンド: When 注文を確定する → Then 注文確定イベントが発行される
GUI:         When 「注文確定」ボタンをクリックする → Then 注文完了画面に遷移する
```

同じビジネスフローの表裏であり、両方が揃うことで仕様の抜け漏れが減る。

## テスト戦略

### テスト粒度の選択肢

GUIのテストには複数の粒度がある。
規模が小さいうちはシンプルなE2Eテストだけで十分。規模に応じて段階的に追加していく。

| テスト粒度 | 1人・単機能 | 小チーム・複数機能 | 中規模・複数チーム | 大規模・複雑UI |
|-----------|-----------|----------------|----------------|-------------|
| **E2Eテスト（正常系）** | ◎ 必須 | ◎ 必須 | ◎ 必須 | ◎ 必須 |
| **E2Eテスト（異常系）** | △ 代表的なものだけ | ○ 推奨 | ◎ 必須 | ◎ 必須 |
| **Page Objectパターン** | △ なくても可 | ○ 推奨 | ◎ 必須 | ◎ 必須 |
| **APIモック** | △ 任意 | ○ 推奨 | ◎ 必須 | ◎ 必須 |
| **コンポーネントテスト** | × 過剰 | △ 複雑なものだけ | ○ 推奨 | ◎ 必須 |
| **ビジュアルリグレッション** | × 過剰 | × 過剰 | △ 重要画面のみ | ○ 推奨 |
| **クロスブラウザテスト** | × 過剰 | △ リリース前のみ | ○ 推奨 | ◎ 必須 |
| **パフォーマンス計測** | × 過剰 | × 過剰 | △ 重要フローのみ | ○ 推奨 |

### 規模ごとの方針

**1人・単機能**
正常系のE2Eテストだけで十分。Page Objectは後から追加できるので最初は不要。

**小チーム・複数機能**
異常系シナリオを追加する。Page Objectを導入してセレクターを一元管理する。
APIモックで「バックエンドが不安定でもテストが通る」状態にする。

**中規模・複数チーム**
チームをまたぐ画面遷移はE2Eで必ずカバーする。コンポーネントテストで共有UIパーツの品質を担保する。
APIモックは必須（チームによってバックエンドの進捗が異なるため）。

**大規模・複雑UI**
全粒度を揃える。ビジュアルリグレッションでデザイン崩れを自動検出する。クロスブラウザはCIで自動実行する。

### 設計成果物とテストの対応

設計成果物のどれが、どのテスト粒度の根拠になるかを整理する。

```
screens.md + screen_flow.md
  → E2Eテスト（画面遷移が正しいか）

components.md
  → コンポーネントテスト（部品単体の振る舞いが正しいか）
  → Page Objectパターン（セレクター命名の根拠）

error_states.md
  → E2Eテスト異常系（エラー時の表示が正しいか）
  → APIモック（エラーレスポンスの定義）

features/
  → E2Eテスト（シナリオをそのままテストケースにする）
```

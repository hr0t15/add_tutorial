# 03. ディレクトリ構成

このドキュメントは、プロジェクトのディレクトリ構成とその設計思想を説明する。
「なぜこの構成にするのか」「各フォルダに何を置くのか」を理解するために読んでほしい。

---

## 全体構成

```
project_root/
├── documents/
│     ├── requirements/     # フェーズ1: 人間とAIが協調して作った設計群
│     └── specification/    # フェーズ2: AIが生成する詳細設計
└── implement/              # フェーズ3: 実装プロジェクト全体
```

`documents/` が設計に関するすべての成果物を束ねる。
`requirements/`（人間主導）と `specification/`（AI生成）を同じリポジトリで管理することも、
別リポジトリに分けることも選択できる柔軟な構成になっている。

---

## 責任の所在

```
要求定義（documents/requirements/）→ 人間が責任を持つ領域（変更には人間の判断が必要）
詳細設計（documents/specification/）→ AIが生成・人間がレビューする領域
実装（implement/）                  → AIに委譲できる領域
```

---

## 変更の影響範囲

仕様が変わったとき、何を再生成すべきかが自明になる。

```
仕様変更 → 要求定義を修正
           → 詳細設計を再生成
           → 実装を再実装
```

---

## documents/requirements/

人間とAIが協調して作った設計群。「何を作るか・なぜ作るか」を定義する。
変更には人間の判断が必要。

```
documents/requirements/
├── overview.md       # プロダクト要件の概要（What + Why + スコープ外）
├── features/         # Gherkinフィーチャーファイル
├── domain/           # ドメインモデル・ユビキタス言語
├── adr/              # アーキテクチャ決定記録（ADR）
└── frontend/         # フロントエンド関連（該当する場合）
    ├── screens.md
    ├── screen_flow.md
    ├── components.md
    ├── error_states.md
    └── features/     # Gherkinシナリオ（フロントエンド）
```

各ディレクトリ・ファイルは、人間の活動と1対1で対応している。

| 成果物 | 対応する人間の活動 |
|-------|----------------|
| overview.md | 何を作るかを言語化する（ヒアリング・壁打ちの成果） |
| features/ | 振る舞いを定義する（BDD） |
| domain/ | ドメインを構造化する（DDD） |
| adr/ | 判断を記録する（アーキテクチャ設計） |
| frontend/ | 画面と状態を設計する（UI設計） |

> **参照:** features/ の書き方は `appendix/A_bdd.md`、domain/ の作り方は `appendix/B_ddd.md`、
> overview.md での要求トレーサビリティは `appendix/E_rdra.md` を参照。

## documents/specification/

AIが要求定義を読み込んで生成する詳細設計。「どう作るか」を定義する。
人間がレビュー・承認したものをコミットする。

```
documents/specification/
├── architecture.md          # パッケージ構成・モジュール依存関係
├── test_plan.md             # テスト方針・カバレッジ計画
├── backend/                 # バックエンド詳細設計（該当する場合）
│   ├── api_design.md        # 内部インターフェース・データ構造定義
│   └── sequences/           # シーケンス図（処理フローの詳細）
└── frontend/                # フロントエンド詳細設計（該当する場合）
    ├── component_spec.md    # コンポーネントのprops・状態遷移の詳細
    └── api_integration.md   # API呼び出し・エラーハンドリングの設計
```

`architecture.md` と `test_plan.md` はバックエンド・フロントエンド横断の全体設計なので直下に置く。
`backend/` と `frontend/` は該当する場合のみ作成する。
UI設計（要求定義）が「画面として何が必要か」を定義するのに対し、
フロントエンド詳細設計（詳細設計）は「それをどう実装するか」を定義する。

---

## implement/

実装プロジェクト全体。ソースコードに加え、パッケージ管理・設定ファイルなども含む。
言語・フレームワークによって内部構成は変わる。

```
implement/
├── （言語・フレームワーク依存のファイル群）
├── go.mod / package.json 等
└── src/ 等
```

`src/` ではなく `implement/` としているのは、ソースコードだけでなくプロジェクト全体のベースディレクトリとして位置づけているためである。

---

## Gitリポジトリの管理単位

この構成は、プロジェクト規模に応じてリポジトリの分割方法を選択できる。

```
選択肢A: 単一リポジトリ（シンプル・小規模向け）
  project_root/ ← 1つのリポジトリ

選択肢B: 2リポジトリ分割（中規模向け）
  documents-repo/  ← 設計ドキュメント専用
  implement-repo/  ← 実装専用（submoduleで参照も可）

選択肢C: documents/を複数プロジェクトで共有（大規模向け）
  shared-documents-repo/ ← guide・チュートリアル等を横断管理
  project-A-impl/
  project-B-impl/
```

### コミット粒度の考え方

```
要求定義     → 人間の判断が入るたびにコミット（意思決定の記録）
詳細設計     → AI生成後のレビュー承認をコミット（生成物の受け入れ記録）
実装         → 機能単位・シナリオ単位でコミット
```

---

## ADRの書き方

ADR（Architecture Decision Record）は、技術的・設計的な判断を「なぜそう決めたか」とセットで記録するドキュメントである。`documents/requirements/adr/` に置く。

後から別の会話でAIに渡したとき、判断の背景ごと伝えられるため、意図に反した実装が減る。

```markdown
# ADR-XXX: タイトル

## ステータス
採用済み / 提案中 / 廃止 / 置き換え済み

## 文脈
どういう状況・制約の中でこの判断が必要になったか。

## 決定
何を選んだか。

## 理由
なぜそれを選んだか。選ばなかった選択肢とその理由も書く。

## 影響
この決定によって生じる制約・影響。

## 参照
関連するADR・仕様書
```

> **特に重要な項目:** 「理由」と「選ばなかった選択肢」。これがないとADRの価値が半減する。
>
> **参照:** ADRのテンプレート・管理方法の詳細は `appendix/D_adr.md` を参照。

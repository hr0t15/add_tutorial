# 02. 開発プロセス全体フロー

このドキュメントは、AIを活用した開発の全体的な流れと、各ステップで何をすべきかを説明する。
「どういう順番で何を作り、どう協調し、どこに置くか」を理解するために読んでほしい。

---

## 全体像

開発は3つのフェーズで構成される。上流ほど人間が主導し、下流ほどAIに委譲する。

```
フェーズ1: 要求定義を育てる（人間主導・AIと協調）
  └─ 「何を作るか・なぜ作るか」を固める

フェーズ2: 詳細設計を生成する（AIが生成・人間がレビュー）
  └─ 「どう作るか」を定義する

フェーズ3: 実装する（AIが主導・人間がレビュー）
  └─ 「作る」
```

フェーズを分けることで、各段階で渡す情報が絞られ、AIへの指示精度が上がる。

> この3フェーズの設計思想は `01_philosophy.md` の「コンテキストを設計するという考え方」に基づいている。
> ディレクトリの詳細は `03_directory_structure.md` を参照。

### 用語とディレクトリの対応

このガイドでは、成果物を指すときに以下の用語を使う。

#### 3つのフェーズ

| 用語 | 意味 | 対応するディレクトリ |
|------|------|-------------------|
| **要求定義** | 「何を作るか・なぜ作るか」を固めた成果物群 | `documents/requirements/` |
| **詳細設計** | 「どう作るか」を定義した設計書群 | `documents/specification/` |
| **実装** | ソースコード・テスト・設定ファイル群 | `implement/` |

#### 要求定義の構成要素

要求定義はフェーズ1の5つのステップに対応する成果物で構成される。

| 用語 | 意味 | 対応する場所 |
|------|------|------------|
| **要件概要** | プロダクトの目的・機能・制約・スコープ外 | `overview.md` |
| **振る舞い定義** | ユーザーから見て「こうなってほしい」振る舞いの仕様 | `features/` |
| **ドメイン定義** | ビジネスの構造と言葉を定義したドメインの仕様 | `domain/` |
| **判断記録** | 技術・設計判断とその理由（ADR） | `adr/` |
| **UI設計** | ユーザーが触れるインターフェースの設計（GUI: 画面と状態 / CLI: コマンドと入出力） | `frontend/` or `cli/` |

> **注意:** 振る舞い定義はユーザーの視点で書く。
> 「システムがDBに書き込む」ではなく「ユーザーが注文を確定すると、完了画面が表示される」。
> システム内部で何が起きるかはドメイン定義の役割である。

## フェーズ1: 要求定義を育てる

フェーズ1が最も時間をかけるべきフェーズであり、人間の判断が最も求められる。
以下の5つのステップをサイクルで回しながら、要求定義を育てていく。

### ステップの全体像

```
① 要件の言語化（壁打ち）
     └─ 「何を作るか」を言葉にする
          ↓
② 振る舞い定義（BDD）
     └─ Gherkinシナリオで「こうなってほしい」を書く
          ↓
③ ドメインの構造化（DDD）
     └─ イベントストーミングで裏側の構造を掘る
     └─ コマンド・イベント・集約を整理する
          ↓
④ 判断の記録
     └─ 技術選定・設計方針をADRとして残す
          ↓
⑤ UI設計
     └─ 画面・状態・遷移を定義する（②と並行して進めてもよい）
          ↓
新たな抜け漏れに気づいたら ① or ② に戻る
十分に収束したら → フェーズ2へ
```

### ステップの詳細

以下の表が、フェーズ1の各ステップで **「何をするか」「何が生まれるか」「どこに置くか」「どう協調するか」** をまとめたものである。

| ステップ | 人間の活動 | 成果物 | 置き場 | 協調パターン |
|---------|----------|-------|-------|-----------|
| ① 要件の言語化 | 目的・価値を言葉にする。スコープ外を決める | overview.md | requirements/ | 壁打ち型 |
| ② 振る舞い定義 | Gherkinシナリオを書く。エッジケースを判断する | .feature ファイル | requirements/features/ | 並走型 |
| ③ ドメインの構造化 | ユビキタス言語を決める。集約の境界を判断する | ubiquitous_language.md, commands_and_events.md, domain_model.md | requirements/domain/ | 壁打ち型 |
| ④ 判断の記録 | 技術・設計のトレードオフを判断し記録する | ADR-XXX.md | requirements/adr/ | 並走型 |
| ⑤ UI設計 | 画面一覧・遷移・コンポーネント・エラー状態を定義する（CLIの場合はコマンド体系・出力フォーマット） | screens.md, screen_flow.md, components.md, error_states.md, features/（またはCLI成果物） | requirements/frontend/ or cli/ | 並走型 |

> **協調パターンの詳細は `04_collaboration_matrix.md` を参照。**
> **各手法の詳細は:** BDDは `05_bdd_and_ddd.md` + `appendix/A_bdd.md`、DDDは `05_bdd_and_ddd.md` + `appendix/B_ddd.md`、
> イベントストーミングは `appendix/C_event_storming.md`、ADRは `appendix/D_adr.md`、UI設計は `06_ui_design.md`（GUI: `06_01_gui.md` / CLI: `06_02_cli.md`）を参照。

### ① 要件の言語化

**目的:** 「何を作るか」「なぜ作るか」「何を作らないか」を言葉にする。

AIとの壁打ちで要件を引き出す。ヒアリング内容や頭の中の構想を、AIの問い返しを受けながら言語化していく。

```
こういうシステムを作りたい。○○ができて、△△は不要。
 抜け漏れや確認すべき点があれば指摘してください
```

壁打ちの結果やメモが溜まったら、overview.md の生成をAIに依頼する。

```markdown
以下は要件についての壁打ちメモです。
 これを元に overview.md を生成してください。

 フォーマット:
 - 概要（このシステムは何か、1〜2文）
 - 機能要件（箇条書き）
 - 非機能要件（箇条書き）
 - スコープ外（何を作らないか）
 - 未決事項（メモの中で結論が出ていないもの）

 メモの中に矛盾や曖昧な箇所があれば指摘してください。

 ---
 [壁打ちメモ・ヒアリングメモを貼る]
```

**成果物: overview.md**
- 概要（このシステムは何か）
- 機能要件（何ができるか）
- 非機能要件（どんな制約があるか）
- スコープ外（何を作らないか）
- 要求トレーサビリティ（要求 → シナリオ → ドメイン概念の対応表。小チーム以上の規模で推奨）

#### 非機能要件の洗い出し

非機能要件は機能要件に比べて見落とされやすい。以下の観点リストをチェックリストとして使うとよい。

| 観点 | 問い | 例 |
|------|------|-----|
| **パフォーマンス** | 応答時間・スループットに制約はあるか？ | API応答200ms以内、同時100ユーザー |
| **可用性** | どの程度の稼働率が必要か？ | 99.9%、計画メンテナンス時間帯あり |
| **セキュリティ** | 認証・認可・データ保護の要件は？ | OAuth2.0、個人情報の暗号化 |
| **スケーラビリティ** | 将来の成長にどう対応するか？ | ユーザー数10倍を想定 |
| **データ** | データ量・保持期間・バックアップは？ | 3年保持、日次バックアップ |
| **互換性** | 対応ブラウザ・OS・外部システム連携は？ | Chrome/Safari最新2バージョン |
| **運用** | 監視・ログ・アラートの要件は？ | エラー率1%超でアラート |
| **法規制** | 法的な制約やコンプライアンス要件は？ | GDPR対応、ログ監査証跡 |

すべての観点を埋める必要はない。**「この観点は今回関係あるか？」を問い、関係ないなら明示的にスコープ外とする**のが重要である。

##### AIとの壁打ちで非機能要件を洗い出す

```
以下のシステムについて、非機能要件の観点
（パフォーマンス・可用性・セキュリティ・スケーラビリティ・データ・互換性・運用・法規制）
から、検討すべき項目を洗い出してください。
各項目について、制約の候補を具体的な数値や条件で提案してください。

---
[overview.md の概要・機能要件を貼る]
```

> 非機能要件の中でトレードオフが発生する判断（例: 応答速度とコストのバランス）は、
> ADRに記録する（`appendix/D_adr.md` 参照）。

> 用語集（グロッサリー）や制約リストも、この段階で overview.md に含めてよい。
> 規模が大きくなったら domain/ubiquitous_language.md に分離する。
> **参照:** 要求トレーサビリティの書き方・導入判断は `appendix/E_rdra.md` を参照。

#### 機能要件をユーザーストーリー形式で書く

機能要件は箇条書きでもよいが、**ユーザーストーリー形式**で書くと「誰が・なぜ」が明示され、
次のステップ（②振る舞い定義）の実例マッピングへの入力がスムーズになる。

```
ユーザーストーリーのテンプレート:
  As a [役割], I want [目的], so that [理由/価値]
```

```
例（レストラン注文システム）:
  As a サーバー, I want テーブルの注文を入力したい, so that 厨房に正確に伝わるから
  As a 厨房スタッフ, I want 注文の優先順位を確認したい, so that 効率的に調理を進められるから
  As a 店長, I want 日別の売上を確認したい, so that 仕入れ量を判断できるから
```

##### なぜこのテンプレートか

テンプレートの各パートが、要件定義で見落としやすい3つの問いを**強制的に**答えさせる。

| パート | 強制される問い | 省略すると起きること |
|--------|-------------|-------------------|
| `As a [役割]` | **誰のための機能か？** | 利用者が曖昧なまま機能だけが膨らむ |
| `I want [目的]` | **何ができるようになるか？** | 目的と手段が混同される |
| `so that [理由]` | **なぜそれが必要か？** | 不要な機能が作られる。優先順位が決められない |

GherkinのGiven/When/Thenと同じ設計思想である。
自然言語の自由記述では書き手によって粒度や観点がバラつくが、
**英語キーワードで構文を固定する**ことで、誰が書いても同じ構造になる。

> ユーザーストーリーは仕様そのものではなく、**実例マッピングの入力**として使う。
> ストーリーを起点に「ルール・具体例・疑問」を掘り下げることで、仕様が育っていく（`appendix/A_bdd.md` 参照）。

##### AIとの壁打ちでストーリーを引き出す

機能要件が漠然としている段階では、AIに問い返してもらうことでストーリーが具体化する。

```
以下のシステムを使う人を洗い出して、
 それぞれの立場で『何をしたいか・なぜしたいか』を
 ユーザーストーリー形式（As a / I want / so that）で列挙してください。
 抜けている役割や、矛盾する要求があれば指摘してください。

 ---
 [システムの概要や壁打ちメモを貼る]
```

> ストーリーの粒度は粗くてよい。「大きすぎるストーリー」は実例マッピングで分解される。
> ストーリーを独立したファイルで管理する必要はない。overview.md の機能要件セクションに含めれば十分である。

### ② 振る舞い定義（BDD）

**目的:** 「こういうときこうなってほしい」をGherkin形式で書く。

①で言語化した要件を、Given/When/Then のシナリオに落とす。
AIと並走しながら書くのが効果的で、AIがシナリオの雛形を生成し、人間がエッジケースの判断をする。

```
overview.md を読んで、主要な機能のGherkinシナリオを書いてください。
 正常系を先に、異常系は後から追加します
```

**成果物: features/ 配下の .feature ファイル**

1周目は正常系だけでよい。粗いシナリオは「次の問いを生む素材」として機能する。
2周目以降で③の結果を反映し、異常系・エッジケースを追加していく。

> **参照:** Gherkinの書き方は `appendix/A_bdd.md`。

---

### ③ ドメインの構造化（DDD）

**目的:** シナリオの裏で「何が起きているか」を掘り下げ、構造化する。

②で書いたシナリオを起点に、イベントストーミングで裏側の構造を発見する。
ここで発見した構造（コマンド・イベント・ポリシー）は、シナリオの抜け漏れを補完する。

```
②で書いたシナリオ → イベントストーミング → コマンド・イベント・ポリシーの発見
  → 「このポリシーに対応するシナリオがない」→ ②に戻ってシナリオ追加
```

**成果物:**
- `domain/ubiquitous_language.md` — プロジェクト内の言葉の定義
- `domain/commands_and_events.md` — コマンド・イベント・ポリシーの一覧
- `domain/domain_model.md` — エンティティ・状態遷移・集約の境界

> ②と③はサイクルで回す。②→③→②→③と往復するうちに、シナリオとドメインモデルの両方が精度を上げていく。
> **参照:** イベントストーミングの進め方は `appendix/C_event_storming.md`、ドメインモデルの作り方は `appendix/B_ddd.md`。

---

### ④ 判断の記録

**目的:** 技術的・設計的な判断を「なぜそう決めたか」とセットで残す。

①〜③を進める中で、技術選定やアーキテクチャの判断が発生する。
これをADR（Architecture Decision Record）として記録する。

**成果物: adr/ 配下の ADR-XXX.md**

ADRがないと、後から別の会話でAIに渡したときに判断の背景が失われ、意図に反した実装が出やすくなる。

> **参照:** ADRの書き方は `appendix/D_adr.md`。

---

### ⑤ UI設計

**目的:** ユーザーインターフェース（GUI / CLI）の「何を作るか」を固める。

バックエンドの②③と並行して進めてよい。
バックエンドでは「ドメインモデル」が設計の核だが、GUIでは「画面とその状態」、CLIでは「コマンドとその入出力」が核になる。

**成果物:**
- `frontend/screens.md` — 画面一覧・URL定義
- `frontend/screen_flow.md` — 画面遷移図
- `frontend/components.md` — UIコンポーネント定義・状態定義
- `frontend/error_states.md` — エラー・空状態の定義
- `frontend/features/` — GUI向けGherkinシナリオ

推奨する作成順序:
1. screens.md で画面の全体像を決める
2. screen_flow.md で遷移条件を定義する
3. components.md で共通部品を洗い出す
4. error_states.md で例外的な状態を網羅する
5. features/ でユーザー操作のシナリオを書く

> **参照:** UI設計の詳細は `06_ui_design.md`（GUI: `06_01_gui.md` / CLI: `06_02_cli.md`）。

---

### サイクルを回すコツ

- **1周目の品質を気にしすぎない** — 粗いシナリオ自体が「次の問いを生む素材」
- 1周目は①→②だけでもよい。③④⑤は2周目以降で段階的に追加する
- ②→③→②のサイクルが最も頻繁に回る（シナリオとドメインモデルの相互補完）
- イベントストーミングで「ヒアリングでは出なかったエッジケース」が見えてくる
- **すべてのステップを毎回やる必要はない** — 規模に応じて必要なものだけ実施する

### 規模による省略の目安

| ステップ | 1人・単機能 | 小チーム | 中規模以上 |
|---------|-----------|---------|----------|
| ① 要件の言語化 | ◎ 必須 | ◎ 必須 | ◎ 必須 |
| ② 振る舞い定義 | ○ 正常系だけ | ◎ 異常系も | ◎ 網羅的に |
| ③ ドメインの構造化 | △ 軽く | ○ 複雑な箇所 | ◎ 全体的に |
| ④ 判断の記録 | △ 重要決定のみ | ○ 推奨 | ◎ 必須 |
| ⑤ UI設計 | △ 簡易 | ○ 推奨 | ◎ 必須 |

> **規模別プラクティスの詳細は `04_collaboration_matrix.md` を参照。**

## フェーズ2: 詳細設計の生成

要求定義が十分に収束したら、AIに詳細設計を生成させる。
フェーズ1の成果物だけを渡し、壁打ちの過程はコンテキストに含めない。

> **ポイント:** フェーズごとに「結論だけ」を渡すことで、AIの精度が上がる。
> この原則の詳細は `appendix/F_context_engineering.md` を参照。

```
documents/requirements/ をすべて読んで、documents/specification/ を生成してください
```

### 詳細設計に含まれるもの

- アーキテクチャ・パッケージ構成
- バックエンド詳細設計（内部インターフェース・データ構造・シーケンス図）
- テスト方針・カバレッジ計画
- UI詳細設計（該当する場合）

**人間の役割:** 生成された詳細設計をレビューし、意図との乖離を修正する。
**協調パターン:** 委譲型（AIが生成、人間がレビュー）

---

## フェーズ3: 実装

詳細設計が固まったら、AIに実装させる。

```
documents/specification/ をすべて読んで、implement/ に実装してください
```

- 実装はfeature単位・1タスク1依頼が最も効果的
- Gherkinシナリオを受け入れ条件として渡すと品質が安定する

```
このシナリオを満たす実装をしてください
```

**人間の役割:** 実装をレビューし、シナリオを満たしているかを確認する。
**協調パターン:** 委譲型（AIが実装、人間がレビュー）

> **参照:** コーディングAIへの委譲時に実行環境（リンター・Hooks・指示ファイル等）を設計する
> 「ハーネスエンジニアリング」については `appendix/I_harness_engineering.md` を参照。

---

## 全体の流れをまとめる

```
フェーズ1                        フェーズ2          フェーズ3
要求定義を育てる                  詳細設計を生成する    実装する

┌──────────────────────────────┐
│ ① 要件概要       → overview.md  │
│       ↓                        │
│ ② 振る舞い定義   → features/    │    architecture.md    ソースコード
│       ↕（サイクル）              │ →  backend/        →  テストコード
│ ③ ドメイン定義   → domain/      │    test_plan.md       設定ファイル
│       ↓                        │    frontend/
│ ④ 判断記録       → adr/        │
│       ↓                        │
│ ⑤ UI設計         → frontend/ or cli/ │
└──────────────────────────────┘
  人間主導・AIと協調               AIが生成            AIが主導
  壁打ち型・並走型                 レビュー型           委譲型
```

## 人間にもAIにも読みやすいドキュメントの条件

各フェーズのドキュメントは、人間とAIの両方が読むことを意識して書く。
特に重要なのは**「なぜ」を明示すること**である。

| 条件 | 悪い例 | 良い例 |
|------|--------|--------|
| 意図が明示されている | 「いい感じに処理する」 | 「〜のためにXをする」 |
| 決定の理由がある | 「案2を採用」 | 「案2を採用。理由: 〜のため」 |
| 制約・対象外が書いてある | （省略） | 「〜は対象外」 |
| 構造が階層化されている | 長文の散文 | 見出し・箇条書き・表 |
| 具体例がある | 「エラー時は適切に処理」 | 「エラー時は〜を出力して終了」 |
| 曖昧語がない | 「高速に」「適切に」 | 「3秒以内に」「ログに記録した上で」 |

> **「なぜ」を書くのは人間の仕事、「何を・どう」はAIと協調できる**

AIは「行間を読む」のが苦手である。人間が当然と思っている前提ほど明示する必要がある。

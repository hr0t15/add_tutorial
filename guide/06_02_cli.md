# 06_02. CLI設計とテスト戦略

`06_ui_design.md` の共通原則を踏まえた、CLI（コマンドラインインターフェース）固有の設計成果物とテスト戦略。

---

## CLI の設計成果物

CLIの設計では、以下の成果物を作る。
GUIにおける「画面」がCLIでは「コマンド」に対応する。

```
documents/requirements/cli/
├── commands.md         # コマンド体系・サブコマンド・フラグ・引数の定義
├── output_format.md    # 出力フォーマット定義（人間向け / JSON / パイプ向け）
├── exit_codes.md       # 終了コード定義
├── error_messages.md   # エラーメッセージ・ヘルプテキストの定義
└── features/           # Gherkinシナリオ（CLI）
    ├── 01_xxx.feature
    └── 02_xxx.feature
```

### 各成果物の役割

| 成果物 | 設計上の役割 | テストへの活用 |
|-------|------------|-------------|
| コマンド体系 | コマンド・サブコマンド・引数・フラグの全体像 | テスト対象のコマンドが確定する |
| 出力フォーマット定義 | 人間向け・JSON・パイプ向けの出力仕様 | 出力の正確性を検証する根拠になる |
| 終了コード定義 | 成功・失敗の終了コード体系 | 異常系テストの期待値になる |
| エラーメッセージ定義 | エラー時の表示・ヘルプテキスト | 異常系シナリオに直結する |
| Gherkinシナリオ | コマンド実行の期待される振る舞い | そのまま統合テストケースになる |

### なぜこの4つか

GUIでは「画面とその状態」が設計の核だが、
CLIでは**「コマンドとその入出力」**が設計の核になる。

```
GUI: 画面の状態遷移がユーザーの期待通りか？
CLI: コマンドの入出力がユーザーの期待通りか？
```

この4つの成果物は、CLIにおける「入出力の契約」を異なる角度で定義している。

```
commands.md        → 入力の契約（何を受け付けるか）
output_format.md   → 正常系出力の契約（何を返すか）
exit_codes.md      → 成功/失敗の契約（どう終了するか）
error_messages.md  → 異常系出力の契約（失敗時に何を伝えるか）
```

## CLI 向け Gherkin シナリオ

GUIと同じGherkin形式で書く（書き方の詳細は `appendix/A_bdd.md` を参照）。

### 正常系

```gherkin
Scenario: 予約を作成する
  Given スタッフAの10:00枠が空いている
  When  `reserve create --staff A --time 10:00 --customer "山田太郎" --menu "カット"` を実行する
  Then  "予約を作成しました: スタッフA 10:00 山田太郎 カット" と出力される
  And   終了コード 0 で終了する

Scenario: 予約一覧をJSON形式で出力する
  Given スタッフAの10:00に「山田太郎 - カット」が予約されている
  When  `reserve list --staff A --format json` を実行する
  Then  JSON形式で予約一覧が出力される
  And   出力に "山田太郎" が含まれる
```

### 異常系

```gherkin
Scenario: 二重予約をしようとする
  Given スタッフAの10:00枠が予約済みである
  When  `reserve create --staff A --time 10:00 --customer "佐藤花子" --menu "カラー"` を実行する
  Then  stderr に "エラー: スタッフAの10:00は既に予約されています" と出力される
  And   終了コード 1 で終了する

Scenario: 必須引数を省略する
  When  `reserve create --staff A` を実行する
  Then  stderr に "エラー: --time は必須です" と出力される
  And   使い方（usage）が表示される
  And   終了コード 2 で終了する
```

### GUIとの違い

CLIのシナリオは「コマンドの入出力」を記述する。

```
GUI: When 「注文確定」ボタンをクリックする → Then 注文完了画面に遷移する
CLI: When `order confirm ORD-001` を実行する → Then "注文 ORD-001 を確定しました" と出力される
```

## コマンド体系の設計

CLIの設計で最も重要なのはコマンド体系である。
ユビキタス言語がコマンド名に反映されることで、ドメインの概念がそのままインターフェースになる。

### 構造の例

```
reserve                          # ルートコマンド
├── create                       # 予約作成
│   ├── --staff <name>           # 必須: スタッフ名
│   ├── --time <HH:MM>          # 必須: 時間帯
│   ├── --customer <name>       # 必須: 顧客名
│   └── --menu <name>           # 必須: メニュー名
├── cancel <reservation-id>      # 予約キャンセル
├── list                         # 予約一覧
│   ├── --staff <name>           # フィルタ: スタッフ名
│   ├── --date <YYYY-MM-DD>     # フィルタ: 日付
│   └── --format <text|json>    # 出力形式（デフォルト: text）
└── waitlist                     # キャンセル待ち
    ├── join                     # キャンセル待ち登録
    └── leave                    # キャンセル待ち解除
```

### コマンド設計の原則

| 原則 | 説明 | 例 |
|------|------|-----|
| ドメイン動詞をそのまま使う | ユビキタス言語の動詞をコマンド名にする | `create`, `cancel`, `join` |
| `--format` で出力形式を切り替える | 人間向け（text）と機械向け（json）を両方サポート | `--format json` |
| 必須引数は明示する | 省略時にエラーメッセージとusageを表示 | `エラー: --staff は必須です` |
| 終了コードを設計する | 成功/失敗/引数エラーを区別する | 0: 成功, 1: 業務エラー, 2: 引数エラー |

## 出力フォーマット設計

CLIの出力は、**誰が読むか**によって形式を変える。

### 人間向け（デフォルト）

```
$ reserve list --staff A
スタッフA の予約一覧 (2025-01-15)
────────────────────────────────
 09:00  （空き）
 10:00  山田太郎 - カット
 11:00  佐藤花子 - カラー
 12:00  （空き）
 ...
```

### 機械向け（JSON）

```json
$ reserve list --staff A --format json
{
  "staff": "A",
  "date": "2025-01-15",
  "slots": [
    {"time": "09:00", "status": "available"},
    {"time": "10:00", "status": "reserved", "customer": "山田太郎", "menu": "カット"},
    {"time": "11:00", "status": "reserved", "customer": "佐藤花子", "menu": "カラー"},
    {"time": "12:00", "status": "available"}
  ]
}
```

### パイプ向け（TSV / 1行1レコード）

```
$ reserve list --staff A --format tsv | grep "reserved"
10:00	reserved	山田太郎	カット
11:00	reserved	佐藤花子	カラー
```

> **ポイント:** 正常出力はstdout、エラーはstderrに出力する。
> これにより、パイプやリダイレクトが正しく動作する。

## 終了コード設計

終了コードはCLIの「見えない出力」であり、スクリプトやCI/CDでの利用に不可欠である。

| 終了コード | 意味 | 例 |
|-----------|------|-----|
| 0 | 成功 | 予約が正常に作成された |
| 1 | 業務エラー | 二重予約、対象が見つからない等 |
| 2 | 引数エラー | 必須引数の不足、不正な形式 |
| 3 以降 | アプリケーション固有のエラー | 必要に応じて定義 |

> 終了コードの体系は `exit_codes.md` に定義し、テストで検証する。

## テスト戦略

### テスト粒度の選択肢

CLIのテストには以下の粒度がある。
GUIのE2Eテストに相当するのが「統合テスト（コマンド実行テスト）」である。

| テスト粒度 | 1人・単機能 | 小チーム・複数機能 | 中規模・複数チーム |
|-----------|-----------|----------------|----------------|
| **統合テスト（正常系）** | ◎ 必須 | ◎ 必須 | ◎ 必須 |
| **統合テスト（異常系）** | △ 代表的なものだけ | ○ 推奨 | ◎ 必須 |
| **コマンドヘルパー** | △ なくても可 | ○ 推奨 | ◎ 必須 |
| **外部サービスモック** | △ 任意 | ○ 推奨 | ◎ 必須 |
| **出力フォーマットテスト** | △ JSON出力があれば | ○ 推奨 | ◎ 必須 |
| **終了コードテスト** | ○ 推奨 | ◎ 必須 | ◎ 必須 |
| **ドメインロジックユニットテスト** | × 過剰 | △ 複雑なものだけ | ○ 推奨 |

### 統合テストの書き方

CLIの統合テストは「コマンドを実行して出力と終了コードを検証する」形式になる。

```
テストの構造:
  1. テストデータをセットアップ（Given）
  2. コマンドを実行（When）
  3. stdout / stderr / 終了コードを検証（Then）
```

### コマンドヘルパー（GUIのPage Object に相当）

GUIのPage Objectパターンと同様に、CLIでもコマンド実行をヘルパーに抽象化する。

```
目的:
  - コマンドの引数構築を一元管理する
  - 出力のパースを共通化する
  - テストの可読性を上げる

例:
  直接実行:  exec("reserve create --staff A --time 10:00 --customer 山田太郎 --menu カット")
  ヘルパー:  ReserveCLI.create(staff: "A", time: "10:00", customer: "山田太郎", menu: "カット")
```

テストが増えてきたら導入する。最初から作る必要はない。

### 設計成果物とテストの対応

```
commands.md
  → 統合テスト（コマンドの入出力が正しいか）
  → コマンドヘルパー（引数構築の根拠）

output_format.md
  → 出力フォーマットテスト（JSON/TSV/テキストの形式が正しいか）

exit_codes.md
  → 終了コードテスト（成功/失敗の区別が正しいか）

error_messages.md
  → 統合テスト異常系（エラー時の出力が正しいか）

features/
  → 統合テスト（シナリオをそのままテストケースにする）
```

## GUI との対応関係

GUIの知識がある読者向けに、GUIとCLIの設計成果物の対応を示す。

| GUI | CLI | 共通する関心事 |
|-----|-----|-------------|
| `screens.md` | `commands.md` | ユーザーが操作できる対象の全体像 |
| `screen_flow.md` | （コマンドは独立実行が基本） | 操作の流れ・依存関係 |
| `components.md` | `output_format.md` | 出力の構造・再利用可能な部品 |
| `error_states.md` | `error_messages.md` + `exit_codes.md` | 異常時のユーザーへのフィードバック |
| Page Object | コマンドヘルパー | テストの抽象化レイヤー |
| E2Eテスト（Playwright等） | 統合テスト（コマンド実行） | エンドツーエンドの振る舞い検証 |

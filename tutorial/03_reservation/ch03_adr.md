# 第3章: ADRを複数書き、相互参照する

## この章のゴール

> **主要な設計判断がADRとして記録・相互参照されている状態**

コンテキストをまたぐ設計判断をADRとして記録する。
チュートリアル1・2ではADRが2つだったが、
今回は **複数のADRが相互に影響し合う** 状況を体験する。

---

## なぜADRの相互参照が必要か

複数のADRが独立していた場合:
```
ADR-001: 楽観的ロック採用
ADR-002: コンテキスト間は同期通信
ADR-003: モノリシックアーキテクチャ
→ 個別に読むと理解できるが、判断間の関係が見えない
```

相互参照がある場合:
```
ADR-001: 楽観的ロック採用
  → ADR-003のモノリシック前提で選択。マイクロサービス化する場合は再検討
ADR-002: コンテキスト間は同期通信
  → ADR-003のモノリシック前提。分離時にはADR-002を見直す
ADR-003: モノリシックアーキテクチャ
  → ADR-001, ADR-002 の前提。変更時は両方を再検討
→ 判断の依存関係が明確で、将来の変更時に何を見直すべきか分かる
```

> `guide/appendix/D_adr.md` の「ADRを廃止・置き換えるとき」も参照。

---

## このチュートリアルで書くADR

| ADR | 判断内容 | 影響範囲 |
|-----|---------|---------|
| ADR-001 | 競合制御の実装方針（楽観的ロック） | 予約コンテキストの実装 |
| ADR-002 | コンテキスト間の通信方式（同期） | 全コンテキスト間 |
| ADR-003 | アーキテクチャ（モノリシック） | プロジェクト全体 |
| ADR-004 | データベース選定 | 全コンテキスト |

---

## ADR-001: 競合制御の実装方針

### AIへの依頼

```
以下の判断についてADRを書いてください。

判断: 二重予約防止に楽観的ロック（Optimistic Locking）を採用する。
背景: 同一スタッフの同一時間帯に2つの予約リクエストが同時に来た場合、
      1つだけが成功し、もう1つは失敗する必要がある。
制約: ADR-003でモノリシックアーキテクチャを前提とする。

選択肢:
1. 楽観的ロック（バージョン番号によるチェック）
2. 悲観的ロック（DBの行ロック）
3. ユニーク制約のみ（DB側で保証）
```

### 成果物のイメージ

```markdown
# ADR-001: 二重予約防止に楽観的ロックを採用する

## ステータス
採用済み

## 文脈
予約システムでは、同一スタッフの同一時間帯に2つの予約リクエストが
同時に到着する可能性がある。
このとき、1つだけが成功し、もう1つは「すでに予約済み」として
失敗する必要がある（二重予約防止）。

ADR-003でモノリシックアーキテクチャを前提としており、
単一データベースに対する排他制御として設計する。

## 決定
楽観的ロック（Optimistic Locking）を採用する。
TimeSlot 集約にバージョン番号を持たせ、
更新時にバージョンが一致しない場合は競合エラーとする。

## 理由
- 予約の競合は頻繁には発生しない（楽観的な想定が妥当）
- 悲観的ロックはDBへの負荷が高く、デッドロックのリスクがある
- ユニーク制約のみでは、キャンセル待ち昇格などのビジネスロジックを
  DB層で表現しきれない
- ドメインモデル（TimeSlot集約）の中で整合性を保証する設計と相性が良い

### 選ばなかった選択肢
- **悲観的ロック:** 確実だが、ロック待ちによるパフォーマンス低下と
  デッドロックリスクが懸念。チュートリアルの規模では過剰
- **ユニーク制約のみ:** DB層での制約は有効だが、キャンセル待ち昇格の
  ロジックを含む複合的な操作をDB制約だけで表現するのは困難

## 影響
- TimeSlot エンティティに version フィールドを追加する必要がある
- 競合時のリトライまたはエラーメッセージの設計が必要
- ADR-003（モノリシック）が前提。マイクロサービス化する場合、
  分散ロックの検討が必要になる

## 参照
- ADR-003（モノリシックアーキテクチャ）
- documents/requirements/reservation/domain/domain_model.md
```

## ADR-002: コンテキスト間の通信方式

### 成果物のイメージ

```markdown
# ADR-002: コンテキスト間通信に同期呼び出しを採用する

## ステータス
採用済み

## 文脈
予約コンテキストは、予約作成時にスタッフの存在確認と
顧客の存在確認を行う必要がある。
この通信が同期（即座に応答を得る）か非同期（イベント経由）かを
決定する必要がある。

## 決定
コンテキスト間の通信はすべて同期呼び出し（メソッド呼び出し）とする。

## 理由
- ADR-003でモノリシックアーキテクチャを前提としているため、
  メソッド呼び出しが最もシンプル
- 予約作成時のスタッフ確認・顧客確認は「即座に結果が必要」な処理であり、
  非同期にする動機がない
- メッセージキュー等のインフラが不要

### 選ばなかった選択肢
- **非同期（イベント駆動）:** マイクロサービス化する場合は有力だが、
  モノリシック構成では複雑さが増すだけ。結果整合性の設計も必要になる
- **REST API（HTTP呼び出し）:** モノリシック内では不要なオーバーヘッド

## 影響
- コンテキスト間の依存関係はインターフェース（interface）で定義し、
  具体的な実装はDI（依存性注入）で差し替え可能にする
- 将来マイクロサービス化する場合、インターフェースの実装を
  HTTP/gRPC クライアントに差し替える
- ADR-003を変更する場合、この ADR も再検討が必要

## 参照
- ADR-003（モノリシックアーキテクチャ）
- ADR-001（楽観的ロック — 同期通信前提で選択）
```

## ADR-003: アーキテクチャ

### 成果物のイメージ

```markdown
# ADR-003: モノリシックアーキテクチャを採用する

## ステータス
採用済み

## 文脈
予約システムは3つのコンテキスト（予約・スタッフ・顧客）で構成される。
これらを1つのアプリケーションとしてデプロイするか、
コンテキストごとにサービスを分けるかを決定する必要がある。

## 決定
モノリシックアーキテクチャを採用する。
3つのコンテキストは同一アプリケーション内のモジュールとして実装する。

## 理由
- チュートリアルの規模では複数サービスの運用は過剰
- コンテキスト間の通信がシンプル（メソッド呼び出し）になる
- データベースを1つで済ませられる
- 将来的に分割する場合も、コンテキストの境界が明確であれば
  モジュールをサービスに切り出すことは可能

### 選ばなかった選択肢
- **マイクロサービス:** コンテキストごとにサービスを分けると理想的だが、
  インフラ（サービス間通信、個別DB、デプロイパイプライン）の
  構築コストがチュートリアルの範囲を超える

## 影響
- ADR-001（楽観的ロック）とADR-002（同期通信）はこの決定が前提
- コンテキスト間はモジュール境界（パッケージ/ディレクトリ）で分離する
- 将来の分割に備え、コンテキスト間の依存はインターフェースで定義する

## 参照
- ADR-001（楽観的ロック — モノリシック前提で選択）
- ADR-002（同期通信 — モノリシック前提で選択）
```

## ADR-004: データベース選定

### 成果物のイメージ

```markdown
# ADR-004: データベースにSQLite を採用する

## ステータス
採用済み

## 文脈
予約システムのデータを永続化する必要がある。
ADR-003でモノリシックアーキテクチャを前提としており、
チュートリアルの環境構築を最小限にしたい。

## 決定
SQLite を採用する。

## 理由
- サーバーのインストールが不要（ファイルベース）
- ADR-001の楽観的ロックに必要なトランザクション・バージョンチェックをサポート
- チュートリアルの規模（スタッフ3人、時間帯10枠）では十分な性能

### 選ばなかった選択肢
- **PostgreSQL:** 本格的だが、サーバーの起動・管理が必要
- **localStorage:** チュートリアル1で使用したが、複数画面間のデータ共有と
  トランザクション（競合制御）ができない
- **インメモリ:** 永続化されないため、リロードでデータが消える

## 影響
- ORMまたはクエリビルダーの選定が必要（実装時に決定）
- 同時接続数の制限があるが、チュートリアルの規模では問題にならない

## 参照
- ADR-001（楽観的ロック — SQLiteのトランザクションで実現）
- ADR-003（モノリシック — 単一DBで全コンテキストのデータを管理）
```

## ADR間の相互参照マップ

4つのADRの依存関係を可視化する。

```
ADR-003: モノリシック ← 最も基盤的な判断
  │
  ├─→ ADR-001: 楽観的ロック（モノリシック前提）
  ├─→ ADR-002: 同期通信（モノリシック前提）
  └─→ ADR-004: SQLite（モノリシック前提）

変更の影響:
  ADR-003 を変更 → ADR-001, ADR-002, ADR-004 すべて再検討
  ADR-001 を変更 → ADR-004 のトランザクション要件を再検討
  ADR-004 を変更 → ADR-001 の楽観的ロックの実現方式を再検討
```

> **ポイント:** ADR-003（モノリシック）が最も基盤的な判断であり、
> 他の3つはその上に乗っている。
> 将来この判断を覆す場合、すべてのADRを見直す必要がある。
> この「変更時に何を見直すべきか」が分かることがADR相互参照の価値。

## 振り返り

- **ADR相互参照の価値:** 判断の依存関係が可視化され、
  将来の変更時に何を見直すべきかが明確になった
- **基盤的な判断の識別:** ADR-003（モノリシック）が他のすべてに影響することが分かった。
  最初にこの判断を固めるべきだった
- **AIへの渡し方:** ADRをまとめて渡すことで、AIが判断間の整合性を保った設計を出しやすくなる

## チュートリアル3で体験したこと

```
第1章: 要件を言語化し、コンテキストを識別する
  → 3つのコンテキスト（予約・スタッフ・顧客）を識別した

第2章: コンテキストごとに要求定義を作る
  → 集約の境界・競合制御をドメインモデルに表現した

第3章: ADRを複数書き、相互参照する
  → 4つのADRの依存関係を可視化した

第4章: コンテキスト単位で詳細設計を生成・実装する
  → コンテキスト単位でコーディングAIに委譲する体験をした

第5章: コンポーネントテストを追加する
  → ドメインルールをコンポーネントテストで保護した
```

## 3つのチュートリアルを通じて学んだこと

### プラクティスの全体像

```
チュートリアル1（Kanban・★☆☆）:
  ✅ 実例マッピングでルール・例・疑問を発見する
  ✅ 壁打ちで要件を言語化する
  ✅ 要求定義 → 詳細設計 → 実装 の流れ
  ✅ Gherkinシナリオ
  ✅ ユビキタス言語
  ✅ ADR
  ✅ Playwright E2E テスト

チュートリアル2（レストラン・★★☆）:
  ✅ イベントストーミング
  ✅ コマンド・イベント・ポリシー
  ✅ 異常系シナリオの体系的洗い出し
  ✅ DDD成果物（commands_and_events.md, domain_model.md）
  ✅ Page Objectパターン
  ✅ APIモック

チュートリアル3（予約・★★★）:
  ✅ 集約・競合制御
  ✅ 境界付けられたコンテキスト
  ✅ コンテキスト単位での委譲
  ✅ コンポーネントテスト
  ✅ ADRの相互参照
```

### 規模と必要なプラクティスの関係

`guide/04_collaboration_matrix.md` のマトリックスを、チュートリアルの体験と紐づける。

| 規模 | 対応するチュートリアル | 核心的なプラクティス |
|------|---------------------|-------------------|
| 1人・単機能 | チュートリアル1 | 実例マッピング → 要求定義 → 詳細設計 → 実装 |
| 小チーム・複数機能 | チュートリアル2 | イベントストーミング・ポリシー・発見手法のサイクル |
| 中規模・複数チーム | チュートリアル3 | 集約・コンテキスト分割・コンテキスト単位委譲 |

> 規模が上がるにつれて、暗黙知の言語化がより重要になる。
> チュートリアル3で最も多くのドキュメントを作ったのは偶然ではない。

## AIとの協調開発で最も重要なこと

3つのチュートリアルを通じて、一貫して重要だったことを整理する。

### 1. フェーズを分けて情報を渡す

```
チュートリアル1: 要求定義 → 詳細設計 → 実装
チュートリアル2: 同じ流れ + DDD成果物
チュートリアル3: 同じ流れ + コンテキスト単位で分割
```

規模が大きくなっても原則は同じ。分割の粒度が細かくなるだけ。

### 2. 「なぜ」を記録する

```
チュートリアル1: ADR 2つ（技術選定の理由）
チュートリアル2: ADR 2つ + ポリシーの「なぜ自動化するか」
チュートリアル3: ADR 4つ（相互参照）+ 集約の「なぜこの境界か」
```

「なぜ」の記録がない判断は、時間が経つと意味が失われる。

### 3. シナリオが全体をつなぐ

```
壁打ちで粗く書いたシナリオ
  → イベントストーミングで異常系を追加
  → featureファイルに正式化
  → 受け入れ条件としてAIに渡す
  → E2Eテスト・コンポーネントテストに変換
```

Gherkinシナリオは仕様であり、テストであり、AIへの指示である。

### 4. 人間が判断し、AIが実行する

```
上流（ビジネス要求・ドメインモデリング）→ 人間が主導
中流（詳細設計・仕様書生成）→ AIが生成、人間がレビュー
下流（実装・テスト生成）→ AIが主導、人間がレビュー
```

この関係は規模が変わっても不変。

### 5. AIに渡す情報を設計する（コンテキストエンジニアリング）

上記4つの原則を支える基盤が、**AIに渡す情報をどう構造化するか**という問題である。

```
チュートリアル1: フェーズ単位での分割（要求定義 → 詳細設計 → 実装）
チュートリアル2: DDD成果物による構造化（コマンド・イベント・ポリシーの明示化）
チュートリアル3: コンテキスト単位での分割（境界を意識した委譲）
```

規模が大きくなるにつれて、AIに渡す情報をどう設計するかがより重要になる。
成果物をファイルに書き出すことでコンテキストを解放し、
フェーズごとに「結論だけ」を渡すことでAIの精度を上げてきた。

> 詳細は `guide/appendix/F_context_engineering.md` を参照。

## 実務への応用ガイド

### 小さく始める

いきなりチュートリアル3のすべてを導入しようとしない。

```
最初の1週間:
  チュートリアル1の流れだけ実践する
  → 実例マッピング → 要求定義 → 詳細設計 → 実装

慣れてきたら:
  イベントストーミングを導入する（チュートリアル2）
  → 複雑な箇所だけでよい
  → 実例マッピングとのサイクルを回す

規模が大きくなったら:
  コンテキスト分割を検討する（チュートリアル3）
  → 「同じ言葉が違う意味で使われ始めた」のがサイン
```

### 発見手法の使い分け

3つのチュートリアルを通じて、発見手法の使い分けを体験した。

| 規模 | 実例マッピング | イベントストーミング | 体験したチュートリアル |
|------|-------------|-------------------|-------------------|
| 1人・単機能 | ◎ これだけで十分 | △ 省略可 | チュートリアル1 |
| 小チーム・複数機能 | ◎ 必須 | ○ 複雑な箇所のみ | チュートリアル2 |
| 中規模・複数チーム | ◎ 必須 | ◎ 必須（サイクルで） | チュートリアル3 |

> 詳細は `guide/05_bdd_and_ddd.md` の「実例マッピングとイベントストーミングの関係」を参照。

### ガイドが認める限界

`guide/01_philosophy.md` の「このガイド自身の限界と課題」で述べられているように、
このガイドにはいくつかの未検証・未解決の課題がある。

3つのチュートリアルを終えた今だからこそ見えてくる問いがある。

```
チュートリアル3を通じて自問すべきこと:
  - 3コンテキスト分割は本当に必要だったか？ 1つで十分だった可能性はないか？
  - ADRを4つ書いたが、後から読み返して「書いてよかった」と思えるか？
  - イベントストーミングを1人+AIでやった結果は、チームでやった場合とどう違うか？
  - overview.md + features/ だけで済んだ箇所はなかったか？
  - コンポーネントテストのコストに見合うだけの安心感が得られたか？
```

> 詳細は `guide/01_philosophy.md` の「このガイド自身の限界と課題」、
> および `guide/appendix/H_spec_driven_development.md` の「このガイド自身の弱み」を参照。

### 最後に

`guide/01_philosophy.md` の言葉を再掲する。

> **ドメイン知識 × 言語化力 × AIとの壁打ち**
>
> この3つが揃うと、開発の質とスピードが大きく上がる。

3つのチュートリアルを通じて、この3つの力を鍛えてきた。
あとは実務で繰り返し実践することで、自分のものになっていく。

3つのチュートリアルを終えた今、自分の実務で「どの成果物が本当に必要だったか」
「省略しても品質が落ちなかったものはどれか」を振り返ることが、
ガイドを自分のものにする第一歩である。

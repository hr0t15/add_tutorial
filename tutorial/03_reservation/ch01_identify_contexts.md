# 第1章: 要件を言語化し、コンテキストを識別する

## この章のゴール

> **3つのコンテキスト（予約・スタッフ・顧客）とその境界が明確になっている状態**

予約システムの要件をAIとの対話で固め、
イベントストーミングで全体像を発見し、
**境界付けられたコンテキスト** を識別する。

チュートリアル2までは1つのコンテキストで完結していた。
このチュートリアルでは初めて **システムを複数のコンテキストに分割する** 体験をする。

---

## チュートリアル2との違い

| 観点 | チュートリアル2 | チュートリアル3 |
|------|--------------|--------------|
| コンテキスト数 | 1つ | **3つ** |
| 集約 | 意識しなくても動いた | **明確な境界が必要** |
| ドメインルール | シンプル（状態遷移のみ） | **競合制御（二重予約防止）** |
| 同じ概念の使い分け | なし | **コンテキストによって意味が変わる** |

> この章ではチュートリアル2で学んだイベントストーミングを再び使う。
> 手法は同じだが、スケールが大きくなったときに何が変わるかを体感する。

---

## ステップ1: 壁打ちで要件を固める

### AIへの最初の問いかけ

```
美容院の予約システムを作りたいと思っています。
スタッフ・時間帯ごとに予約を管理し、二重予約を防止するシステムです。
要件を一緒に整理してもらえますか？
```

### AIから出てくる問い

- スタッフは何人いるか？ スタッフの追加・削除はできるか？
- 時間帯の単位は？（30分刻み？ 1時間刻み？）
- メニュー（カット・カラー等）によって所要時間は変わるか？
- 予約の変更はできるか？
- キャンセル待ちはあるか？
- 顧客情報はどこまで管理するか？
- 予約の確認通知は送るか？

### 判断する

| 問い | 判断 | 理由 |
|------|------|------|
| スタッフ数 | 固定3人（A, B, C） | シンプルにする |
| 時間帯 | 1時間刻み（9:00〜18:00） | 時間帯の分割は対象外 |
| メニューと所要時間 | 全メニュー1時間固定 | 可変時間は複雑すぎる |
| 予約変更 | 可能（キャンセル+再予約として扱う） | 変更を独立機能にすると複雑 |
| キャンセル待ち | あり（FIFO順で昇格） | ポリシーの学習に最適 |
| 顧客情報 | 名前・電話番号 | 最小限 |
| 確認通知 | なし（スコープ外） | 通知インフラはチュートリアル範囲外 |

### スコープ外

```
- スタッフの追加・削除
- メニュー別の所要時間
- 時間帯のカスタム（30分刻み等）
- 予約の確認通知（メール・SMS）
- 顧客のログイン・認証
- 決済
- スタッフのシフト管理（全スタッフ全時間帯で稼働と仮定）
```

## ステップ2: 実例マッピングを経て正常系シナリオを書き出す

チュートリアル1で学んだ実例マッピングを使い、各ストーリー（予約作成、キャンセル、キャンセル待ち等）の
ルール・例・疑問を整理してからシナリオを書く。

`guide/05_bdd_and_ddd.md` が推奨する通り、中規模では実例マッピングとイベントストーミングの
両方をサイクルで回すことが重要である。この章ではステップ2で実例マッピング、ステップ3でイベントストーミングを行い、
相互に補完し合う流れを体験する。

> 壁打ちメモが多い場合は、`guide/appendix/A_bdd.md` の
> 「雑多なメモからの実例マッピング生成」プロンプトで2段階に整理すると効果的である。

###

### 予約の作成

```
シナリオ: 予約を作成する
  Given スタッフAの10:00枠が空いている
  When  顧客「山田太郎」がスタッフAの10:00に「カット」を予約する
  Then  スタッフAの10:00に「山田太郎 - カット」が表示される
  And   スタッフAの10:00枠が予約済みになる
```

### 予約の一覧表示

```
シナリオ: スタッフ別の予約状況を確認する
  Given スタッフAの10:00に「山田太郎 - カット」が予約されている
  And   スタッフBの10:00枠は空いている
  When  10:00の予約状況を表示する
  Then  スタッフAの10:00は「山田太郎 - カット」と表示される
  And   スタッフBの10:00は「空き」と表示される
```

### 予約のキャンセル

```
シナリオ: 予約をキャンセルする
  Given スタッフAの10:00に「山田太郎 - カット」が予約されている
  When  「山田太郎」の予約をキャンセルする
  Then  スタッフAの10:00枠が空きに戻る
```

### キャンセル待ちと昇格

```
シナリオ: キャンセル待ちに登録する
  Given スタッフAの10:00枠が予約済みである
  When  顧客「佐藤花子」がスタッフAの10:00のキャンセル待ちに登録する
  Then  スタッフAの10:00のキャンセル待ちリストに「佐藤花子」が追加される

シナリオ: キャンセル待ちが自動的に昇格する
  Given スタッフAの10:00に「山田太郎」の予約があり、
        キャンセル待ちに「佐藤花子」「鈴木一郎」が登録されている
  When  「山田太郎」の予約がキャンセルされる
  Then  スタッフAの10:00に「佐藤花子」の予約が自動的に作成される
  And   キャンセル待ちリストから「佐藤花子」が消える
  And   「鈴木一郎」はキャンセル待ちに残る
```

### 顧客情報

```
シナリオ: 顧客を登録する
  When  名前「山田太郎」、電話番号「090-1234-5678」で顧客を登録する
  Then  顧客リストに「山田太郎」が追加される
```

## ステップ3: イベントストーミングで全体像を発見する

チュートリアル2で学んだイベントストーミングを再び実施する。

### AIへの依頼

```
予約システムで起きうるドメインイベントをすべて洗い出してください。
以下の正常系シナリオを参考にしてください。

[上記シナリオを貼る]

イベントは過去形、コマンドは命令形で命名してください。
ポリシー（自動実行ルール）も識別してください。
```

### 発見されるコマンド・イベント・ポリシー

```
アクター      コマンド                →  ドメインイベント
─────────────────────────────────────────────────────────
Staff     →  CreateReservation      →  ReservationCreated
Staff     →  CancelReservation      →  ReservationCancelled
Customer  →  JoinWaitingList        →  WaitingListJoined
Customer  →  LeaveWaitingList       →  WaitingListLeft
System    →  PromoteFromWaitingList →  WaitingListPromoted
Staff     →  RegisterCustomer       →  CustomerRegistered
Staff     →  UpdateCustomer         →  CustomerUpdated

ポリシー:
  ReservationCancelled
    → [ポリシー: キャンセル待ち昇格]
    → キャンセル待ちリストの先頭を自動予約する
```

## ステップ4: 境界付けられたコンテキストを識別する

ここがチュートリアル3の核心。

### 「同じ言葉が違う意味で使われている」に気づく

イベントストーミングの結果を眺めると、同じ「予約」という概念が
異なる文脈で異なる意味を持っていることに気づく。

```
予約の文脈:
  「予約」= スタッフ × 時間帯に紐づく予約枠の確保
  → 気にすること: 二重予約の防止、キャンセル待ち、時間帯の空き

スタッフの文脈:
  「予約」= スタッフのスケジュールの1コマ
  → 気にすること: 誰がいつ空いているか、稼働可能な時間帯

顧客の文脈:
  「予約」= 顧客の予約履歴の1件
  → 気にすること: 連絡先、過去の予約履歴
```

### AIへの依頼

```
イベントストーミングの結果をもとに、
境界付けられたコンテキストを識別してください。

以下の観点で分割してください。
- 同じ言葉が異なる意味で使われている箇所
- 変更理由が異なるグループ
- 独立して開発・デプロイできる単位
```

### 識別される3つのコンテキスト

```
予約コンテキスト（Reservation Context）
  ─────────────────────────────────────
  関心事: 予約枠の管理・二重予約防止・キャンセル待ち
  集約:   Reservation, TimeSlot, WaitingList
  コマンド: CreateReservation, CancelReservation,
           JoinWaitingList, PromoteFromWaitingList

スタッフコンテキスト（Staff Context）
  ─────────────────────────────────────
  関心事: スタッフの情報・稼働スケジュール
  エンティティ: Staff, Schedule, Availability
  コマンド: （このチュートリアルではスタッフは固定のためコマンドなし）

顧客コンテキスト（Customer Context）
  ─────────────────────────────────────
  関心事: 顧客の情報・連絡先
  エンティティ: Customer, ContactInfo
  コマンド: RegisterCustomer, UpdateCustomer
```

### コンテキスト間の関係

```
予約コンテキスト
  │
  ├── staffId で参照 ──→ スタッフコンテキスト
  │                      （スタッフの存在確認・空き確認）
  │
  └── customerId で参照 ──→ 顧客コンテキスト
                            （顧客の存在確認・情報取得）
```

> **ポイント:** コンテキスト間はIDで参照する。
> 予約コンテキストはスタッフの詳細情報を知らない。知る必要もない。
> 「staffId で空きを問い合わせる」だけでよい。

## ステップ5: 気づき・疑問を整理する

```
気づき・疑問:
- キャンセル待ちの上限は？ → このチュートリアルでは上限なしとする
- 同じ顧客が同じ時間帯に複数予約できる？ → 不可とする
- キャンセル待ちからの自動予約が失敗したら？
  → 次の待機者に回す
- スタッフコンテキストの「稼働可能」はどう管理する？
  → 全時間帯で稼働と仮定（スコープ外の判断を反映）
- コンテキスト間の通信は同期？非同期？
  → 第3章のADRで判断を記録する
```

## 振り返り

- **コンテキスト分割の判断基準:** 「同じ言葉が異なる意味で使われている」ところに境界がある
- **IDでの参照:** コンテキスト間は詳細を共有せず、IDで参照する
- **予約コンテキストの特殊性:** 「二重予約防止」というルールが、集約の設計を必要とする。
  これは次章で具体化する

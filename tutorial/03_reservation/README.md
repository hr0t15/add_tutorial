# チュートリアル3: 予約システム

## 概要

このチュートリアルでは、美容院・ホテル等の予約システムを題材に
**ドメインルール・制約・コンテキスト分割**を学ぶ。

チュートリアル1・2で学んだすべてのプラクティスを統合し、
「複雑なビジネスルールをどう設計に落とすか」
「規模が大きくなったときに何が変わるか」を体感することが目的。

---

## 学習目標

このチュートリアルを終えると以下ができるようになる。

- [ ] 競合制御（二重予約防止）をドメインモデルで表現できる
- [ ] 集約の境界を意識した設計ができる
- [ ] 境界付けられたコンテキストを識別できる
- [ ] コンテキスト単位でコーディングAIに委譲できる
- [ ] コンポーネントテストを書ける
- [ ] ADRを複数書いて相互参照できる

---

## 題材: 予約システム

### 作るもの

スタッフ・時間帯ごとに予約を管理し、二重予約を防止するシステム。

```
予約可能な枠の表示 → 予約の作成 → 確定 → キャンセル → キャンセル待ち昇格
```

### なぜ予約システムか

- 「同じ時間に二重予約できない」という制約がドメインモデルの重要性を体感させる
- キャンセル待ち・昇格などのポリシーが複雑で実務に近い
- 予約・スタッフ・顧客でコンテキストが自然に分かれる

## 3つのコンテキスト

```
予約コンテキスト          スタッフコンテキスト      顧客コンテキスト
──────────────────       ─────────────────       ────────────────
Reservation（集約）       Staff                   Customer
TimeSlot                 Schedule                ContactInfo
WaitingList              Availability
```

このコンテキスト分割が、コーディングAIへの委譲単位になる。

```
予約コンテキストのdocuments・specificationを渡します。
このコンテキストだけ実装してください。
```

---

## チュートリアルの章構成

```
第1章: 要件を言語化し、コンテキストを識別する
  └─ 予約システムの要件をAIとの対話で固める
  └─ 実例マッピングとイベントストーミングをサイクルで回して要件を深掘りする
  └─ 境界付けられたコンテキストを識別する
  ゴール: 3つのコンテキスト（予約・スタッフ・顧客）とその境界が明確になっている状態

第2章: コンテキストごとに要求定義を作る
  └─ コンテキストごとに ubiquitous_language / commands_and_events / domain_model を作る
  └─ 集約の境界・競合制御を domain_model に表現する
  ゴール: 各コンテキストの要求定義が揃っている状態

第3章: ADRを複数書き、相互参照する
  └─ 競合制御の実装方針
  └─ コンテキスト間の通信方式（同期/非同期）
  └─ ADR間の相互参照を整理する
  ゴール: 主要な設計判断がADRとして記録・相互参照されている状態

第4章: コンテキスト単位で詳細設計を生成・実装する
  └─ 予約コンテキスト → スタッフコンテキスト → 顧客コンテキストの順で進める
  └─ コンテキスト単位でコーディングAIに委譲する体験をする
  ゴール: 各コンテキストのシナリオを満たす動くアプリケーションがある状態

第5章: コンポーネントテストを追加する
  └─ 複雑なドメインルール（二重予約防止等）をコンポーネントテストでカバーする
  ゴール: ドメインルールがコンポーネントテストで保護されている状態

振り返り
  └─ 3つのチュートリアルを通じて何を学んだか
  └─ 実務への応用ガイド
```

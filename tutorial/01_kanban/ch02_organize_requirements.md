# 第2章: 要求定義を整える

## この章のゴール

> **要求定義が一通り揃っている状態**

第1章で言語化した要件・シナリオ・判断を、プロジェクトのドキュメント構造に落とす。
ここで作るファイルが、第3章でAIに詳細設計を生成させるための入力になる。

```
documents/requirements/
├── overview.md                      # プロダクト要件概要（What + Why + スコープ外）
├── features/
│   └── 01_card_management.feature   # 振る舞いシナリオ（Gherkin）
├── domain/
│   └── ubiquitous_language.md       # ユビキタス言語
└── adr/
    ├── ADR-001_frontend_framework.md
    └── ADR-002_data_persistence.md
```

---

## なぜ形式を整えるのか

第1章のメモをそのままAIに渡しても動きはする。
しかし形式を整えることで、以下の利点が生まれる。

- **AIへの指示精度が上がる:** 構造化された入力はAIが誤解しにくい
- **人間のレビューがしやすくなる:** どこに何があるか自明になる
- **変更時の影響範囲が分かる:** 仕様が変わったときに何を更新すべきか明確になる

> `guide/03_directory_structure.md` に設計思想の詳細がある。

## ステップ1: プロダクト要件概要を作る（overview.md）

第1章の「機能要件」と「スコープ外」をプロダクト要件概要にまとめる。
**What（何を作るか）とWhy（なぜそうするか）** を書くのがポイント。

### AIへの依頼

overview.mdという以下のフォーマットに従ったファイルを作りたい。
第1章で作成した壁打ちメモを、フォーマットを指定したプロンプトの末尾に貼り付けることで、
AIに生成を依頼する。

```
以下の壁打ちメモを元に、overview.md として使える内容を作成してください。
以下のフォーマットに従ってください。
（環境が対応していれば、そのままファイルとして書き出しても構いません）

フォーマット:
- 概要（このシステムは何か、1〜2文）
- 機能要件（箇条書き）
- 非機能要件（箇条書き）
- スコープ外（何を作らないか）
- 未決事項（メモの中で結論が出ていないもの）

メモの中に矛盾や曖昧な箇所があれば指摘してください。

---
# Kanbanボード 壁打ちメモ

## 機能要件（スコープ内）
- ボードには3つの固定カラム: Todo / In Progress / Done
- カードを新規作成できる（タイトルを入力）
- カードは作成時にTodoカラムに追加される
- カードを隣接するカラムに移動できる
- カードを削除できる
- ボードの状態はリロードしても保持される

## スコープ外
- カラムの追加・削除・並び替え
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え（ドラッグで順序変更）

## 判断済みの問い
- カラムの飛び越し移動 → 許可しない（隣接カラムのみ）
- 削除の取り消し → なし（スコープ外）
- 空タイトルでのカード作成 → 許可しない
```

> **補足:** 壁打ちメモが雑多な場合は、2段階プロンプトが効果的である（`guide/02_development_process.md` 参照）。

### AIの出力をレビューする

AIが生成したoverview.mdを、以下の観点でレビューする。

- 機能要件に抜け漏れがないか（第1章の判断がすべて反映されているか）
- スコープ外の記載が十分か
- 「なぜ」が読み取れるか（例: 「飛び越し不可」だけでなく「隣接カラムのみ」と意図が分かるか）
- 非機能要件としてAIが補完した内容は妥当か

今回のチュートリアルでは、レビューの結果、以下の内容を採用した。

### 成果物: overview.md

```markdown
# Kanbanボード 仕様書

## 概要

カード（タスク）を3つの固定カラム間で移動して管理するシンプルなKanbanボード。
1人のユーザーがブラウザ上で操作する。

## 機能要件

### ボード
- 3つの固定カラムを持つ: Todo / In Progress / Done
- カラムの追加・削除・並び替えはできない

### カード
- タイトル（必須・空文字不可）を持つ
- 作成時にTodoカラムに追加される
- 隣接するカラムにのみ移動できる（飛び越し不可）
- 削除できる（取り消し不可）

### 永続化
- ボードの状態はリロードしても保持される

## 非機能要件

- ブラウザで動作するWebアプリケーション
- レスポンスは即座に反映される（楽観的UI）

## スコープ外

- カラムのカスタマイズ（追加・削除・並び替え）
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え
- 削除の取り消し
```

## ステップ2: featureファイルを作る（features/）

第1章の粗いシナリオを正式なGherkin形式に整える。

> Gherkinの書き方の詳細は `guide/appendix/A_bdd.md` を参照。

### AIへの依頼

01_card_management.feature という正式なGherkin形式のfeatureファイルを作りたい。
第1章で採用した実例マッピングとシナリオを、以下のルールを指定したプロンプトの末尾に貼り付けることで、
AIに変換を依頼する。

```
以下の実例マッピングと振る舞いシナリオを、
正式なGherkin形式のfeatureファイルとして使える内容に変換してください。
以下のルールに従ってください。
（環境が対応していれば、そのままファイルとして書き出しても構いません）

- 1ファイル = 1機能（Feature）
- 実例マッピングのルールごとにコメントで区切る
- Background で共通の前提条件をまとめる
- 正常系・異常系を同じファイルにまとめる
- シナリオ名は具体的に書く

---
# 実例マッピング: カード管理

## ストーリー
カードを作成・移動・削除してタスクを管理できる

## ルールと例

### ルール: カードにはタイトルが必須
- 例: タイトル「買い物リストを作る」でカード作成 → 成功、Todoに追加される
- 例: タイトル空でカード作成 → 失敗、エラー表示

### ルール: カードは隣接カラムにのみ移動できる
- 例: TodoからIn Progressに移動 → 成功
- 例: TodoからDoneに直接移動 → 失敗、移動されない
- 例: DoneからIn Progressに戻す → 成功

### ルール: カードは作成時にTodoに配置される
- 例: カードを新規作成する → Todoカラムに表示される

### ルール: ボードの状態はリロード後も保持される
- 例: カードを配置した状態でリロード → 同じ状態が復元される

## 疑問
- [ ] 削除したカードは元に戻せるか？ → 今回はスコープ外と判断済み
- [ ] 同じタイトルのカードを複数作れるか？
```

### AIの出力をレビューする

AIが生成したfeatureファイルを、以下の観点でレビューする。

- 第1章のシナリオがすべて反映されているか
- シナリオ名だけ読んで「何をテストしているか」分かるか
- 1シナリオに複数のWhenが入っていないか
- Background で共通の前提条件がまとめられているか

今回のチュートリアルでは、レビューの結果、以下の内容を採用した。

### 第1章との差分

粗いシナリオから以下が改善されている。

| 改善点 | 例 |
|--------|-----|
| Background で共通前提をまとめた | 「ボードが表示されている」を毎回書かなくてよい |
| シナリオの並びを整理した | 作成 → 移動 → 削除 → 永続化の順 |
| 逆方向移動のシナリオを追加した | Done → In Progress |
| 移動失敗時の「残っている」確認を追加した | And Todoカラムに「タスクA」が残っている |

### 成果物: features/01_card_management.feature

```gherkin
Feature: カード管理

  Kanbanボード上でカードの作成・移動・削除を行う。

  Background:
    Given Kanbanボードが表示されている

  # --- 作成 ---

  Scenario: カードを新規作成する
    When タイトル「買い物リストを作る」で新しいカードを作成する
    Then Todoカラムに「買い物リストを作る」が表示される

  Scenario: 空のタイトルではカードを作成できない
    When タイトルを空のまま新しいカードを作成しようとする
    Then カードは作成されない
    And エラーメッセージが表示される

  # --- 移動 ---

  Scenario: カードをTodoからIn Progressに移動する
    Given Todoカラムに「タスクA」がある
    When 「タスクA」をIn Progressに移動する
    Then In Progressカラムに「タスクA」が表示される
    And Todoカラムから「タスクA」が消えている

  Scenario: カードをIn ProgressからDoneに移動する
    Given In Progressカラムに「タスクA」がある
    When 「タスクA」をDoneに移動する
    Then Doneカラムに「タスクA」が表示される

  Scenario: カードをDoneからIn Progressに戻す
    Given Doneカラムに「タスクA」がある
    When 「タスクA」をIn Progressに移動する
    Then In Progressカラムに「タスクA」が表示される

  Scenario: TodoからDoneに直接移動できない
    Given Todoカラムに「タスクA」がある
    When 「タスクA」をDoneに移動しようとする
    Then 移動は行われない
    And Todoカラムに「タスクA」が残っている

  # --- 削除 ---

  Scenario: カードを削除する
    Given Todoカラムに「タスクA」がある
    When 「タスクA」を削除する
    Then ボードから「タスクA」が消えている

  # --- 永続化 ---

  Scenario: ボードの状態がリロード後も保持される
    Given Todoカラムに「タスクA」がある
    And In Progressカラムに「タスクB」がある
    When ページをリロードする
    Then Todoカラムに「タスクA」が表示される
    And In Progressカラムに「タスクB」が表示される
```

## ステップ3: ユビキタス言語を定義する（domain/）

プロジェクト内で使う言葉を統一する。
Kanbanボードのように身近な題材でも、言葉を明示的に定義することで認識のズレを防ぐ。

> ユビキタス言語の詳細は `guide/appendix/B_ddd.md` を参照。

### AIへの依頼

ubiquitous_language.md というユビキタス言語の定義ファイルを作りたい。
ここまでに作成した overview.md と 01_card_management.feature を、
以下のプロンプトの末尾に貼り付けることで、AIに用語の整理を依頼する。

```
以下の仕様書とfeatureファイルに登場する用語を整理して、
ユビキタス言語の定義一覧として使える内容を作成してください。
以下のフォーマットに従ってください。
（環境が対応していれば、そのままファイルとして書き出しても構いません）

フォーマット:
- カテゴリごとに表形式で整理
- 各用語に英語名を併記
- 定義はfeatureファイルの振る舞いと矛盾しないようにする

---
[overview.md の内容を貼る]

---
[features/01_card_management.feature の内容を貼る]
```

### AIの出力をレビューする

AIが生成したユビキタス言語を、以下の観点でレビューする。

- 同じ意味の言葉が複数存在しないか（例: 「タスク」と「カード」が混在していないか）
- 定義が曖昧でないか（「適切に処理する」のような記述がないか）
- featureファイルで使っている言葉と一致しているか

今回のチュートリアルでは、レビューの結果、以下の内容を採用した。

### 成果物: domain/ubiquitous_language.md

```markdown
# ユビキタス言語

## ボード関連

| 用語 | 定義 |
|------|------|
| **ボード（Board）** | カラムとカードを表示・管理する画面全体 |
| **カラム（Column）** | カードの状態を表すグループ。Todo / In Progress / Done の3つが固定で存在する |
| **カード（Card）** | タスクを表す最小単位。タイトルを1つ持つ。いずれか1つのカラムに属する |

## 操作関連

| 用語 | 定義 |
|------|------|
| **作成（Create）** | 新しいカードをTodoカラムに追加すること |
| **移動（Move）** | カードを隣接するカラムに移すこと。飛び越し移動はできない |
| **削除（Delete）** | カードをボードから完全に取り除くこと。取り消しはできない |

## 状態関連

| 用語 | 定義 |
|------|------|
| **Todo** | 未着手のカードが属するカラム。カード作成時の初期状態 |
| **In Progress** | 作業中のカードが属するカラム |
| **Done** | 完了したカードが属するカラム |
```

## ステップ4: ADRを書く（adr/）

このチュートリアルでは技術選定の判断を2つADRとして記録する。

> ADRの書き方の詳細は `guide/appendix/D_adr.md` を参照。

### なぜチュートリアルでADRを書くのか

ADRの真の価値は「後から別の会話でAIに渡したとき、判断の背景ごと伝えられる」点にある。
第3章でAIに詳細設計を生成させるとき、ADRがあると意図に反した技術選定が起きにくくなる。

### AIへの依頼

ADRファイルを作りたい。技術的判断の内容を、以下のテンプレートを指定したプロンプトに記載することで、
AIにADR形式への整理を依頼する。

今回のチュートリアルでは、個人の学習プロジェクトとして手軽に始められることを重視している。
この前提のもと、以下の2つの技術的判断を行った。

- フロントエンドは **React + TypeScript** — 情報量が多く、学習中に詰まっても解決しやすい
- データの永続化は **ローカルストレージ** — サーバー不要で、ブラウザだけで完結する

実際のプロジェクトでは、チームのスキルセットや運用要件に応じて判断は変わる。
ここで大事なのは選定の結果ではなく、「なぜそう決めたか」をADRとして残すことである。

```
以下の技術的判断について、ADR形式として使える内容を作成してください。
以下のテンプレートに従ってください。
特に「理由」と「選ばなかった選択肢」を必ず含めてください。
（環境が対応していれば、そのままファイルとして書き出しても構いません）

テンプレート:
- タイトル: 何を決めたかを一言で
- ステータス: 採用済み
- 文脈: なぜこの判断が必要になったか
- 決定: 何を選んだか
- 理由: なぜそれを選んだか。選ばなかった選択肢とその理由も含める
- 影響: この決定によるトレードオフ
- 参照: 関連するADR

判断1: フロントエンドはReact + TypeScriptを使う
判断2: データの永続化はローカルストレージを使う（バックエンドサーバーは立てない）
```

> **注意:** 技術選定は学習者の好みやスキルに合わせて変えてよい。
> ここではチュートリアルの手軽さを優先して上記を選んでいる。
> 重要なのは「何を選んだか」ではなく「なぜ選んだか」を記録することである。

### AIの出力をレビューする

AIが生成したADRを、以下の観点でレビューする。

- 「理由」に「なんとなく」ではなく具体的な根拠があるか
- 「選ばなかった選択肢」が書かれているか
- 他のADRとの相互参照が適切か

今回のチュートリアルでは、レビューの結果、以下の内容を採用した。

### 成果物: adr/ADR-001_frontend_framework.md

```markdown
# ADR-001: フロントエンドフレームワークにReactを採用する

## ステータス
採用済み

## 文脈
Kanbanボードはブラウザ上で動作するWebアプリケーションである。
カードの作成・移動・削除といった状態変更をリアルタイムにUIへ反映する必要がある。
チュートリアルの題材であるため、学習コストと情報量の多さを重視する。

## 決定
React + TypeScript を採用する。

## 理由
- コンポーネントベースの設計がカラム・カードの構造と相性が良い
- 状態管理のエコシステムが豊富
- 情報量が多く、学習中に詰まっても解決しやすい
- TypeScriptにより型安全な開発ができる

### 選ばなかった選択肢
- **Vue.js:** 候補だったが、Reactの方がエコシステムの情報量で優位と判断
- **素のHTML/JS:** チュートリアルとしてはシンプルだが、状態管理が煩雑になりやすい
- **Svelte:** 軽量で魅力的だが、情報量がReact/Vueに比べて少ない

## 影響
- Reactの基礎知識が前提になる
- ビルドツール（Vite等）のセットアップが必要

## 参照
- ADR-002（データ永続化方式）
```

### 成果物: adr/ADR-002_data_persistence.md

```markdown
# ADR-002: データ永続化にローカルストレージを採用する

## ステータス
採用済み

## 文脈
ボードの状態（カードとその所属カラム）をリロード後も保持する必要がある。
チュートリアルの題材であり、インフラの構築コストを最小限にしたい。
複数ユーザーの同時利用は想定しない（スコープ外）。

## 決定
ブラウザのローカルストレージ（localStorage）を使用する。
バックエンドサーバーは立てない。

## 理由
- サーバー不要で環境構築が簡単
- チュートリアルのスコープ（1人利用）に十分
- JSON形式でカード一覧を保存するだけで要件を満たせる

### 選ばなかった選択肢
- **バックエンドAPI + DB:** 本格的だが、チュートリアル1の範囲では過剰
- **IndexedDB:** localStorageで十分な規模のため不要
- **ファイル保存:** ブラウザから直接ファイルに書くのはUX上好ましくない

## 影響
- データ量が大きくなるとlocalStorageの容量制限（5MB程度）に引っかかる可能性がある
- 別ブラウザ・別デバイスではデータが共有されない
- チュートリアル2以降でバックエンドを導入する場合、永続化層を差し替える必要がある

## 参照
- ADR-001（フロントエンドフレームワーク）
```

## 振り返り

- **形式は目的ではなく手段:** フォーマットに落とすことで、AIへの入力精度とレビューのしやすさが上がった
- **AIに整形を任せる:** 粗いメモを渡して「この形式にしてください」と頼むのが効率的
- **ユビキタス言語の価値:** 言葉を統一することで、仕様書・シナリオ・AIへの指示がブレなくなる
- **ADRの価値:** 「なぜこの技術にしたか」を記録しておくと、第3章でAIが詳細設計を生成するときに矛盾が減る

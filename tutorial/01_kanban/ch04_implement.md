# 第4章: 実装する

## この章のゴール

> **シナリオを満たす動くアプリケーションがある状態**

詳細設計をAIに渡して実装させる。
featureファイルのシナリオを受け入れ条件として使い、シナリオが満たされているかで品質を判断する。

---

## フェーズ3の位置づけ

```
フェーズ1: 要求定義を育てる（第1章・第2章）← 完了
フェーズ2: 詳細設計を生成する（第3章）    ← 完了
フェーズ3: 実装する（この章）              ← いまここ
```

フェーズ3はAIが主導し、人間がレビューする。
`guide/04_collaboration_matrix.md` でいう「委譲型」の協調パターンである。

---

## 実装の進め方

### 原則: feature単位・1タスク1依頼

`guide/02_development_process.md` にある通り、実装は **feature単位・1タスク1依頼** が最も効果的。
大きな依頼を一度に出すより、小さな単位で依頼してレビューを挟む方が品質が安定する。

このチュートリアルでは以下の順序で進める。

```
① プロジェクトのセットアップ
② データ構造・永続化層の実装
③ ボード・カラム・カードの表示
④ カードの作成
⑤ カードの移動
⑥ カードの削除
⑦ 全シナリオの確認
```

---

## ステップ1: プロジェクトのセットアップ

### AIへの依頼

```
documents/specification/architecture.md を読んで、
implement/ にReact + TypeScript + Vite のプロジェクトをセットアップしてください。

ディレクトリ構成は architecture.md に従ってください。
ADR-001, ADR-002 も参照してください。
```

### 人間が確認すること

- `npm run dev` でローカルサーバーが起動するか
- ディレクトリ構成が architecture.md と一致しているか
- TypeScript の設定が入っているか

---

## ステップ2: データ構造・永続化層の実装

### AIへの依頼

```
documents/specification/backend/api_design.md を読んで、以下を実装してください。

1. src/types/ に型定義（Card, Board, ColumnId）
2. src/storage/ に localStorage の保存・復元関数

以下のシナリオを意識してください。
「ボードの状態がリロード後も保持される」
```

### 人間が確認すること

- 型定義がユビキタス言語と一致しているか
- localStorage の保存・復元が正しく動くか（ブラウザの開発者ツールで確認できる）

---

## ステップ3: ボード・カラム・カードの表示

### AIへの依頼

```
documents/specification/architecture.md のコンポーネント構成に従い、
Board, Column, Card コンポーネントを実装してください。

まずは固定データで3つのカラムとサンプルカードが表示される状態にしてください。
```

### 人間が確認すること

- 3つのカラム（Todo / In Progress / Done）が表示されるか
- カードが正しいカラムに表示されるか
- 見た目が最低限整っているか（細かいデザインは後でよい）

---

## ステップ4: カードの作成

### AIへの依頼

```
カード作成機能を実装してください。
以下のシナリオを受け入れ条件として満たしてください。

Scenario: カードを新規作成する
  Given Kanbanボードが表示されている
  When タイトル「買い物リストを作る」で新しいカードを作成する
  Then Todoカラムに「買い物リストを作る」が表示される

Scenario: 空のタイトルではカードを作成できない
  Given Kanbanボードが表示されている
  When タイトルを空のまま新しいカードを作成しようとする
  Then カードは作成されない
  And エラーメッセージが表示される

documents/specification/backend/sequences/card_create.md も参照してください。
```

### 人間が確認すること

- ブラウザ上でカードが作成できるか
- 空タイトルでエラーが表示されるか
- 作成したカードがTodoカラムに追加されるか
- リロードしてもカードが残っているか（永続化）

> **ポイント:** シナリオをそのまま受け入れ条件として渡す。
> これが最もブレが少ないAIへの指示になる。

---

## ステップ5: カードの移動

### AIへの依頼

```
カード移動機能を実装してください。
以下のシナリオをすべて受け入れ条件として満たしてください。

Scenario: カードをTodoからIn Progressに移動する
  Given Todoカラムに「タスクA」がある
  When 「タスクA」をIn Progressに移動する
  Then In Progressカラムに「タスクA」が表示される
  And Todoカラムから「タスクA」が消えている

Scenario: カードをIn ProgressからDoneに移動する
  Given In Progressカラムに「タスクA」がある
  When 「タスクA」をDoneに移動する
  Then Doneカラムに「タスクA」が表示される

Scenario: カードをDoneからIn Progressに戻す
  Given Doneカラムに「タスクA」がある
  When 「タスクA」をIn Progressに移動する
  Then In Progressカラムに「タスクA」が表示される

Scenario: TodoからDoneに直接移動できない
  Given Todoカラムに「タスクA」がある
  When 「タスクA」をDoneに移動しようとする
  Then 移動は行われない
  And Todoカラムに「タスクA」が残っている

documents/specification/backend/sequences/card_move.md も参照してください。
UIは各カードに「→」「←」ボタンを付ける形で構いません。
```

### 人間が確認すること

- 各方向の移動が正しく動くか
- 飛び越し移動ができないか（Todoのカードに「Done」方向のボタンが2回押せないか等）
- 端のカラム（Todoの「←」、Doneの「→」）で不要なボタンが表示されないか
- 移動後にリロードしても状態が保持されるか

---

## ステップ6: カードの削除

### AIへの依頼

```
カード削除機能を実装してください。
以下のシナリオを受け入れ条件として満たしてください。

Scenario: カードを削除する

  When 「タスクA」を削除する
```

## ステップ7: 全シナリオの確認

全機能の実装が完了したら、featureファイルの全シナリオを手動で確認する。

### 確認チェックリスト

| # | シナリオ | 結果 |
|---|---------|------|
| 1 | カードを新規作成する | |
| 2 | 空のタイトルではカードを作成できない | |
| 3 | カードをTodoからIn Progressに移動する | |
| 4 | カードをIn ProgressからDoneに移動する | |
| 5 | カードをDoneからIn Progressに戻す | |
| 6 | TodoからDoneに直接移動できない | |
| 7 | カードを削除する | |
| 8 | ボードの状態がリロード後も保持される | |

> **ポイント:** この手動確認を第5章でPlaywrightに置き換える。
> featureファイルのシナリオがそのままテストケースになることを、次章で体験する。

## うまくいかないとき

AIの実装が期待と異なる場合、以下のパターンで対処する。

### パターン1: シナリオを満たしていない

```
以下のシナリオが満たされていません。修正してください。

Scenario: TodoからDoneに直接移動できない
  → 現状: ボタンは非表示だがドラッグで移動できてしまう
  → 期待: どの手段でも移動できないこと
```

### パターン2: 仕様の解釈が異なる

```
カードの移動UIについて認識が異なります。
現状: ドラッグ&ドロップで移動する
期待: 「→」「←」ボタンで移動する

理由: チュートリアル1ではシンプルさを優先し、ボタンUIを採用したい。
ドラッグ&ドロップは将来の改善として検討する。
```

### パターン3: 実装の品質が気になる

```
動作は正しいですが、以下の点を改善してください。
- Card コンポーネントが Board の状態を直接操作している
  → props 経由でコールバックを渡す形にしてください
```

いずれの場合も **「何が問題で、どうなっていてほしいか」** を具体的に伝える。

## 振り返り

- **feature単位の委譲:** 小さな単位で依頼してレビューを挟む方が、一括で依頼するより品質が安定した
- **シナリオが受け入れ条件:** 「このシナリオを満たしてください」が最も明確な指示だった
- **詳細設計の価値:** 設計書があることで、AIが「どう作るか」を大きく外すことが少なかった
- **人間のレビュー:** AIの実装をそのまま受け入れず、シナリオで確認するサイクルが重要

# 第1章: 壁打ちで要件を言語化する

## この章のゴール

> **要件と粗いシナリオが言語化されている状態**

具体的には、以下が手元にある状態を目指す。

- Kanbanボードで「何ができて、何ができないか」が言葉になっている
- 粗い振る舞いシナリオ（Given/When/Then）が数本書き出されている
- スコープ外が明示されている

まだドキュメントの形式に整える必要はない。メモやテキストファイルで十分。
形式に落とすのは第2章で行う。

---

## 壁打ちとは何か

壁打ちとは、**頭の中にある漠然としたイメージをAIとの対話を通じて言葉にしていく作業**である。

`guide/01_philosophy.md` では、言語化には2つの段階があると述べている。

```
第1段階: 頭の中にあるものを言葉にする
  → 壁打ちが有効なのはここ
  → 問い返されることで「自分がそう思っていたこと」に気づける

第2段階: 言葉にしたものを相手に伝わる形に整える
  → AIが整理・補完を担える
```

この章では主に第1段階を体験する。
最初のシナリオや仕様が粗くても構わない。**往復するうちに言語化が深まり、仕様が育っていく**。

---

## ステップ1: 最初の問いかけ

AIを起動し、以下のように問いかけてみよう。

```
シンプルなKanbanボードを作りたいと思っています。
カード（タスク）をカラム（状態）間で移動できるものです。
まず機能要件だけに絞って、箇条書きで出してもらえますか？
```

> **Tips: まとめて聞くと、まとめて返ってくる**
>
> AIは問いかけの広さに応じた量で返してくる。
> 「要件を整理して」と聞けば要件・設計・注意点まで一気に出てくるし、
> 「機能要件だけ箇条書きで」と聞けばそこだけ返ってくる。
> 壁打ちでは、一度に受け取る量を自分で制御する意識を持つとよい。
>
> 例えば上のプロンプトから「まず機能要件だけに絞って」を外して
> 「要件を一緒に整理してもらえますか？」と聞くと、
> 設計方針や技術選定まで一度に返ってきて議論しづらくなることがある。
> まず必要な範囲の回答を得てから、気になる点を掘り下げていくとよい。

### AIが問い返してくること

AIは曖昧な部分を具体化するために問い返してくる。例えば以下のような問いが返ってくるだろう。

- カラムは固定か、ユーザーがカスタムできるか？
- カードにはどんな情報を持たせるか（タイトルだけ？説明文？期限？）
- 複数ユーザーで使うか、1人で使うか？
- カードの並び順はどうするか？
- WIP（作業中）の上限はあるか？

### 人間が判断すること

これらの問いに対して「どうしたいか」を決めるのは人間の役割である。
技術的な正解があるわけではなく、**「この開発で達成したいことは何か」**を起点に判断する。
個人の便利ツールなのか、限られた内部メンバーで使うものなのか、不特定多数に公開するプロダクトなのかでも、同じ問いへの答えは変わってくる。

このチュートリアルでは以下のように決める。

| 問い | 判断 | 理由 |
|------|------|------|
| カラム構成 | 固定（Todo / In Progress / Done） | シンプルにする。カスタムは今回不要 |
| カードの情報 | タイトルのみ | 最小限で始める |
| ユーザー | 1人（認証なし） | チュートリアルの範囲を絞る |
| 並び順 | 作成順（上が新しい） | 直感的で実装もシンプル |
| WIP制限 | なし | チュートリアル2以降で扱えばよい |

> **ポイント:** 「なし」「不要」と判断したものも明示的に記録する。
> これが後のスコープ外リストになる。

## ステップ2: 要件を言葉にする

壁打ちで出てきた要件を一通り洗い出し、箇条書きで整理する。
AIに「ここまでの内容を要件として整理してください」と頼むのも有効。

ステップ1の壁打ちからは、例えば以下のような機能要件が出てきたとする。

```
- ボードには3つの固定カラムがある: Todo / In Progress / Done
- カードを新規作成できる（タイトルを入力）
- カードは作成時にTodoカラムに追加される
- カードを隣接するカラムに移動できる
- カードを削除できる
- ボードの状態はリロードしても保持される
- カラムの追加・削除・並び替え
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え（ドラッグで順序変更）
```

> **補足:** `guide/02_development_process.md` ではユーザーストーリー形式（`As a / I want / so that`）
> での記述を推奨している。このチュートリアルではアクターが1人なので箇条書きで十分だが、
> アクターが複数いる場合（チュートリアル2以降）はストーリー形式にすると「誰のための機能か」が明確になる。

### 優先づけとスコープの決定

洗い出した要件すべてを一度に作る必要はない。
以下のような観点で優先づけを行い、今回のスコープに入れるものと入れないものを分ける。

- **なくてはならない機能か？** — それがないとプロダクトとして成立しない機能はMVPに含める
- **開発の目的に照らして必要か？** — 学習目的なら最小限でよいし、業務利用なら運用に耐える機能が要る
- **今の開発体制・期間で現実的か？** — 人数・スキル・期限を踏まえて、今回やるかどうかを決める
- **後から追加できるか？** — 後回しにしてもアーキテクチャに影響しないものは後のイテレーションに回せる

このチュートリアルでは、個人用の学習プロジェクトとして最小限の構成を目指す。

#### 今回のスコープ

```
- ボードには3つの固定カラムがある: Todo / In Progress / Done
- カードを新規作成できる（タイトルを入力）
- カードは作成時にTodoカラムに追加される
- カードを隣接するカラムに移動できる
- カードを削除できる
- ボードの状態はリロードしても保持される
```

#### スコープ外

```
- カラムの追加・削除・並び替え
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え（ドラッグで順序変更）
```

### 壁打ちで気づくこと

この整理をしていると、新しい問いが浮かぶことがある。

- 「隣接するカラムにしか移動できない？ TodoからDoneに直接移動は？」
- 「削除したカードは元に戻せる？」
- 「空のタイトルでカードを作れてしまう？」

これらは振る舞いシナリオとして書き出すべきヒントになる。
今は「気づいた」ことが重要で、すべてに即答する必要はない。

このチュートリアルでは以下のように決める。

| 問い | 判断 |
|------|------|
| カラムの飛び越し移動 | 許可しない（隣接カラムのみ） |
| 削除の取り消し | なし（スコープ外） |
| 空タイトル | 作成を許可しない |

## ステップ3: 実例マッピングでルール・例・疑問を整理する

壁打ちで出た要件やメモを、いきなりGherkinシナリオにするのではなく、
まず**実例マッピング**の形式に整理する。

> 実例マッピングの詳細は `guide/appendix/A_bdd.md` を参照。

### 実例マッピングとは

ストーリー（作りたい機能）を「ルール」「例」「疑問」に分解する発見手法である。
ルールを先に明示することで、シナリオの「なぜこの例が必要か」が明確になる。

### AIへの依頼（2段階プロンプト・ステージ1）

壁打ちメモを渡して、実例マッピングの形式に整理を依頼する。
今回のチュートリアルでは、ステップ1・2の内容を以下のようにまとめた。

```
# Kanbanボード 壁打ちメモ

## 機能要件（スコープ内）
- ボードには3つの固定カラム: Todo / In Progress / Done
- カードを新規作成できる（タイトルを入力）
- カードは作成時にTodoカラムに追加される
- カードを隣接するカラムに移動できる
- カードを削除できる
- ボードの状態はリロードしても保持される

## スコープ外
- カラムの追加・削除・並び替え
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え（ドラッグで順序変更）

## 判断済みの問い
- カラムの飛び越し移動 → 許可しない（隣接カラムのみ）
- 削除の取り消し → なし（スコープ外）
- 空タイトルでのカード作成 → 許可しない
```

このメモを、以下のフォーマットを指定したプロンプトの末尾に貼り付けることで、AIに実例マッピングへの整理を依頼する。

```
以下はKanbanボードについての壁打ちメモです。
これを実例マッピングの形式に整理してください。

フォーマット:
- ストーリー: 1文で
- ルールと例: ルールごとに、成功する例・失敗する例を列挙
- 疑問: メモの中で結論が出ていないもの、暗黙の前提になっているもの

メモに書かれていないが当然ありそうなルール（暗黙のルール）も候補として提示してください。

---
# Kanbanボード 壁打ちメモ

## 機能要件（スコープ内）
- ボードには3つの固定カラム: Todo / In Progress / Done
- カードを新規作成できる（タイトルを入力）
- カードは作成時にTodoカラムに追加される
- カードを隣接するカラムに移動できる
- カードを削除できる
- ボードの状態はリロードしても保持される

## スコープ外
- カラムの追加・削除・並び替え
- カードの説明文・期限・ラベル
- 複数ユーザー・認証
- WIP制限
- カード内での並び替え（ドラッグで順序変更）

## 判断済みの問い
- カラムの飛び越し移動 → 許可しない（隣接カラムのみ）
- 削除の取り消し → なし（スコープ外）
- 空タイトルでのカード作成 → 許可しない
```

### AIの出力をレビューする

AIは暗黙のルールや疑問点も含めて、充実した実例マッピングを返してくる。
ただし、すべてをそのまま採用する必要はない。以下の観点でレビューする。

- AIが提示した「暗黙のルール」候補は妥当か？
- 各ルールに対して正常例・異常例の両方があるか？
- 疑問に対して判断を下す（今決めるか、後回しにするか）
- 今回の開発の範囲に照らして、扱うルールはどれか？

> **ポイント:** ルールが明示されると、「このルールに対する異常系は？」と考えやすくなる。
> いきなりシナリオを書くよりも、ルールを先に出す方が網羅性が自然に上がる。

今回のチュートリアルでは、レビューの結果、以下の4ルールに絞って進めることにした。

```markdown
# 実例マッピング: カード管理

## ストーリー
カードを作成・移動・削除してタスクを管理できる

## ルールと例

### ルール: カードにはタイトルが必須
- 例: タイトル「買い物リストを作る」でカード作成 → 成功、Todoに追加される
- 例: タイトル空でカード作成 → 失敗、エラー表示

### ルール: カードは隣接カラムにのみ移動できる
- 例: TodoからIn Progressに移動 → 成功
- 例: TodoからDoneに直接移動 → 失敗、移動されない
- 例: DoneからIn Progressに戻す → 成功

### ルール: カードは作成時にTodoに配置される
- 例: カードを新規作成する → Todoカラムに表示される

### ルール: ボードの状態はリロード後も保持される
- 例: カードを配置した状態でリロード → 同じ状態が復元される

## 疑問
- [ ] 削除したカードは元に戻せるか？ → 今回はスコープ外と判断済み
- [ ] 同じタイトルのカードを複数作れるか？
```

## ステップ4: 実例マッピングからシナリオを書き出す

実例マッピングが整理できたら、その成果をGiven/When/Then形式に変換する。

### AIへの依頼（2段階プロンプト・ステージ2）

レビュー済みの実例マッピングを、以下のフォーマットを指定したプロンプトの末尾に貼り付けることで、
AIにGiven/When/Then形式のシナリオへの変換を依頼する。

```
以下の実例マッピングの内容を、Given/When/Then形式のシナリオに変換してください。
ルールごとにコメントで区切ってください。
疑問の項目は # TODO コメントとして残してください。
この段階では完璧なGherkin構文でなくて構わないです。

---
# 実例マッピング: カード管理

## ストーリー
カードを作成・移動・削除してタスクを管理できる

## ルールと例

### ルール: カードにはタイトルが必須
- 例: タイトル「買い物リストを作る」でカード作成 → 成功、Todoに追加される
- 例: タイトル空でカード作成 → 失敗、エラー表示

### ルール: カードは隣接カラムにのみ移動できる
- 例: TodoからIn Progressに移動 → 成功
- 例: TodoからDoneに直接移動 → 失敗、移動されない
- 例: DoneからIn Progressに戻す → 成功

### ルール: カードは作成時にTodoに配置される
- 例: カードを新規作成する → Todoカラムに表示される

### ルール: ボードの状態はリロード後も保持される
- 例: カードを配置した状態でリロード → 同じ状態が復元される

## 疑問
- [ ] 削除したカードは元に戻せるか？ → 今回はスコープ外と判断済み
- [ ] 同じタイトルのカードを複数作れるか？
```

> **ポイント:** 2段階に分けることで、ステージ1で「ルールと例が正しいか」を確認してから
> シナリオに進められる。いきなりGherkinが出てくるよりもレビューしやすい。

### AIの出力をレビューする

AIが生成したシナリオを、以下の観点でレビューする。

- 実例マッピングのルール・例がシナリオとして網羅されているか？
- Given/When/Thenの各ステップが具体的で、曖昧な表現になっていないか？
- 正常系とエッジケースが区別されているか？
- 不要なシナリオや重複したシナリオがないか？

今回のチュートリアルでは、レビューの結果、以下のシナリオを採用した。

#### 正常系シナリオ

```gherkin
シナリオ: カードを新規作成する
  Given ボードが表示されている
  When  タイトル「買い物リストを作る」で新しいカードを作成する
  Then  Todoカラムに「買い物リストを作る」が表示される

シナリオ: カードをTodoからIn Progressに移動する
  Given Todoカラムに「買い物リストを作る」がある
  When  「買い物リストを作る」をIn Progressに移動する
  Then  In Progressカラムに「買い物リストを作る」が表示される
  And   Todoカラムから「買い物リストを作る」が消えている

シナリオ: カードをIn ProgressからDoneに移動する
  Given In Progressカラムに「買い物リストを作る」がある
  When  「買い物リストを作る」をDoneに移動する
  Then  Doneカラムに「買い物リストを作る」が表示される

シナリオ: カードを削除する
  Given Todoカラムに「買い物リストを作る」がある
  When  「買い物リストを作る」を削除する
  Then  ボードから「買い物リストを作る」が消えている

シナリオ: ボードの状態が保持される
  Given Todoカラムに「タスクA」、In Progressカラムに「タスクB」がある
  When  ページをリロードする
  Then  Todoカラムに「タスクA」、In Progressカラムに「タスクB」が表示される
```

#### エッジケース（壁打ちで気づいたもの）

```gherkin
シナリオ: 空のタイトルではカードを作成できない
  Given ボードが表示されている
  When  タイトルを空のまま新しいカードを作成しようとする
  Then  カードは作成されない
  And   エラーが表示される

シナリオ: TodoからDoneに直接移動できない
  Given Todoカラムに「タスクA」がある
  When  「タスクA」をDoneに移動しようとする
  Then  移動は行われない

シナリオ: Doneからin Progressに戻せる
  Given Doneカラムに「タスクA」がある
  When  「タスクA」をIn Progressに移動する
  Then  In Progressカラムに「タスクA」が表示される
```

> **ポイント:** 正常系を先に書き、エッジケースは気づいたものだけ書く。
> 網羅性はこの段階では求めない。チュートリアル2でイベントストーミングを学ぶと、
> もっと体系的にエッジケースを洗い出せるようになる。

## 振り返り

この章で体験したことを振り返る。

- **壁打ちの価値:** AIに問い返されることで「自分が暗黙に想定していたこと」が明示された
- **実例マッピングの価値:** ルールを先に明示することで、シナリオの「なぜこの例が必要か」が明確になった
- **2段階プロンプトの価値:** メモ → 実例マッピング → シナリオと段階を踏むことで、途中でレビューできた
- **判断は人間がする:** AIは選択肢を出すが、「今回のスコープでどうするか」を決めるのは自分
- **粗さを許容する:** 最初から完璧なシナリオを目指さない。粗いシナリオが「次の問いを生む素材」になる
- **スコープ外の明示:** 「やらないこと」を決めることで、「やること」が明確になった

# 振り返り

## チュートリアル1で体験したこと

5つの章を通じて、AIを使った開発フローを一通り体験した。

```
第1章: 壁打ちで要件を言語化する
  → 頭の中のイメージを言葉にし、粗いシナリオを書き出した

第2章: 要求定義を整える
  → メモを要件概要・振る舞い定義・ユビキタス言語・判断記録の形式に落とした

第3章: 詳細設計を生成する
  → 要求定義を渡してAIに詳細設計を生成させ、レビューした

第4章: 実装する
  → 詳細設計を渡してAIに実装させ、シナリオで品質を確認した

第5章: Playwrightでテストする
  → featureファイルのシナリオをE2Eテストとして自動化した
```

---

## 学んだ原則

### 1. フェーズを分けて情報を渡す

```
要求定義だけ渡す → 詳細設計を生成
詳細設計だけ渡す → 実装を生成
```

全部まとめて渡すより、フェーズごとに情報を絞って渡す方がAIの出力精度が上がる。
これは `guide/01_philosophy.md` の「コンテキスト設計」の原則そのものである。
詳細は `guide/appendix/F_context_engineering.md` を参照。

### 2. 人間が判断し、AIが実行する

| フェーズ | 人間の役割 | AIの役割 |
|---------|-----------|---------|
| 要件定義 | 判断・決定 | 問い返し・整理 |
| 詳細設計 | レビュー・承認 | 生成 |
| 実装 | スコープ定義・レビュー | 実装 |
| テスト | テスト観点の確認 | テストコード生成 |

上流ほど人間が主導し、下流ほどAIに委譲する。
この関係は `guide/04_collaboration_matrix.md` のマトリックスに対応している。

### 3. 発見プロセス（実例マッピング）の価値

実例マッピングでルール・例・疑問を先に整理してからGherkinを書いた。
ルールが明示されることで、シナリオの網羅性が上がり、「なぜこの例が必要か」が分かりやすくなった。
`guide/appendix/A_bdd.md` が示す Discovery → Formulation → Automation の流れを体感した。

### 4. シナリオが全体をつなぐ

```
第1章で実例マッピング → 粗いシナリオ
  → 第2章でGherkin形式に整えた
  → 第4章で受け入れ条件として使った
  → 第5章でE2Eテストに変換した
```

振る舞いシナリオは「仕様の記述」であり「テストの設計」であり「AIへの指示」でもある。
1つの成果物が複数の役割を果たすことが、BDDの強みである。

### 5. 「なぜ」を記録する価値

ADRに「なぜReactか」「なぜlocalStorageか」を記録した。
詳細設計の生成時にAIがこれを参照し、意図に沿った設計を出してくれた。

判断の理由は時間が経つと忘れる。記録してあれば、将来の自分にもAIにも伝わる。

## このチュートリアルでやらなかったこと

以下はチュートリアル2・3で段階的に導入する。

| やらなかったこと | 導入されるチュートリアル | 理由 |
|----------------|---------------------|------|
| 実例マッピング（深掘り） | チュートリアル2・3 | チュートリアル1では1ストーリーで体験。サイクル的な活用は後で |
| イベントストーミング | チュートリアル2 | Kanbanボードは単純でイベントの連鎖が少ないため |
| 異常系の体系的な洗い出し | チュートリアル2 | イベントストーミングとセットで学ぶ方が効果的 |
| ポリシー（自動実行ルール） | チュートリアル2 | レストランの題材の方がポリシーが直感的 |
| 集約・競合制御 | チュートリアル3 | 予約の「二重予約防止」で体験する方が実感が湧く |
| 境界付けられたコンテキスト | チュートリアル3 | 複数コンテキストが自然に出る予約システムで学ぶ |
| Page Objectパターン | チュートリアル2 | テストが増えてから整理の必要性が分かる |
| APIモック | チュートリアル2 | バックエンドが存在して初めて意味がある |
| コンポーネントテスト | チュートリアル3 | ドメインルールが複雑になってから価値が分かる |

## チュートリアル2へ

チュートリアル2「レストラン注文システム」では、以下が加わる。

- **イベントストーミング:** 「注文→調理→配膳→会計」のイベント連鎖を発見する
- **ポリシー:** 「調理完了したらサーバーに通知する」のような自動実行ルール
- **異常系の体系的洗い出し:** イベントストーミングで「ここで失敗したら？」が自然に見える
- **DDD成果物:** commands_and_events.md, domain_model.md を作る
- **Page Object・APIモック:** テストの保守性を上げる

チュートリアル1で学んだ開発フロー（要求定義 → 詳細設計 → 実装）はそのまま使う。
その上に、BDD・DDDのプラクティスを積み重ねていく。

> チュートリアル2に進む前に、以下を読んでおくことを推奨する。
> - `guide/05_bdd_and_ddd.md` の「実例マッピングとイベントストーミングの関係」— 両手法の使い分け・サイクルの回し方
> - `guide/appendix/C_event_storming.md` — イベントストーミングの進め方

### ガイドが認める限界

`guide/01_philosophy.md` の「このガイド自身の限界と課題」で述べられているように、
このチュートリアルで作った成果物セット（overview.md + features/ + ubiquitous_language.md + ADR）が
本当に必要十分かは未検証である。

1人開発なら overview.md + features/ だけで十分かもしれない。
実務では自分のプロジェクトで費用対効果を確認してほしい。

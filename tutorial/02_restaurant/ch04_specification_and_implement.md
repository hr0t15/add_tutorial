# 第4章: 詳細設計を生成・実装する

## この章のゴール

> **正常系・異常系のシナリオを満たす動くアプリケーションがある状態**

チュートリアル1で学んだフロー（詳細設計の生成 → 実装）を使い、
レストラン注文システムを実装する。

今回の新しい点: **異常系シナリオを受け入れ条件に含める**。

---

## この章の構成

チュートリアル1では詳細設計の生成と実装を別々の章に分けたが、
フローは学習済みなのでこの章でまとめて進める。

```
ステップ1: 詳細設計を生成する
ステップ2: バックエンドを実装する
ステップ3: フロントエンドを実装する
ステップ4: 全シナリオを確認する
```

---

## ステップ1: 詳細設計を生成する

### AIへの依頼

```
documents/requirements/ 配下のファイルをすべて読んで、
documents/specification/ を生成してください。

以下のファイルを作ってください。
- architecture.md（フロントエンド・バックエンドの構成）
- test_plan.md（テスト方針）
- backend/api_design.md（REST APIエンドポイント・データ構造）
- backend/sequences/（主要な処理フローのシーケンス）

特に以下を重視してください。
- ADR-001に従い、フロントエンドとバックエンドをREST APIで分離する
- ADR-002に従い、リアルタイム通知はポーリングで実現する
- commands_and_events.md のポリシーを設計に反映する
- domain_model.md の状態遷移を正確に反映する
```

### 詳細設計で特に確認すべき点

チュートリアル1と比べて、以下が新たに含まれる。

| 項目 | 確認内容 |
|------|---------|
| REST APIエンドポイント | コマンドごとにエンドポイントが定義されているか |
| ポリシーの実装方針 | 3つのポリシーがどこで・どう実行されるか明記されているか |
| 状態遷移の制約 | OrderItem の状態遷移制約がAPIレベルで反映されているか |
| ポーリング設計 | 厨房画面・配膳画面のポーリング間隔・エンドポイントが定義されているか |

### backend/api_design.md のイメージ（抜粋）

```markdown
## REST API エンドポイント

### 注文

| メソッド | パス | 説明 | 対応コマンド |
|---------|------|------|------------|
| POST | /api/tables/:tableId/orders | 注文作成 | PlaceOrder |
| POST | /api/orders/:orderId/items | 品目追加 | AddItemToOrder |
| DELETE | /api/order-items/:itemId | 品目キャンセル | CancelOrderItem |

### 厨房

| メソッド | パス | 説明 | 対応コマンド |
|---------|------|------|------------|
| GET | /api/kitchen/items | 未調理品目一覧 | （クエリ） |
| PATCH | /api/order-items/:itemId/cook | 調理完了 | MarkItemCooked |

### 配膳

| メソッド | パス | 説明 | 対応コマンド |
|---------|------|------|------------|
| GET | /api/serving/items | 配膳待ち品目一覧 | （クエリ） |
| PATCH | /api/order-items/:itemId/serve | 配膳済み | MarkItemServed |

### 会計

| メソッド | パス | 説明 | 対応コマンド |
|---------|------|------|------------|
| POST | /api/tables/:tableId/bill | 会計依頼 | RequestBill |
| POST | /api/tables/:tableId/settle | 会計完了 | SettleBill |
```

> **ポイント:** REST APIエンドポイントが commands_and_events.md のコマンドと1対1で対応している。
> イベントストーミングの成果がAPIの設計に直結していることが分かる。

## ステップ2: バックエンドを実装する

### 実装の順序

コマンドの依存関係を考慮して、以下の順序で進める。

```
① プロジェクトセットアップ・データモデル
② メニュー取得API
③ 注文作成・品目追加API
④ 品目キャンセルAPI（状態チェック付き）
⑤ 調理完了API
⑥ 配膳済みAPI
⑦ 会計API + テーブルクリアポリシー
⑧ ポーリング用クエリAPI
```

### AIへの依頼例（③ 注文作成）

```
documents/specification/ を参照して、注文作成APIを実装してください。

以下のシナリオを受け入れ条件として満たしてください。

Scenario: テーブルの注文を入力する
  Given テーブルT1が空席状態である
  When サーバーがT1を選択し「カレー」「サラダ」を注文する
  Then T1の注文に「カレー」「サラダ」が追加される
  And 厨房画面に「T1: カレー」「T1: サラダ」が表示される

Scenario: 品切れの品目を注文しようとする
  Given テーブルT1が空席状態である
  And 「カレー」が品切れ状態である
  When サーバーがT1の注文で「カレー」を選択しようとする
  Then 「品切れです」と表示される
  And 「カレー」は注文に追加されない
```

> **チュートリアル1との違い:** 異常系シナリオも受け入れ条件に含めている。
> 「品切れの品目を注文しようとする」シナリオがあることで、
> AIはバリデーション処理を自然に実装する。

### AIへの依頼例（⑦ 会計 + ポリシー）

```
会計API（RequestBill, SettleBill）を実装してください。

以下のシナリオを受け入れ条件として満たしてください。

Scenario: テーブルの会計を行う
  Given テーブルT1の全品目が配膳済みである
  When サーバーがT1の会計を行う
  Then T1の合計金額が表示される
  And T1が空席状態に戻る

Scenario: 未配膳の品目がある状態で会計する
  Given テーブルT1に「カレー」（配膳済み）と「サラダ」（未配膳）がある
  When サーバーがT1の会計を行う
  Then 「未配膳の品目があります」と警告が表示される
  And 会計は完了する
  And T1が空席状態に戻る

注意: 会計完了後のテーブルクリアはポリシーとして実装してください。
（commands_and_events.md のポリシー「テーブルクリア」を参照）
```

## ステップ3: フロントエンドを実装する

### 画面ごとに実装する

```
① テーブル一覧画面（サーバー）
② 注文画面（サーバー）
③ 厨房画面（ポーリングで注文を取得）
④ 配膳画面（ポーリングで調理完了品目を取得）
```

### AIへの依頼例（③ 厨房画面）

```
厨房画面を実装してください。

要件:
- 未調理の注文品目が一覧表示される
- 各品目に「調理完了」ボタンがある
- ポーリング（5秒間隔）で新しい注文を取得する
- ADR-002に従い、WebSocketは使わない

以下のシナリオを参照してください。

Scenario: 厨房が調理完了を記録する
  Given 厨房画面に「T1: カレー」が未調理で表示されている
  When 厨房が「T1: カレー」を調理完了にする
  Then 「T1: カレー」が調理完了状態になる
  And サーバーの配膳画面に「T1: カレー 配膳待ち」が表示される
```

## ステップ4: 全シナリオを確認する

全10シナリオを手動で確認する。

### 確認チェックリスト

| # | シナリオ | 種別 | 結果 |
|---|---------|------|------|
| 1 | テーブルの注文を入力する | 正常系 | |
| 2 | 追加注文をする | 正常系 | |
| 3 | 厨房受付前に注文品目をキャンセルする | 正常系 | |
| 4 | 厨房が調理完了を記録する | 正常系 | |
| 5 | サーバーが配膳済みにする | 正常系 | |
| 6 | テーブルの会計を行う | 正常系 | |
| 7 | 品切れの品目を注文しようとする | 異常系 | |
| 8 | 調理完了した品目はキャンセルできない | 異常系 | |
| 9 | 調理未完了の品目は配膳できない | 異常系 | |
| 10 | 未配膳の品目がある状態で会計する | 異常系 | |

### ポリシーの確認

シナリオに加えて、ポリシーが正しく動作しているかも確認する。

| ポリシー | 確認方法 |
|---------|---------|
| 調理完了通知 | 厨房で調理完了 → 配膳画面に表示されるか |
| 全品配膳済み通知 | 全品を配膳済み → テーブル一覧に通知が出るか |
| テーブルクリア | 会計完了 → テーブルが空席に戻るか |

## 振り返り

- **異常系シナリオの価値:** 受け入れ条件に異常系を含めることで、AIがバリデーション処理を自然に実装した
- **コマンドとAPIの対応:** イベントストーミングで発見したコマンドが、そのままAPIエンドポイントになった
- **ポリシーの実装:** 「自動で起きるべきこと」がポリシーとして識別されていたため、実装の漏れがなかった
- **DDD成果物の効果:** domain_model.md の状態遷移があることで、不正な状態遷移を防ぐ実装が自然にできた

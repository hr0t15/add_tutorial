# 第1章: 壁打ちで要件を言語化する・正常系シナリオを書く

## この章のゴール

> **正常系シナリオが言語化されている状態**

レストラン注文システムの「何を作るか・なぜ作るか」をAIとの対話で固め、
正常系の振る舞いシナリオを書き出す。

チュートリアル1で学んだ壁打ちの流れを使い、今回はより複雑なドメインに挑む。

---

## チュートリアル1との違い

Kanbanボードとの最大の違いは **アクターが複数いる** ことである。

```
Kanbanボード: ユーザー1人がカードを操作する
レストラン:   お客様・サーバー（給仕）・厨房の3者が連携する
```

複数のアクターが登場すると、「誰が何をしたとき、何が起きるか」の組み合わせが増える。
この複雑さをどう整理するかが、第2章のイベントストーミングにつながる。

この章ではまず正常系（うまくいくケース）だけに集中する。

---

## ステップ1: 壁打ちで要件を固める

### AIへの最初の問いかけ

```
レストランの注文管理システムを作りたいと思っています。
テーブルごとに注文を管理し、厨房・サーバーが連携するシステムです。
要件を一緒に整理してもらえますか？
```

### AIから出てくる問い

チュートリアル1と同様に、AIは曖昧な部分を問い返してくる。
今回は複数のアクターがいるため、問いの幅が広い。

- テーブルは何卓あるか？ テーブルの追加・削除はできるか？
- メニューはどう管理するか？ 品切れはあるか？
- 注文の追加注文はできるか？
- 注文のキャンセルはできるか？ どのタイミングまで？
- 厨房への通知方法は？ リアルタイムか？
- 会計は個別会計か、テーブル単位か？
- サーバーは複数人いるか？ テーブルの担当制か？

### 判断する

| 問い | 判断 | 理由 |
|------|------|------|
| テーブル数 | 固定4卓（T1〜T4） | シンプルにする |
| メニュー | 固定リストから選択 | メニュー管理は対象外 |
| 品切れ | あり（品切れ表示） | イベントストーミングで面白いケースになる |
| 追加注文 | 可能 | レストランとして自然 |
| キャンセル | 厨房受付前のみ可能 | 調理開始後のキャンセルは複雑すぎる |
| 厨房通知 | リアルタイム（画面に表示） | 複数アクター連携の体験に必要 |
| 会計 | テーブル単位 | シンプルにする |
| サーバー | 1人（担当制なし） | 複数サーバーの管理は対象外 |

### ユーザーストーリーで機能要件を整理する

このチュートリアルではアクターが3者（お客様・サーバー・厨房）いる。
箇条書きだけでは「誰のための機能か」が曖昧になりやすいので、
ユーザーストーリー形式で整理する（`guide/02_development_process.md` 参照）。

```
As a サーバー, I want テーブルの注文を入力したい, so that 厨房に正確に伝わるから
As a サーバー, I want 品目を追加注文したい, so that お客様の追加リクエストに対応できるから
As a サーバー, I want 調理完了した品目を確認したい, so that 素早く配膳できるから
As a サーバー, I want テーブルの会計を処理したい, so that お客様をお見送りできるから
As a 厨房スタッフ, I want 注文一覧をリアルタイムで見たい, so that 優先順位を判断して調理できるから
As a 厨房スタッフ, I want 調理完了を記録したい, so that サーバーに配膳を促せるから
```

> これらのストーリーが、第2章のイベントストーミングや実例マッピングの入力になる。
> この段階では粗くてよい。掘り下げは後のステップで行う。

### スコープ外

```
- テーブルの追加・削除
- メニューの編集・カテゴリ管理
- 個別会計・割り勘
- 複数サーバーの担当制
- 予約管理
- 在庫管理（品切れの設定はできるが、在庫数のカウントはしない）
- 調理開始後のキャンセル
```

## ステップ2: アクターと画面を整理する

複数のアクターが登場するため、**誰がどの画面を見るか**を先に整理する。

### AIへの依頼

```
ここまでの要件をもとに、アクターと画面の対応を整理してください。
```

### 成果物のイメージ

```
アクター × 画面:

お客様（Customer）
  → 直接操作はしない（サーバーが代行する）

サーバー（Server）
  → テーブル一覧画面: 各テーブルの状態を確認する
  → 注文画面: テーブルを選んでメニューから注文を入力する
  → 配膳画面: 調理完了した品目を確認して配膳済みにする

厨房（Kitchen）
  → 厨房画面: 注文された品目を一覧で確認し、調理完了を記録する
```

> **ポイント:** お客様は直接操作しない（サーバーが代行する）という判断は重要。
> 「誰が操作するか」を明確にしないと、UIの設計で迷う。

---

## ステップ3: 正常系シナリオを書き出す

要件が固まったら、正常系のシナリオを書き出す。
チュートリアル1で学んだGiven/When/Then形式を使う。

### 注文の流れ

```gherkin
シナリオ: サーバーがテーブルの注文を入力する
  Given テーブルT1が空席状態である
  When  サーバーがT1を選択し「カレー」「サラダ」を注文する
  Then  T1の注文に「カレー」「サラダ」が追加される
  And   厨房画面に「T1: カレー」「T1: サラダ」が表示される

シナリオ: 追加注文をする
  Given テーブルT1に「カレー」の注文がある
  When  サーバーがT1を選択し「ドリンク」を追加注文する
  Then  T1の注文に「ドリンク」が追加される
  And   厨房画面に「T1: ドリンク」が表示される
```

### 厨房の操作

```gherkin
シナリオ: 厨房が調理完了を記録する
  Given 厨房画面に「T1: カレー」が未調理で表示されている
  When  厨房が「T1: カレー」を調理完了にする
  Then  「T1: カレー」が調理完了状態になる
  And   サーバーの配膳画面に「T1: カレー 配膳待ち」が表示される
```

### 配膳

```gherkin
シナリオ: サーバーが配膳済みにする
  Given 配膳画面に「T1: カレー 配膳待ち」が表示されている
  When  サーバーが「T1: カレー」を配膳済みにする
  Then  「T1: カレー」が配膳済み状態になる
```

### 会計

```gherkin
シナリオ: テーブルの会計を行う
  Given テーブルT1の全品目が配膳済みである
  When  サーバーがT1の会計を行う
  Then  T1の合計金額が表示される
  And   T1が空席状態に戻る
```

### キャンセル

```gherkin
シナリオ: 厨房受付前に注文をキャンセルする
  Given テーブルT1に「サラダ」の注文があり、まだ厨房が受け付けていない
  When  サーバーが「サラダ」をキャンセルする
  Then  T1の注文から「サラダ」が消える
  And   厨房画面から「T1: サラダ」が消える
```

---

## ステップ4: 気づきを記録する

正常系を書いていると、気になる点が出てくる。
今は解決しなくてよい。**記録しておく**ことが重要。

```
気づき・疑問:
- 厨房が注文を「受け付ける」ステップは必要か？ 自動受付でよいか？
- 品切れの品目を注文しようとしたらどうなる？
- 全品目が配膳済みでなくても会計できるべきか？（途中退席など）
- 調理完了からサーバーへの通知はどう実現する？ ポーリング？ WebSocket？
- 1つのテーブルに複数回の注文（追加注文）がある場合、会計はどうまとめる？
```

> **ポイント:** これらの疑問は第2章のイベントストーミングで体系的に洗い出す。
> 今の段階で「気づいた」こと自体が価値である。

---

## この章の成果物

```
✅ 機能要件（アクター・画面・操作の整理）
✅ スコープ外リスト
✅ 正常系シナリオ 6本
✅ 気づき・疑問リスト
```

> チュートリアル1と比べて、シナリオの複雑さが増している。
> 複数のアクターが関わるため、「誰が・何を・いつ」が重要になっている。

## 次の章へ

第2章では、イベントストーミングを実施する。

正常系シナリオを書いただけでは見えない「裏で何が起きているか」を掘り下げ、
異常系・エッジケースを体系的に発見する。

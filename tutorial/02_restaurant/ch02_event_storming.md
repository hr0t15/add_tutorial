# 第2章: イベントストーミングで構造を掘る

## この章のゴール

> **コマンド・イベント・ポリシーが整理され、異常系シナリオが追加されている状態**

第1章で書いた正常系シナリオの「裏で何が起きているか」を、
イベントストーミングで掘り下げる。

この章がチュートリアル2の核心であり、**イベントストーミングの前と後で何が変わるか**を体感する。

---

## イベントストーミングとは

`guide/appendix/C_event_storming.md` に詳細があるが、ここでは実践に必要な最低限を説明する。

**「このシステムで何が起きるか？」を時系列で洗い出す手法。**

通常はホワイトボードと付箋で行うが、このチュートリアルではAIとの対話で進める。

### 基本要素

| 要素 | 説明 | 命名規則 | 例 |
|------|------|---------|-----|
| **ドメインイベント** | 起きた出来事 | 過去形 | OrderPlaced |
| **コマンド** | イベントを引き起こす指示 | 命令形 | PlaceOrder |
| **アクター** | コマンドを発行する人 | 名詞 | Server, Kitchen |
| **ポリシー** | イベント発生時に自動実行されるルール | 「〜したとき〜する」 | 調理完了したらサーバーに通知 |

> **ポリシーが今回の新概念。** 「料理が完成したらサーバーに通知する」のように、
> 人間が意識しなくても自動で動くルールがポリシーである。

---

## チュートリアル1の実例マッピングとの関係

チュートリアル1では実例マッピングで「ユーザーから見た振る舞い」を発見した。
このチュートリアルでは、それに加えてイベントストーミングで「システムの裏側の構造」を掘る。

`guide/05_bdd_and_ddd.md` の「実例マッピングとイベントストーミングの関係」で解説されている通り、
両者は同じドメインを異なる角度から照らす関係にある。
レストランのように複数アクターとポリシーが絡む場合は、
両者をサイクルで回すことで精度が上がる。

このチュートリアルでは `guide/05_bdd_and_ddd.md` の「パターンA: 実例マッピング先行」を体験している。

```
第1章で正常系シナリオを書いた（実例マッピング的な発見）
  ↓  ← 今ここ（1周目: 実例マッピング → イベントストーミング）
この章でイベントストーミングを実施し、裏側の構造を掘る
  ↓
ポリシー・異常系を発見し、シナリオにフィードバックする（イベントストーミング → 実例マッピングへの還流）
  ↓
第3章で成果物を整理する
```

> 壁打ちメモが雑多な場合は、`guide/appendix/B_ddd.md` の
> 「雑多なメモからのイベントストーミング成果生成」プロンプトを使うと、
> メモ → イベントストーミング成果 → DDD成果物 の2段階で整理できる。

## ステップ1: ドメインイベントを洗い出す

### AIへの依頼

```
レストラン注文システムで起きうるドメインイベントをすべて洗い出してください。
以下の正常系シナリオを参考にしてください。

[第1章のシナリオをここに貼る]

イベントは過去形で命名してください（例: OrderPlaced）。
時系列順に並べてください。
```

### AIが出すイベント候補

```
時系列:

1. OrderPlaced          （注文が入力された）
2. OrderReceivedByKitchen （厨房が注文を受信した）
3. ItemCookingStarted    （品目の調理が開始された）
4. ItemCooked            （品目の調理が完了した）
5. ItemServed            （品目が配膳された）
6. BillRequested         （会計が依頼された）
7. BillSettled           （会計が完了した）
8. TableCleared          （テーブルが空席に戻った）
```

### 人間が判断すること

AIが出した候補に対して、以下を判断する。

- **不要なイベントはないか？**
  「OrderReceivedByKitchen」と「ItemCookingStarted」は分ける必要があるか？
  → このチュートリアルでは注文が入った時点で厨房に表示されるので、
  「受信」は自動。「調理開始」も明示的に管理しない。
  → `OrderReceivedByKitchen` と `ItemCookingStarted` は削除。

- **足りないイベントはないか？**
  「追加注文」はどうなる？ → `OrderPlaced` と同じイベントでよいか、
  `ItemAddedToOrder` として分けるか？
  → 追加注文は既存注文への品目追加なので `ItemAddedToOrder` として分ける。

- **キャンセルのイベントは？**
  → `OrderItemCancelled`（注文品目がキャンセルされた）を追加。

### 整理後のイベント一覧

```
1. OrderPlaced            （注文が作成された）
2. ItemAddedToOrder       （品目が注文に追加された）
3. OrderItemCancelled     （注文品目がキャンセルされた）
4. ItemCooked             （品目の調理が完了した）
5. ItemServed             （品目が配膳された）
6. BillRequested          （会計が依頼された）
7. BillSettled            （会計が完了した）
8. TableCleared           （テーブルが空席に戻った）
```

## ステップ2: コマンドとアクターを追加する

### AIへの依頼

```
整理したドメインイベントに対して、
それぞれを引き起こすコマンドとアクターを追加してください。

コマンドは命令形で命名してください（例: PlaceOrder）。
```

### 整理結果

```
アクター      コマンド              →  ドメインイベント
─────────────────────────────────────────────────────
Server    →  PlaceOrder            →  OrderPlaced
Server    →  AddItemToOrder        →  ItemAddedToOrder
Server    →  CancelOrderItem       →  OrderItemCancelled
Kitchen   →  MarkItemCooked        →  ItemCooked
Server    →  MarkItemServed        →  ItemServed
Server    →  RequestBill           →  BillRequested
Server    →  SettleBill            →  BillSettled
System    →  ClearTable            →  TableCleared
```

### 気づくこと

- `ClearTable` のアクターは System（自動）。会計完了後に自動でテーブルを空席にする。
  → これはポリシーの候補である。
- Server が多くのコマンドを発行する。Kitchen は `MarkItemCooked` のみ。
- お客様（Customer）はコマンドを発行しない（サーバーが代行する）。

## ステップ3: ポリシーを発見する

ここがイベントストーミングの最も価値がある部分。
**「このイベントが起きたとき、自動的に何かが起きるか？」** を問う。

### AIへの依頼

```
各ドメインイベントについて、
「このイベントが起きたとき、自動的に実行されるべきルール」がないか検討してください。
それがポリシーです。
```

### 発見されるポリシー

```
ポリシー1: 調理完了通知
  トリガー: ItemCooked
  ルール:   品目が調理完了したとき、サーバーの配膳画面に通知する

ポリシー2: テーブルクリア
  トリガー: BillSettled
  ルール:   会計が完了したとき、テーブルを自動的に空席状態に戻す

ポリシー3: 全品配膳済み通知
  トリガー: ItemServed
  ルール:   テーブルの全品目が配膳済みになったとき、
            サーバーに「会計可能」を通知する
```

### ポリシーの価値

ポリシーは**人間が意識していなかった自動化ルール**を明示化する。

第1章のシナリオでは「調理完了したらサーバーの配膳画面に表示される」と書いたが、
それが**ポリシーとして設計すべきルール**であることは明示されていなかった。

ポリシーを識別することで:
- 実装時に「どこに自動処理を入れるか」が明確になる
- テストで「自動的に起きるべきこと」を検証できる
- 「自動でやるべきか、手動にすべきか」の判断ポイントが見える

## ステップ4: 異常系・エッジケースを発見する

イベントストーミングの流れを眺めると、「ここで失敗したらどうなる？」が自然に見えてくる。

### AIへの依頼

```
各コマンドとイベントについて、
「失敗する可能性」「エッジケース」を洗い出してください。
```

### 発見される異常系

```
注文時:
  - 品切れの品目を注文しようとしたら？
    → エラーを表示し、注文に追加しない

キャンセル時:
  - すでに調理完了した品目をキャンセルしようとしたら？
    → キャンセル不可。調理完了後はキャンセルできない

配膳時:
  - 調理完了していない品目を配膳済みにしようとしたら？
    → 不可。調理完了した品目のみ配膳できる

会計時:
  - まだ配膳されていない品目があるのに会計しようとしたら？
    → 判断が必要: 許可する？ しない？
    → このチュートリアルでは「未配膳の品目がある場合は警告を出すが、会計は可能」とする
    → 理由: 品目をキャンセルして会計するケースがある

テーブル状態:
  - 空席でないテーブルに新しい注文を入れようとしたら？
    → 追加注文として扱う（新しいOrderは作らない）
```

> **ポイント:** 「会計時に未配膳品目があったら？」のような問いは、
> 正常系シナリオだけでは気づきにくい。
> イベントの流れを追うことで「ここで分岐がある」と発見できる。

## ステップ5: 異常系シナリオにフィードバックする

発見した異常系をGiven/When/Then形式で書き出す。

```gherkin
シナリオ: 品切れの品目を注文しようとする
  Given テーブルT1が空席状態である
  And   「カレー」が品切れ状態である
  When  サーバーがT1の注文で「カレー」を選択しようとする
  Then  「品切れです」と表示される
  And   「カレー」は注文に追加されない

シナリオ: 調理完了した品目はキャンセルできない
  Given テーブルT1に「カレー」の注文があり、調理が完了している
  When  サーバーが「カレー」をキャンセルしようとする
  Then  キャンセルは行われない
  And   「調理完了済みのためキャンセルできません」と表示される

シナリオ: 調理未完了の品目は配膳できない
  Given テーブルT1に「カレー」の注文があり、まだ調理が完了していない
  When  サーバーが「カレー」を配膳済みにしようとする
  Then  配膳済みにならない

シナリオ: 未配膳の品目がある状態で会計する
  Given テーブルT1に「カレー」（配膳済み）と「サラダ」（未配膳）がある
  When  サーバーがT1の会計を行う
  Then  「未配膳の品目があります」と警告が表示される
  And   会計は完了する
  And   T1が空席状態に戻る
```

## イベントストーミングの全体像

ここまでの成果を1枚にまとめる。

```
Server → [PlaceOrder] → OrderPlaced
                              │
                              ↓ （厨房画面に表示）
Server → [AddItemToOrder] → ItemAddedToOrder
Server → [CancelOrderItem] → OrderItemCancelled
                              │
Kitchen → [MarkItemCooked] → ItemCooked
                              │
                              ↓ ポリシー: 調理完了通知
                              │
Server → [MarkItemServed] → ItemServed
                              │
                              ↓ ポリシー: 全品配膳済み通知
                              │
Server → [RequestBill] → BillRequested
Server → [SettleBill] → BillSettled
                              │
                              ↓ ポリシー: テーブルクリア
                              │
System → [ClearTable] → TableCleared
```

## 振り返り: イベントストーミングで何が変わったか

| 観点 | 第1章（壁打ちだけ） | 第2章（イベントストーミング後） |
|------|-------------------|---------------------------|
| シナリオ数 | 正常系 6本 | 正常系 6本 + 異常系 4本 |
| ポリシー | 暗黙的（「通知される」と書いただけ） | 明示的（3つのポリシーとして識別） |
| エッジケース | 「気づき・疑問リスト」として未整理 | 具体的なシナリオに落とされた |
| ドメインの理解 | 「注文→調理→配膳→会計」の流れ | コマンド・イベント・ポリシーの構造 |

> **最大の変化:** 「暗黙のルール」がポリシーとして明示された。
> これにより、実装時に「ここに自動処理を入れる」という判断が明確になる。

# チュートリアル2: レストラン注文システム

## 概要

このチュートリアルでは、レストランの注文システムを題材に
**BDDとDDDの基礎、特にイベントストーミング**を学ぶ。

チュートリアル1で学んだ開発フローを前提とし、
「イベントストーミングがなぜ必要か」「やるとどう変わるか」を体感することが目的。

---

## 学習目標

このチュートリアルを終えると以下ができるようになる。

- [ ] イベントストーミングをAIと一緒に実施できる
- [ ] ドメインイベント・コマンド・ポリシーを識別できる
- [ ] commands_and_events.md を作れる
- [ ] domain_model.md を作れる
- [ ] ポリシーをfeatureファイルのシナリオに落とせる
- [ ] イベントストーミングで気づいた異常系をシナリオにフィードバックできる
- [ ] 異常系シナリオを書けるようになる

---

## 題材: レストラン注文システム

### 作るもの

テーブルごとに注文を管理し、厨房・サーバーが連携するシステム。

```
お客様注文 → 厨房受信 → 調理完了 → 配膳 → 会計
```

### なぜレストランか

- ドメインイベントの連鎖が自然で分かりやすい
- ポリシー（「料理が完成したらサーバーに通知」）が直感的
- 複数のアクター（お客様・サーバー・厨房）が登場してイベントストーミングが映える

## チュートリアルの章構成

```
第1章: 壁打ちで要件を言語化する・正常系シナリオを書く
  └─ レストラン注文システムの要件をAIとの対話で固める
  └─ 正常系の振る舞いシナリオを書き出す
  ゴール: 正常系シナリオが言語化されている状態

第2章: イベントストーミングで構造を掘る
  └─ AIとイベントストーミングを実施する
  └─ ドメインイベント・コマンド・ポリシーを洗い出す
  └─ 異常系・エッジケースに気づき、シナリオにフィードバックする
  ゴール: コマンド・イベント・ポリシーが整理され、異常系シナリオが追加されている状態

第3章: 要求定義を整える
  └─ イベントストーミングの成果を commands_and_events / domain_model に落とす
  └─ ユビキタス言語を更新する
  └─ featureファイルを正常系・異常系合わせて整理する
  ゴール: DDD成果物を含む要求定義が揃っている状態

第4章: 詳細設計を生成・実装する
  └─ チュートリアル1と同じ流れで詳細設計の生成 → 実装
  └─ 異常系シナリオを受け入れ条件に含める
  ゴール: 正常系・異常系のシナリオを満たす動くアプリケーションがある状態

第5章: Page Object・APIモックを導入する
  └─ Playwrightテストを Page Objectパターンで整理する
  └─ APIモックでバックエンド非依存のテストを書く
  ゴール: 保守性の高いE2Eテストが通る状態

振り返り
  └─ イベントストーミングの前後で何が変わったか
  └─ チュートリアル3で何が加わるか
```
